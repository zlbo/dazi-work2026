## 三、详细问题记录

### 3.1 ClickHouse 表设计相关问题

#### 问题 1：主键必须是排序键的前缀

**错误现象**：

```
DB::Exception: Primary key must be a prefix of the sorting key
```

**问题原因**：

在 MergeTree 引擎中，PRIMARY KEY 定义的列必须是 ORDER BY 子句的前缀。例如，以下定义是错误的：

```sql
CREATE TABLE example (
    id String,
    date_key Int32,
    PRIMARY KEY (id)
) ENGINE = MergeTree()
ORDER BY (date_key, id)  -- 错误：PRIMARY KEY (id) 不是 ORDER BY 的前缀
```

**解决方案**：

将 PRIMARY KEY 和 ORDER BY 调整为一致的顺序，确保主键列是排序键的前缀：

```sql
CREATE TABLE example (
    id String,
    date_key Int32,
    PRIMARY KEY (id)
) ENGINE = MergeTree()
ORDER BY (id, date_key)  -- 正确：PRIMARY KEY (id) 是 ORDER BY 的前缀
```

**涉及表数量**：

本次部署的 23 张表中有大量表存在此问题，包括 fact_project_cost、fact_project_output、fact_payment、fact_receivable、fact_cash_flow、fact_risk、fact_change_order、fact_claim、fact_project_indicator 等。

**优化建议**：

1. **平台层面**：在执行 CREATE TABLE 语句前，增加自动校验逻辑，检测 PRIMARY KEY 是否为 ORDER BY 的前缀，若不是则给出明确的错误提示（而非等到 ClickHouse 返回错误）
2. **文档层面**：在 DaziScript 编写指南中增加 ClickHouse 表设计规范章节，明确说明主键与排序键的关系
3. **IDE/编辑器层面**：提供表定义的代码模板，自动生成符合规范的主键和排序键定义

#### 问题 2：Nullable 字段与非空值冲突

**错误现象**：

```
Invalid None value in non-Nullable column, column name: `parent_key`
```

**问题原因**：

某些字段（如 parent_key）在某些数据行中为 None，但表的定义中该字段为非 Nullable 的 String 类型。ClickHouse 不允许向非 Nullable 列插入 None 值。

**解决方案**：

1. **方案一**：修改表定义，将字段改为 Nullable 类型（推荐）

```sql
parent_key Nullable(String)
```

2. **方案二**：在插入数据时，将 None 替换为空字符串（适用于层级关系中根节点的场景）

```python
"parent_key": ""  # 替代 None
```

**本次采用**：两种方案结合使用，对于确实需要表示"无父级"概念的字段，采用方案一；对于普通字段中偶发的 None，采用方案二。

**优化建议**：

1. **平台层面**：提供 Nullable 字段的自动检测，在插入数据前扫描并给出警告
2. **SDK 层面**：在 insert_rows 方法中增加 None 值的自动处理策略配置（转为空字符串或 null）
3. **文档层面**：明确哪些字段建议使用 Nullable，哪些不建议使用

---

#### 问题 3：Unrecognized column 错误

**错误现象**：

```
Unrecognized column 'project_code' in table fact_project_output
```

**问题原因**：

插入数据时指定的列名在目标表中不存在。可能是以下原因：
- 拼写错误
- 表定义已更新但脚本未同步
- 错误的字段映射

**解决方案**：

1. 检查表的实际结构：`DESCRIBE TABLE table_name`
2. 对比脚本中的字段列表，移除不存在的字段
3. 添加缺失的必需字段

**本次具体情况**：

fact_project_output 表缺少 confirmation_status 字段，脚本中使用了该字段但表未定义。

**优化建议**：

1. **平台层面**：在执行 INSERT 前增加字段校验，提前报告不匹配的列
2. **SDK 层面**：insert_rows 方法增加 dry_run 模式，仅校验不实际插入
3. **IDE 层面**：提供表结构自动补全功能

### 3.2 DaziScript API 相关问题

#### 问题 4：不支持的 import 语句

**错误现象**：

```
DaziScript 不允许导入模块'dazi_script'
DaziScript 不允许导入模块'dazi.ontology'
```

**问题原因**：

DaziScript 执行环境是一个受限的沙箱，不允许直接 import 外部模块。脚本必须使用平台预置的 `space`、`output`、`input` 等全局对象。

**错误写法**：

```python
import dazi_script as s  # 错误
from dazi.ontology import OntologyService  # 错误
```

**正确写法**：

```python
def main():
    space_id = "space__panda_construction"
    s = space.get(space_id)  # 使用全局 space 对象获取空间实例

    # 使用 s.sql、s.onto 等进行操作
    s.sql.execute("...")
    s.sql.insert_rows("...", [...])
```

**优化建议**：

1. **平台层面**：
   - 在脚本发布阶段增加 import 语句检测，直接拒绝包含禁止 import 的脚本
   - 提供更明确的错误提示，告知用户正确的初始化方式
2. **文档层面**：在所有脚本示例中统一使用正确的 `space.get()` 模式，不再使用 `import dazi_script`
3. **IDE 层面**：在脚本模板中预置正确的初始化代码

#### 问题 5：OntologyModule API 参数错误

**错误现象**：

```
OntoModule.define_object_type() got an unexpected keyword argument 'display_name'
```

**问题原因**：

不了解 OntoModule 的正确 API 签名，使用了错误的参数名。

**错误写法**：

```python
s.onto.define_object_type(
    code="Project",
    name="项目",
    display_name="项目",  # 不存在此参数
    classification="归属关系",  # 参数名错误
    description="工程项目主数据",
    properties=[...]  # 不支持此参数
)
```

**正确写法**：

```python
s.onto.define_object_type(
    code="Project",
    name="项目",
    description="工程项目主数据",
    category_347="归属关系"
)
# 属性需要单独定义
s.onto.define_property(object_type, property_name, display_name, semantic_role, qualified_name)
```

**优化建议**：

1. **平台层面**：
   - 增加 API 参数的运行时校验，给出明确的可用参数列表
   - 提供 API 的自动补全功能（IDE 或 Web UI）
2. **文档层面**：整理所有 OntoModule API 的完整签名和示例
3. **SDK 层面**：增加类型提示 (type hints)，IDE 可据此提供补全

#### 问题 7：Qualified Name 不是 dimension 错误

### 3.3 脚本格式与执行相关问题

#### 问题 8：脚本类型 (type) 配置

**问题描述**：

发布脚本时需要指定 `--type` 参数，但不同类型（如 init、data、rule、category）的脚本在执行时有不同的限制和行为。

**本次使用的 type 类型**：

| 脚本 | type 参数 |
|------|-----------|
| commercial_cost_ontology_init.py | init |
| commercial_cost_seed_data.py | data |
| commercial_cost_business_rules.py | rule |
| commercial_cost_category_mount.py | category |

**现象**：

1. 发布后需要重新执行才能使修改生效（即使是相同版本号的脚本）
2. 不同 type 的脚本在 sandbox 中的权限可能不同

**优化建议**：

1. **平台层面**：
   - 增加 `--force` 参数强制重新执行已存在的脚本
   - 明确每种 type 类型的脚本在 sandbox 中的可用 API
2. **文档层面**：详细说明每种脚本类型的用途、限制和执行时机

---

#### 问题 9：脚本幂等性设计

**问题描述**：

多次执行同一脚本可能产生重复数据或错误。

**本次解决方案**：

在脚本中添加幂等检查：

```python
def main():
    # 检查是否已有数据
    try:
        existing = s.sql.query("SELECT COUNT(*) FROM dim_project")
        if existing and existing[0][0] > 0:
            output.print("数据已存在，跳过插入")
            return
    except Exception:
        pass

    # 继续插入数据...
```

**涉及脚本**：

- commercial_cost_seed_data.py：检查 dim_project 是否有数据
- commercial_cost_business_rules.py：检查 fact_business_rules 是否有规则

**优化建议**：

1. **平台层面**：提供 `s.script.is_executed()` 方法查询脚本执行历史
2. **SDK 层面**：提供幂等装饰器 `@idempotent` 简化幂等逻辑
3. **文档层面**：推荐所有数据类脚本添加幂等检查

### 3.4 数据处理相关问题

#### 问题 11：datetime.date 类型与 Date 类型不匹配

**错误现象**：

在某些场景下，Python 的 `datetime.date` 对象无法直接插入 ClickHouse 的 Date 类型字段。

**解决方案**：

确保日期格式正确：

```python
from datetime import datetime

# 方式一：使用 date 对象
calendar_date = datetime(2024, 1, 15).date()

# 方式二：确保格式为 YYYY-MM-DD
"calendar_date": "2024-01-15"
```

**优化建议**：

1. **SDK 层面**：在 insert_rows 中增加日期类型的自动转换
2. **文档层面**：明确日期字段的推荐格式

#### 问题 12：Decimal 精度问题

**问题描述**：

金额类字段使用 `Decimal(18,2)` 定义，插入时需要确保 Python 端也是正确精度。

**解决方案**：

使用 `float` 类型时，ClickHouse 会自动处理精度：

```python
# 金额字段
"budget_amount": float(project.get("total_budget", 0))
```

**优化建议**：

1. **SDK 层面**：提供 Decimal 类型的自动处理

### 3.6 挂载 API 不存在问题

#### 问题 14：OntologyModule 缺少 mount_* 方法

**错误现象**：

```
'OntoModule' object has no attribute 'mount_table_to_category'
'OntoModule' object has no attribute 'mount_cube_to_category'
'OntoModule' object has no attribute 'mount_object_to_category'
'OntoModule' object has no attribute 'mount_link_to_category'
```

**问题描述**：

```python
try:
    s.onto.mount_table_to_category(item, category_name)
except Exception as e:
    output.print(f"警告: 挂载 {item} 失败 - {str(e)}")
```

**优化建议**：

1. **平台层面**：
   - 如果挂载 API 是规划中的功能，则尽快实现
   - 如果不需要此功能，则从文档中移除相关说明

## 六、经验教训总结

### 6.1 开发流程建议

1. **先看示例再看文档**：参考 `资源/examples/onto/设备运营/setup/equip_ops_ontology_init.py` 等已成功的示例，比看文档更高效
2. **小步快跑，逐步验证**：每次修改后立即发布和执行，快速定位问题
3. **幂等性设计**：所有数据类脚本都应考虑幂等，避免重复执行问题
4. **错误信息需要解读**：ClickHouse 的错误信息通常是准确的，需要仔细阅读

### 6.2 技术要点

1. **ClickHouse 表设计**：
   - PRIMARY KEY 必须是 ORDER BY 的前缀
   - Nullable 字段需要显式声明
   - 空字符串与 NULL 是不同的概念

2. **DaziScript**：
   - 不允许 import 外部模块
   - 使用 `space.get()` 获取空间实例
   - 使用 `s.sql`、`s.onto` 等进行操作

3. **Ontology API**：
   - `define_object_type()` 的参数：code, name, description, category_347
   - 属性需要单独用 `define_property()` 定义
   - `bind_source()` 绑定 Cube 时 Cube 必须已存在

### 6.3 调试技巧

1. **查看已注册的对象**：使用 `dazi onto object list --space <space_id>`
2. **查看错误日志**：CLI 输出中有详细的错误堆栈
3. **分步执行**：将复杂脚本拆分为多个简单脚本，逐一验证
