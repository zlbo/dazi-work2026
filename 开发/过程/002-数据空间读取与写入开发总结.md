## 1. 业务目标

在数据流程中实现「**从数据空间读表 → 内存 SQL 关联 → 回写数据空间**」的标准链路：

| 步骤 | 节点类型           | 说明                                               |
| ---- | ------------------ | -------------------------------------------------- |
| 1    | `dataspace-source` | 从 XSH矿业01 读取 `cost_center_info`（成本中心表） |
| 2    | `dataspace-source` | 从 XSH矿业01 读取 `cost_type_info`（成本类型表）   |
| 3    | `sql-query`        | DuckDB 内存 SQL 关联，产出宽表变量                 |
| 4    | `dataspace-sink`   | 写入 temp_space001 的 `temp_test003`               |

与 VS-flow0 使用 `database-source` / `database-sink` + `connectionId` 不同，本案例走 **数据空间抽象层**（`spaceId`），不直接绑定 ClickHouse 连接 ID。

## 2. 节点类型与配置对照

### 2.1 读取：`dataspace-source`

| 配置项                 | 说明                                                                       |
| ---------------------- | -------------------------------------------------------------------------- |
| `spaceId`              | `ads_dataspaces.id`，如 `space__xsh_01`（`dazi-flow dataspace list` 核对） |
| `output_variable_name` | 产出表变量名，供下游 `sql-query` 作表名                                    |
| `code.sql`             | 针对空间内**物理表名**写 SQL，写在 `节点/<名>/code.sql`                    |

**与 `database-source` 的区别**

| 维度     | `dataspace-source`                | `database-source`      |
| -------- | --------------------------------- | ---------------------- |
| 配置键   | `spaceId`                         | `connectionId`         |
| 连接方式 | 数据空间存储引擎（`get_storage`） | 直连 `ads_connections` |
| 适用场景 | 空间内已注册 native 表            | 外部库或未纳入空间的表 |

### 2.2 写入：`dataspace-sink`

| 配置项                | 说明                                                   |
| --------------------- | ------------------------------------------------------ |
| `spaceId`             | 目标数据空间 ID，如 `temp_space001`                    |
| `tableName`           | 空间内目标物理表名，如 `temp_test003`                  |
| `input_variable_name` | 要写入的表变量名（与上游 `output_variable_name` 一致） |
| `mode`                | `append`（默认）或 `replace`                           |
| `syncMetadata`        | 写后是否同步元数据（默认 true）                        |
| `display_name`        | 空间内表中文显示名（可选）                             |

**无 `code.*` 文件**，配置均在 `flow.json` 节点 `data` 中。

### 2.3 中间关联：`sql-query`

| 配置项                 | 说明                                                    |
| ---------------------- | ------------------------------------------------------- |
| `output_variable_name` | 本节点产出变量名                                        |
| `attached_variables`   | 需引用的上游表变量名列表                                |
| `code.sql`             | SQL 中**直接用变量名作表名**（如 `FROM 成本中心表 cc`） |

## 5. 关键代码示例

### 5.1 读取成本中心表（`节点/读取成本中心表/code.sql`）

```sql
-- spaceId = space__xsh_01
-- output_variable_name = 成本中心表
SELECT id, name, code, parent_id, mine_area, status
FROM cost_center_info
UNION ALL
SELECT
    cost_center AS id,
    cost_center AS name,
    cost_center AS code,
    '' AS parent_id,
    mine_area,
    'active' AS status
FROM (
    SELECT cost_center, mine_area
    FROM budget_plan
    WHERE cost_center IS NOT NULL
    GROUP BY cost_center, mine_area
) bp
WHERE bp.cost_center NOT IN (SELECT id FROM cost_center_info)
LIMIT 100000;
```

### 5.2 读取成本类型表（`节点/读取成本类型表/code.sql`）

```sql
-- spaceId = space__xsh_01
-- output_variable_name = 成本类型表
SELECT id, name, code, category, unit, description
FROM cost_type_info
UNION ALL
SELECT
    cost_type AS id,
    cost_type AS name,
    cost_type AS code,
    cost_category AS category,
    '' AS unit,
    '' AS description
FROM (
    SELECT cost_type, cost_category
    FROM budget_plan
    WHERE cost_type IS NOT NULL
    GROUP BY cost_type, cost_category
) bp
WHERE bp.cost_type NOT IN (SELECT id FROM cost_type_info)
LIMIT 100000;
```

### 5.3 SQL 关联（`节点/SQL_成本关联/code.sql`）

```sql
-- 上游变量：成本中心表、成本类型表
-- output_variable_name = 成本中心类型宽表
SELECT
    cc.id              AS 成本中心ID,
    cc.name            AS 成本中心名称,
    cc.mine_area       AS 矿区,
    ct.id              AS 成本类型ID,
    ct.name            AS 成本类型名称,
    ct.category        AS 成本类别
FROM 成本中心表 cc
CROSS JOIN 成本类型表 ct
```

### 5.4 写入节点（`flow.json` 片段）

```json
{
```

## 6. 问题与修复

### 6.1 执行引擎未注册 `dataspace-source` / `dataspace-sink`

**现象**

- `node-exec` 报错：`Unknown node type: dataspace-source`
- `flows validate` 报错：`未注册的节点类型: dataspace-source`

**原因**

文档与画布 API 已支持该类型，但当时搭子服务器执行引擎尚未注册节点处理器。

**修复**

平台升级后重新 `flows validate 111`，errors 为空即可运行。

**经验**

- 不能仅凭文档判断可运行，**必须 `flows validate` + `node-exec` 实测**。
- 升级前临时可降级为 `database-source` / `database-sink` + `clickhouse__space_xsh_01`（与 VS-flow0 同模式）。

### 6.2 空表导致下游 SQL 失败

**现象**

- `dataspace-source` 节点状态 `success`，但变量 `row_count=0`、`columns=[]`。
- `sql-query` 报错：`Need at least one non-root column in the file`（Parquet 无列结构）。

**原因**

`cost_center_info`、`cost_type_info` 在 ClickHouse 中当前为 **0 行**。`SELECT *` 在无数据时产出的 Parquet **不含列元数据**，DuckDB 无法注册为表。

**修复**

1. **显式列名**：`SELECT id, name, … FROM …`，不要裸 `SELECT *`。
2. **保证至少一行或有效 schema**：本案例用 `UNION ALL` 从 `budget_plan` 补充 distinct 维表值；维表有数据后 `NOT IN (SELECT id FROM cost_center_info)` 自动不再补充。
3. **单节点验证**：`flow variable pull --name 成本中心表` 确认 `columns` 非空、`status=ready`。

### 6.3 维表之间无直接外键

**现象**

`cost_center_info` 与 `cost_type_info` 无直接 JOIN 键；空间关系定义中二者均通过 `daily_cost_report` 关联（`cost_center → cost_center_info.id`，`cost_type → cost_type_info.id`）。

**处理**

本案例演示用 `CROSS JOIN` 生成组合宽表。生产场景建议：

- 增加 `dataspace-source` 读取 `daily_cost_report`；
- 在 `sql-query` 中按关系 JSON 做 `LEFT JOIN` 两维表。

## 10. 与 database-source/sink 的选型建议

| 场景                                   | 推荐                                                 |
| -------------------------------------- | ---------------------------------------------------- |
| 读写在**已注册数据空间**内的 native 表 | **`dataspace-source` / `dataspace-sink`**            |
| 读写**外部库**或未纳入空间的表         | `database-source` / `database-sink` + `connectionId` |
| 执行引擎尚未注册 dataspace 节点        | 临时降级 database 节点，或等平台升级                 |
| 需要写后同步空间元数据、表中文名       | `dataspace-sink` + `syncMetadata` + `display_name`   |

## 11. 后续同类开发建议

1. **开发前拉空间上下文**：侧栏下载 `资源/dataspaces/<空间名>/` 与单表 md，@ 给 AI 填 SQL。
2. **先 validate 再写代码**：避免画布能存、执行不能跑。
3. **空表先 variable pull**：无列结构时优先显式 SELECT 列名，再考虑 UNION 补数或改读事实表。
4. **多表 JOIN 查关系**：空间 `relationships.json` 或 `XSH矿业01.md` 中的关系数量，按事实表关联维表比维表互 JOIN 更贴近业务。
5. **文档入口**：日常命令与变量约定以 [`flows-guide.md`](../../资源/docs/flow/flows-guide.md) 为索引，节点细节见 [`node-code-guide.md`](../../资源/docs/flow/node-code-guide.md)。
