## 2. 根因分解

### 2.1 技术根因（代码）

| 问题 | 说明 |
|------|------|
| Excel 解析策略选错 | `header=[1,2,3]` 合并后列名 **不等于** `基本信息_组织` 等规范名；对该托管文件应使用 `表结构.json` 推荐的 `header=None` + `skiprows` + `names` |
| 总计行过滤顺序 | 若先裁剪列再过滤「总计」，会丢失 `基本信息_项目` 字段，总计行漏网（v6 通过后 v7 修复） |

### 2.2 流程根因（验证闭环）

| 问题 | 说明 |
|------|------|
| **假通过** | `node-exec` 返回 success，但 `variable pull` 的 schema/预览与本次代码预期不一致，Agent 未将其视为失败 |
| **push 时机混乱** | 在 variable 未对齐前 push，把错误代码固化到平台 v5 |
| **未读权威表结构** | 同工作区已有 `资源/files/.../表结构.json` 与同 `managed_file_id` 的参考实现（`成本报表数据08`），首轮未引用 |

## 3. 提示词 / 文档哪里不对

### 3.1 【严重】`node-exec` 与 `node push` 顺序矛盾

| 来源 | 顺序 |
|------|------|
| 用户任务模板（节点改错上下文） | **先** `node-exec`，**通过再** `node push` |
| `flow/run-fix-loop` 改错循环 Step 4 | IF `node-exec` 成功 → **再 push** |
| `flow/ai-workflow-playbook` §1 决策树 & §3 | **先** `node push`，**再** `node-exec` |

Agent 同时读到三份材料时会随机选一种，且无法解释「dirty 本地代码时 node-exec 到底跑哪份」。

**建议统一口径（二选一，写死一种）**：

- **方案 A（推荐，与 run-fix-loop 一致）**：本地 dirty 时 `node-exec` 上传本地 code 执行 → 通过后 `node push` 固化  
- **方案 B（与 playbook 一致）**：必须先 `node push` 再 `node-exec`，**禁止**在未 push 时声称代码已上线  

无论选哪种，三处文档 + VS Code 任务模板必须一致，并写明 **dirty 节点时 CLI 行为**。

### 3.2 【严重】成功判据不完整

`flow/run-fix-loop` 改错循环 Step 4 仅写：

```text
IF node-exec 成功 → push → 结束
```

`variable pull` 只出现在「提交规则」一条，**不在成功分支的必做步骤**。导致 Agent 只看 exit code，不看输出是否符合业务。

**应改为成功判据（缺一不可）**：

1. `node-exec` → `success: true`
2. `variable pull --name <output_variable_name>` → **列名 / 行数 / 样例行** 与需求一致
3. 若 1 与 2 矛盾 → 视为**未通过**，禁止 push，继续改错循环

### 3.3 【中等】`excel-python` 改错上下文缺少「表结构必读」

节点改错模板只 @ `flows-guide`、`node-code-guide`、`run-fix-loop`，未要求：

- 读取 `资源/files/<显示名>_<id8>/表结构.json`（含 `pandas_read`、`data_start_row`、`columns`）
- 搜索同 `managed_file_id` 的已有流程 `code.py` 作参考

`node-code-guide` §5 示例 A 已是正确写法，但与用户节点里遗留的 `header=[1,2,3]` 写法**未划清「禁止 / 仅当表结构明确推荐时可用」**。

### 3.4 【中等】`node-exec` 与 `variable pull` 一致性未文档化

实测现象：

- `node-exec` 显示通过时，`variable pull` 仍可能返回**上一次 debug run** 的旧 schema（本次 v5 push 前曾出现 6 列直接成本）
- `last-run.json` 的 `output_file` 已更新，但 variable API 的 `updatedAt` / schema 滞后

文档与提示词均未说明：**何时必须以 variable schema 为准、何时以 last-run 为准**。

### 3.5 【轻微】`快速启动_*.md` 脚手架信息过期

`快速启动_财务流程04.md` 生成时写「代码节点 0」，实际已有 `excel-python` 节点，削弱 Agent 对流程结构的信任。

## 4. 交给平台的优化清单（可排期）

| 编号 | 优化项 | 预期效果 |
|------|--------|----------|
| CLI-1 | `node-exec` 成功时输出：`codeSource`（`local-dirty` / `platform`）、`codeVersion`、本次 `debugRunId` | 消除「跑的是哪份代码」歧义 |
