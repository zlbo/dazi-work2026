## 一、分析概述

### 1.1 分析目标
建立"现实数据"到"本体建模表"的桥梁，明确每个本体建模表的数据来源。

### 1.2 前置条件
- 已完成阶段二（本体设计与确认）
- 已完成阶段三（本体建模与落地），平台中已创建15张本体建模表

### 1.3 分析范围
- **本体建模表清单**：平台中实际存在的6张事实表和9张维度表

> **重要**：本报告中的"理想表"即平台实际存在的本体建模表（`fact_*`事实表和`dim_*`维度表），不存在独立于本体模型的"中间表"或"过渡表"概念。

## 四、来源分析

### 4.1 来源类型定义

| 类型 | 说明 | 示例 |
|------|------|------|
| **类型A：直接使用** | 现有表可直接映射到本体建模表 | tb_project_indicator → fact_project_indicator |
| **类型B：合并/转换** | 多个现有表合并或转换生成 | 多个表 → fact_project_cost |
| **类型C：Excel导入** | Excel台账中手工填报数据导入 | 产值台账 → fact_project_output |
| **类型D：平台自动生成** | 平台自动维护的维度表 | dim_date由平台自动生成 |

### 4.2 本体建模表来源分析

#### fact_project_output

```yaml
目标表: fact_project_output
来源类型: C (Excel导入)
来源说明: Excel台账（表3.1-业主产值详表、表3.1.1-业主待确产值详表）包含产值数据，需要导入到数据库
依赖表: 无
转换逻辑: |
  1. 从Excel台账提取产值数据
  2. 转换列名：工程名称→project_id, 已确认产值→confirmed_output, 待确认产值→unconfirmed_output
  3. 补充字段：report_period从文件名提取，total_output = confirmed_output + unconfirmed_output
数据质量检查点: |
  - project_id是否在dim_project中存在
  - confirmed_output + unconfirmed_output = total_output
  - report_period格式是否正确（YYYY-MM）
优先级: P0
```

#### fact_project_cost

```yaml
目标表: fact_project_cost
来源类型: B (合并/转换)
来源说明: 需要从多个现有表合并生成
依赖表:
  - tb_project_major_cost (主控费用)
  - tb_project_cost_pay_detail (成本付款明细)
  - tb_project_cost_unconfirmed_analyse (待确认成本分析)
转换逻辑: |
  1. 从tb_project_major_cost提取成本代码、名称、目标成本
  2. 从tb_project_cost_pay_detail提取已发生成本
  3. 从tb_project_cost_unconfirmed_analyse提取待确认成本
  4. 合并生成统一的成本事实表
数据质量检查点: |
  - cost_code格式是否正确
  - target_cost >= incurred_cost（正常情况下）
  - cost_level是否为L1/L2/L3
优先级: P0
```

#### fact_project_indicator

```yaml
目标表: fact_project_indicator
来源类型: A (直接使用)
来源说明: 现有表结构与本体建模表一致，可直接映射使用
依赖表: 无
转换逻辑: 直接映射，字段一一对应
数据质量检查点: |
  - 确认tb_project_indicator表存在且有数据
  - indicator_code不为空
  - report_period格式正确
优先级: P0
```

#### fact_project_payment

```yaml
目标表: fact_project_payment
来源类型: B (合并/转换)
来源说明: 基于现有成本付款明细表转换
依赖表:
  - tb_project_cost_pay_detail
转换逻辑: |
  1. 从tb_project_cost_pay_detail提取付款数据
  2. 按项目和周期汇总
数据质量检查点: |
  - payment_amount >= 0
  - payment_date格式正确
优先级: P1
```

#### fact_project_balance

```yaml
目标表: fact_project_balance
来源类型: B (合并/转换)
来源说明: 基于现有收支汇总表转换
依赖表:
  - tb_project_income_outcome_summary
转换逻辑: |
  1. 从tb_project_income_outcome_summary提取收支数据
  2. 标准化字段名称
数据质量检查点: |
  - balance_amount计算正确
优先级: P1
```

#### fact_project_risk

```yaml
目标表: fact_project_risk
来源类型: B (合并/转换)
来源说明: 基于现有风险清单分析表转换
依赖表:
  - tb_project_risk_list_analyse
转换逻辑: |
  1. 从tb_project_risk_list_analyse提取风险数据
  2. 添加project_id关联
  3. 标准化alert_level字段（green/yellow/red）
数据质量检查点: |
  - risk_type不为空
  - alert_level属于[green/yellow/red]
优先级: P1
```

#### dim_project

```yaml
目标表: dim_project
来源类型: B (合并/转换)
来源说明: 需要从多个现有表提取项目基础信息
依赖表:
  - tb_project_indicator (project_id)
  - tb_project_major_cost (project_id)
转换逻辑: |
  1. 从现有表中提取所有唯一的project_id
  2. 项目名称需要从其他来源补充或手动维护
  3. project_code需要从现有数据推断
数据质量检查点: |
  - project_id不为空
  - project_name不为空
  - project_status属于[在建/竣工/结算/归档]
优先级: P0
```

#### dim_date

```yaml
目标表: dim_date
来源类型: D (平台自动生成)
来源说明: 平台自动维护的日期维度表
依赖表: 无（平台自动生成）
转换逻辑: 平台自动生成，无需人工干预
数据质量检查点: |
  - date格式正确
  - year/month/day字段正确
优先级: P0
```

## 八、风险与建议

### 8.1 技术风险

| 风险 | 描述 | 缓解措施 |
|------|------|---------|
| Excel格式变化 | Excel台账格式可能变化 | 定期检查Excel格式，建立版本控制 |
| 数据质量问题 | 现有数据可能存在错误 | 建立数据质量检查机制 |
| 数据源不可用 | 数据源连接可能中断 | 建立监控和告警机制 |

### 8.2 业务风险

| 风险 | 描述 | 缓解措施 |
|------|------|---------|
| 业务规则变更 | 企业业务规则可能变更 | 定期与业务人员沟通 |
| 数据口径不一致 | 不同表的数据口径可能不一致 | 统一数据口径定义 |

### 8.3 建议

1. **优先完成P0核心表**：确保核心业务功能能够正常运行
2. **建立数据质量监控**：定期检查数据质量
3. **文档化所有映射关系**：便于后续维护和追溯
4. **预留扩展空间**：设计时考虑未来业务扩展需求
