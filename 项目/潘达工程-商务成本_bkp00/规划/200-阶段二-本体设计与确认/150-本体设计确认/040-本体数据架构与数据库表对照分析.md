## 三、Object Type与数据库表对照分析

### 3.1 ProjectOutput（项目产值）

#### 目标数据结构

```yaml
ProjectOutput:
  - project_id: string (项目唯一标识)
  - report_period: string (报告周期YYYY-MM)

#### 数据库表对照

| 目标属性 | 数据库表 | 字段 | 可用性 | 说明 |

| confirmed_output | ❌ 无直接来源 | - | **需要模拟** | Excel表3.1为空模板 |

```
结论：tb_project_output 完全需要模拟

### 3.2 ProjectCost（项目成本）

#### 目标数据结构

```yaml
ProjectCost:
  - project_id: string (项目唯一标识)
  - report_period: string (报告周期)
  - cost_code: string (成本代码)
  - cost_name: string (成本名称)
  - cost_level: string (成本层级L1/L2/L3)
  - target_cost: decimal (目标成本)
  - incurred_cost: decimal (已发生成本)
  - unconfirmed_cost: decimal (待确认成本)
  - total_cost: decimal (合计成本)
```

#### 数据库表对照

| 目标属性 | 数据库表 | 字段 | 可用性 | 说明 |
|---------|---------|------|--------|------|
| project_id | tb_project_major_cost | project_id | ✅ 可直接使用 | 存在 |
| report_period | ❌ 无直接来源 | - | **需要补充** | tb_project_major_cost没有周期字段 |
| cost_code | tb_project_major_cost | code | ✅ 可直接使用 | 存在 |
| cost_name | tb_project_major_cost | name | ✅ 可直接使用 | 存在 |
| cost_level | - | - | **需要推断** | 需要从code推断 |
| target_cost | tb_project_major_cost | target_cost_amount | ✅ 可直接使用 | 存在 |
| incurred_cost | tb_project_cost_pay_detail | cost_actual_confirmed_acc | ✅ 可直接使用 | 存在 |
| unconfirmed_cost | tb_project_cost_unconfirmed_analyse | estimate_amount | ✅ 可直接使用 | 存在 |
| total_cost | - | - | **需要计算** | incurred_cost + unconfirmed_cost |

#### 分析结论

```
结论：tb_project_cost 70%可从现有表获取，30%需要补充

可直接使用：
  - project_id (from tb_project_major_cost)
  - cost_code (from tb_project_major_cost)
  - cost_name (from tb_project_major_cost)
  - target_cost (from tb_project_major_cost)
  - incurred_cost (from tb_project_cost_pay_detail)
  - unconfirmed_cost (from tb_project_cost_unconfirmed_analyse)

需要处理：
  - report_period: 需要补充周期字段
  - cost_level: 需要从cost_code推断
  - total_cost: 需要计算
```

## 五、业务逻辑加工规则

### 5.1 ProjectCost 合并逻辑

```yaml
# 从三个表合并ProjectCost数据

数据来源:
  - tb_project_major_cost: project_id, code(cost_code), name(cost_name), target_cost_amount
  - tb_project_cost_pay_detail: project_id, cost_actual_confirmed_acc
  - tb_project_cost_unconfirmed_analyse: contract_code, estimate_amount

合并步骤:
  1. 以 tb_project_major_cost 为主表（包含project_id, cost_code, cost_name, target_cost）
  2. LEFT JOIN tb_project_cost_pay_detail ON (project_id + cost_code)
  3. LEFT JOIN tb_project_cost_unconfirmed_analyse ON contract_code
  4. 计算 incurred_cost = cost_actual_confirmed_acc（从步骤2）
  5. 计算 unconfirmed_cost = estimate_amount（从步骤3）
  6. 计算 total_cost = incurred_cost + unconfirmed_cost
  7. 推断 cost_level 从 cost_code（如 "01" → L1, "01-01" → L2）
  8. 补充 report_period（需要额外处理）

注意事项:
  - cost_code 格式需要统一（如 "01" vs "01-01"）
  - 部分记录可能没有关联的付款明细或待确认成本
```

### 5.2 ProjectIndicator 确认逻辑

```yaml
# 确认tb_project_indicator字段含义

原始字段:
  - code: 可能是 project_id
  - title: 指标名称
  - value: 指标值
  - remark: 报告周期

确认步骤:
  1. 查询tb_project_indicator表的实际数据
  2. 确认code字段是否为project_id
  3. 确认remark字段是否为report_period
  4. 如不是，需要创建映射关系

注意事项:
  - 需要与业务人员确认字段含义
  - 如字段含义不符，需要调整本体设计
```

### 5.3 RiskAlert 转换逻辑

```yaml
# 从warning字段转换alert_level

原始数据 (warning):
  - "red" → alert_level: "red"
  - "yellow" → alert_level: "yellow"
  - "green" → alert_level: "green"
  - 其他值 → 需要确认业务规则

转换步骤:
  1. 直接映射 warning → alert_level
  2. 确认warning字段的值域
  3. 如有异常值，记录并与业务人员确认
```
