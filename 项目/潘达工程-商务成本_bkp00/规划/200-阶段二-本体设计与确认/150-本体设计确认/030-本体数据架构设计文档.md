## 一、架构概述

### 1.1 设计目标
设计完整的本体数据架构，为数据对接提供明确的依据。

### 1.2 设计原则
- **相对理想策略**：设计理想架构，同时考虑现实数据源约束
- **可追溯性**：每个本体属性必须能追溯到具体的数据源
- **可扩展性**：设计应支持未来业务扩展
- **qualified_name规范**：属性必须配置 `qualified_name`，格式为 `{CubeName}.{member}`

### 1.3 平台规约要求

| 规约项 | 要求 | 参考 |
|--------|------|------|
| qualified_name | 格式：`CubeName.member`，如 `ProjectCube.project_name` | 《本体规划指南》 |
| semantic_role | 维度用 `dimension`，度量用 `measure` | 《本体规划指南》 |
| bind_source | 每个Object Type必须绑定到具体Cube | 《本体规划指南》 |
| Link Type端点 | Link两端必须是Object Type code，不能是Cube名 | 《本体规划指南》 |

### 1.4 架构层次

```
┌─────────────────────────────────────────────────────────────┐
│                         本体层（Ontology）                    │
│  Object Type / Link Type / Action Type                     │
├─────────────────────────────────────────────────────────────┤
│                      Cube层（平台自动生成）                   │
│  维度 / 度量 / 派生度量                                    │
├─────────────────────────────────────────────────────────────┤
│                        物理表层（Table）                     │
│  理想表结构设计                                            │
└─────────────────────────────────────────────────────────────┘
```
