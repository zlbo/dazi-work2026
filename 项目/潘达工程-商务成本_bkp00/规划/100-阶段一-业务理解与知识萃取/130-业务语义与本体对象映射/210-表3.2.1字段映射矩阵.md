# 210-表3.2.1字段映射矩阵

## 一、映射关系概述

表3.2.1的数据主要来源于:
1. **手动填写**(90%的数据)
2. **表3.2引用**(10%的数据)

表3.2.1是待确认成本的**专项分析表**,数据来源于表3.2的待确认列。

## 三、数据库表设计

### 3.1 tb_project_unconfirmed_cost(待确认成本分析表)

这个表专门用于存储待确认成本的分析数据:

| 字段名 | 类型 | 说明 | 示例 |
|-------|------|------|------|
| id | bigint | 主键 | 自增 |
| project_id | bigint | 项目ID | 外键 |
| year | int | 年份 | 2024 |
| month | int | 月份 | 6 |
| contract_code | varchar(50) | 合同编码 | 'HT-2024-001' |
| contract_name | varchar(200) | 合同名称 | '劳务分包合同' |
| contract_content | varchar(500) | 合同内容 | '主体结构劳务' |
| supplier_name | varchar(200) | 供应商名称 | 'XX劳务公司' |
| contract_amount | decimal(18,2) | 合同总金额 | 5000000.00 |
| payment_ratio | decimal(10,4) | 付款比例 | 0.8000 |
| tax_rate | decimal(10,4) | 税率 | 0.0900 |
| settlement_status | varchar(10) | 结算状态 | '否' |
| unconfirmed_amount | decimal(18,2) | 待确金额(含税) | 200000.00 |
| reason_analysis | varchar(500) | 主要原因分析 | '业主审核滞后' |
| remark | varchar(500) | 备注 | 补充说明 |
| attachment_required | tinyint | 是否需要附件 | 0/1 |
| attachment_status | varchar(20) | 附件状态 | '未上传'/'已上传' |
| created_by | varchar(50) | 创建人 | - |
| created_time | datetime | 创建时间 | - |

**索引设计**:
- PRIMARY KEY (id)
- UNIQUE KEY (project_id, year, month, contract_code)
- INDEX (project_id, year, month)
- INDEX (unconfirmed_amount)

**表间关系**:
```
tb_project_unconfirmed_cost
    ↓ 被引用
表3.2.1 (Excel待确成本分析表)
    ↓ 被引用
表3!CO列 (Excel汇总表)
    ↓ 被引用
表1、表2
```

## 七、特殊情况处理

### 7.1 附件要求判断

| 条件 | 判断标准 | 处理方式 |
|------|---------|---------|
| 附件必需 | J列金额 > 500000 | attachment_required = 1 |
| 附件可选 | J列金额 ≤ 500000 | attachment_required = 0 |

**SQL判断逻辑**:
```sql
CASE 
    WHEN unconfirmed_amount > 500000 THEN 1 
    ELSE 0 
END as attachment_required
```

### 7.2 空值处理

| 场景 | 处理方式 | 说明 |
|------|---------|------|
| 合同编码为空 | 禁止保存 | 必填字段 |
| 合同名称为空 | 禁止保存 | 必填字段 |
| 待确金额为空 | 允许为空(视为0) | 可选 |
| 原因分析为空 | 允许为空 | 可选 |
| 备注为空 | 允许为空 | 可选 |

### 7.3 数据校验

| 校验项 | 校验规则 | 说明 |
|-------|---------|------|
| 待确金额非负 | J列 >= 0 | 不能为负数 |
| 付款比例范围 | 0 <= G列 <= 1 | 付款比例0-100% |
| 税率范围 | 0 <= H列 <= 1 | 税率0-100% |

### 7.4 与表3.2的一致性验证

| 验证项 | 验证公式 | 说明 |
|-------|---------|------|

## 八、SQL查询示例

### 8.1 查询待确成本明细

```sql
SELECT 
    contract_code as 合同编码,
    contract_name as 合同名称,
    supplier_name as 供应商,
    unconfirmed_amount as 待确金额,
    reason_analysis as 原因分析,
    CASE 
        WHEN unconfirmed_amount > 500000 THEN '必需附件'
        ELSE '可选附件'
    END as 附件要求
FROM tb_project_unconfirmed_cost
WHERE project_id = ? 
  AND year = ? 
  AND month = ?
ORDER BY unconfirmed_amount DESC;
```

### 8.2 查询待确成本汇总

```sql
SELECT 
    COUNT(*) as 待确合同数,
    SUM(unconfirmed_amount) as 待确总金额,
    AVG(unconfirmed_amount) as 平均待确金额,
    MAX(unconfirmed_amount) as 最大待确金额
FROM tb_project_unconfirmed_cost
WHERE project_id = ? 
  AND year = ? 
  AND month = ?;
```

### 8.3 查询需要附件的待确成本

```sql
SELECT 
    contract_code as 合同编码,
    contract_name as 合同名称,
    unconfirmed_amount as 待确金额,
    reason_analysis as 原因分析,
    attachment_status as 附件状态
FROM tb_project_unconfirmed_cost
WHERE project_id = ? 
  AND year = ? 
  AND month = ?
  AND unconfirmed_amount > 500000
ORDER BY unconfirmed_amount DESC;
```

### 8.4 验证与表3.2的一致性

```sql
-- 验证表3.2.1与表3.2的待确认成本一致性
SELECT 
    (SELECT SUM(unconfirmed_amount) 
     FROM tb_project_unconfirmed_cost 
     WHERE project_id = ? AND year = ? AND month = ?) as 表3.2.1汇总,
    (SELECT SUM(cost_unconfirmed_acc) 
     FROM tb_project_cost_detail 
     WHERE project_id = ? AND cost_id = ?) as 表3.2汇总,
    (SELECT SUM(unconfirmed_amount) 
     FROM tb_project_unconfirmed_cost 
     WHERE project_id = ? AND year = ? AND month = ?) -
    (SELECT SUM(cost_unconfirmed_acc) 
     FROM tb_project_cost_detail 
     WHERE project_id = ? AND cost_id = ?) as 差异;
```

## 九、附件管理

### 9.1 附件上传要求

| 阈值 | 要求 | 说明 |
|------|------|------|
| >50万元 | 必须上传详细计算式 | 强制性要求 |
| ≤50万元 | 建议上传(可选) | 最佳实践 |

### 9.2 附件内容要求

| 内容项 | 说明 |
|-------|------|
| 工程量计算 | 详细的工程量计算过程 |
| 价格依据 | 单价确定依据 |
| 税额计算 | 税额计算过程 |
| 合计金额 | 含税金额计算 |

### 9.3 附件状态管理

| 状态 | 说明 | 处理 |
|------|------|------|
| 未上传 | 附件未上传 | 催促上传 |
| 已上传 | 附件已上传 | 审核中 |
| 审核通过 | 附件审核通过 | 完成 |
| 审核不通过 | 附件审核不通过 | 重新上传 |

## 十、表3.2.1的核心价值

### 10.1 专项分析价值

1. **原因分析** - K列专门分析待确原因
2. **附件支撑** - 超过50万需提供计算式
3. **明细追溯** - 每条待确成本都有详细记录
4. **汇总验证** - 与表3.2的汇总数据一致性验证

### 10.2 风险预警价值

1. **金额预警** - 自动标记超过50万的待确成本
2. **附件预警** - 未上传附件时提醒
3. **长期预警** - 长期未确认的成本预警
4. **原因预警** - 频繁出现的原因类型预警

### 10.3 不可替代性

1. **原因分析** - 详细的待确原因记录
2. **附件管理** - 强制附件上传机制
3. **明细追溯** - 每条待确成本可追溯
4. **汇总验证** - 与表3.2的一致性保障
