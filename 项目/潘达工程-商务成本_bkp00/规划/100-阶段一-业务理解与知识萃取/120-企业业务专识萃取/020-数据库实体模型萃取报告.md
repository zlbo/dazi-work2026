## 三、表业务含义分析

### 3.1 核心业务表

#### tb_project_indicator（项目指标表）

```yaml
表业务含义:
  表名: tb_project_indicator
  业务含义: 项目指标表
  业务实体: ProjectIndicator
  行业通识对应: L1-实体"指标"
  核心字段:
    - 字段名: project_id
      语义: 项目唯一标识
      数据类型: string
      约束: NOT NULL
      业务含义: 项目的唯一标识符
      行业通识对应: L1-实体"项目"的project_id属性
    - 字段名: indicator_code
      语义: 指标代码
      数据类型: string
      约束: NOT NULL
      业务含义: 指标的唯一代码（如PROFIT_RATE）
      行业通识对应: L1-实体"指标"的indicator_code属性
    - 字段名: indicator_value
      语义: 指标值
      数据类型: decimal
      约束: NULL
      业务含义: 指标的具体数值
      行业通识对应: L1-实体"指标"的indicator_value属性
    - 字段名: report_period
      语义: 报告周期
      数据类型: string
      约束: NOT NULL
      业务含义: 指标所属的报告周期（如2024-01）
      行业通识对应: L1-实体"指标"的report_period属性
    - 字段名: remark
      语义: 备注
      数据类型: string
      约束: NULL
      业务含义: 指标的说明或注释
```

**业务价值**：存储项目关键绩效指标数据，支撑项目绩效评估和决策分析。

## 七、结论与建议

### 7.1 核心发现

#### 7.1.1 数据库结构分析发现

1. **实体完整性**：现有表覆盖了项目指标、成本、风险等核心业务领域，与行业通识骨架高度一致
2. **关系隐式性**：表间关系主要通过字段名关联，缺乏显式外键约束，需要在本体中显式定义
3. **数据类型合理**：字段类型符合业务需求，string类型用于标识，decimal类型用于金额
4. **Project实体缺失**：没有专门的Project表，需要从其他表推断，建议在本体中创建专门的Project Object Type
5. **企业特有规则体现**：数据库结构体现了企业特有的业务规则，如成本三级分类、三色预警等
