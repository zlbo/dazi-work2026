## 一、核心策略选择（通用）

### 1.1 策略定位：相对理想型

本项目采用**"相对理想型"策略**，这是整个项目的基础指导思想：

| 维度 | 说明 |
|------|------|
| **理想化** | 本体架构和数据架构应该支撑业务分析的**完整、科学、面向未来**的状态 |
| **相对** | 理想架构必须基于**现实可获得的数据源**，不能脱离实际 |
| **平衡** | 在理想设计与现实约束之间寻求平衡 |

**策略实施要点**：
1. 基于行业通识和大模型知识构建完整的业务本体模型
2. 避免现有资源的约束，先构建理想版本
3. 后续根据实际资源逐步优化和迭代

### 1.2 规划与实施分离原则

> **核心原则**：本规划文档仅负责**设计层面**的规划，**具体技术实施**（本体建模、数据流程开发等）必须使用架构师提供的标准方法执行。架构师的方法具有最高优先级和权威性。

```
规划阶段（本文档）           实施阶段（架构师方法）
──────────────────           ────────────────────
│ 业务域定义       │  →  │  SDK脚本生成        │
│ 物理表设计       │     │  脚本发布验证        │
│ Cube层设计       │     │  数据接入           │
│ 本体对象定义     │     │  Flow流程开发       │
│ 链接关系定义     │     │  应用开发           │
└──────────────────┘     └───────────────────┘
```

### 1.4 规划与架构师文档关系

> **核心原则**：本项目采用"规划与实施分离"策略，规划文档负责业务设计，架构师文档负责技术规约，两者通过明确的引用机制协同工作。

#### 1.4.1 职责边界

| 维度 | 规划文档（核心纲领） | 架构师文档 |
|------|---------------------|-----------|
| **业务定义** | ✅ 业务域、业务对象、业务关系 | ❌ |
| **架构设计** | ✅ 三层架构设计、数据流向 | ❌ |
| **技术规约** | ❌ | ✅ 命名规范、格式要求、API使用 |
| **实施细节** | ❌ | ✅ SDK脚本编写、发布流程、测试方法 |
| **方法论** | ✅ 业务方法论（五层次蒸馏） | ✅ 技术方法论（脚本编写指南） |

#### 1.4.2 引用机制

规划文档必须通过引用而非重复架构师文档内容：

| 架构师文档 | 引用场景 |
|-----------|---------|
| 《本体规划指南》 | 本体规划流程、文档结构规范 |
| 《本体命名规范》 | Object Type、Cube、物理表命名 |
| 《本体分类规划与SDK扩展方案》 | 平台分类、SDK使用规范 |
| 《本体脚本编写指南》 | 实施阶段脚本编写 |

**引用格式**：`参考《本体规划指南》§3 Cube层设计要求`

#### 1.4.3 优先级规则

1. **架构师文档具有最高优先级**
2. 规划文档与架构师文档冲突时，以架构师文档为准
3. 规划文档可以补充业务特定的规则，但不得违反架构师规范
4. 架构师文档未覆盖的业务场景，由规划文档补充

#### 1.4.4 协作流程

```
规划阶段                          实施阶段
─────────                          ─────────
│ 业务域定义       │  →  │  引用架构师文档        │
│ 架构设计         │     │  SDK脚本生成          │
│ 交付物定义       │     │  脚本发布验证          │
└─────────┘        │     └─────────────────┘
                   ↓
              成果交付给架构师角色
                   ↓
              架构师按照规范实施
```

### 1.5 方法优化：预研机制

> **优化说明**：为平衡理想设计与现实落地，增加**预研机制**作为相对理想型策略的补充

**预研机制核心内容**：
1. **预研时机**：在阶段二本体设计期间，可同步启动数据源预研
2. **预研内容**：数据源盘点、差距分析、可行性评估
3. **预研成果**：作为参考资料，不影响理想设计的科学性
4. **预研价值**：提前识别风险，缩短后续落地周期

**预研约束**：
- 预研成果不得作为本体设计的直接依据
- 理想设计必须保持独立性和科学性
- 预研仅作为落地风险的预判和准备

### 1.4 三层架构的来源关系

```
┌─────────────────────────────────────────────────────────────────────┐
│                         本体层（Ontology）                            │
│  Object Type、Link Type、Action Type                                │
│  ← 基于业务模型设计                                                 │
├─────────────────────────────────────────────────────────────────────┤
│                      Cube层（平台自动生成）                           │
│  维度、度量、聚合                                                    │
│  ← 基于**实际存在的物理表**自动生成                                  │
├─────────────────────────────────────────────────────────────────────┤
│                        物理表层（Table）                             │
│  实际数据库表                                                        │
│  ← 来自：① 现有数据库表 ② Excel台账导入 ③ ETL转换生成 ④ 新建表    │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.4 阶段与步骤的区别

| 层级 | 定义 | 示例 |
|------|------|------|
| **阶段（Phase）** | 项目的大阶段划分，通常5-6个，代表主要里程碑 | 业务理解、本体设计、建模落地 |
| **步骤（Step）** | 每个阶段内的具体操作步骤，是完成阶段目标的必要环节 | 知识蒸馏、数据映射、物理表设计 |

### 1.5 数据来源标记体系规范

> **核心原则**：所有业务知识和规则必须标记来源，确保可追溯性和可信度。采用**轻量级标记体系**，避免过度复杂。

#### 标记维度

| 标记项 | 说明 | 示例 |
|-------|------|------|
| **来源类型** | 标记知识的可信度层级 | `企业业务素材` / `行业通用知识` |
| **来源说明** | 具体来源细节，根据来源类型区分详细程度 | |

#### 不同场景的标记要求

| 场景 | 来源类型 | 来源说明要求 | 示例 |
|------|---------|-----------|------|
| **Excel台账萃取** | 企业业务素材 | 需说明具体文件、报表、单元格/公式 | `Excel台账：潘达建工（房建）商务成本报表模板.xlsx → 表3.1公式分析` |
| **数据库表萃取** | 企业业务素材 | 需说明具体数据库、表名、字段 | `数据库：建工成本01（DuckDB）→ tb_project_indicator表结构分析` |
| **行业通识蒸馏** | 行业通用知识 | 只需标记来源类型，无需详细说明 | `大模型蒸馏：建筑行业通用知识` |
| **业务语义与本体映射** | 混合 | 需标记每个映射关系的来源 | `来源：Excel台账表3.1字段映射` |
| **本体设计建模** | 混合 | 需标记每个属性/关系的来源 | `来源：企业特有规则ER001` |

#### 冲突解决优先级

| 优先级 | 来源类型 | 说明 |
|-------|---------|------|
| 1 | 企业业务素材 | 最高优先级，以企业实际业务为准 |
| 2 | 行业通用知识 | 仅作为参考，需验证后使用 |

## 二、项目阶段划分（共6阶段+阶段零）

> **说明**：以下为标准阶段划分，实际项目可根据需要裁剪或调整。

### 阶段零：空间挂接与平台理解
**目标**：完成项目启动的技术准备工作  
**包含**：
- 数据空间挂接
- 平台架构理解

### 阶段一：业务理解与知识萃取
**目标**：**先通过行业通识蒸馏形成完整的业务知识体系，再基于此萃取企业特有知识**  
**核心策略**：行业通识蒸馏是整个项目的**基础和骨架**，具有提纲挈领的作用，其重要性超过企业专识萃取。通过大模型蒸馏形成完整的行业知识体系，为后续所有工作提供统一的语言、框架和方向。

> **行业领域定位**：本项目的行业领域为**建筑工程行业**（综合型建工企业），而非局限于房建。建筑工程行业涵盖房屋建筑、市政工程、道路桥梁、园林绿化、水利水电、装饰装修等细分领域。本体设计应具备足够的**行业代表性**，确保业务模型的通用性和扩展性。
> 
> **深度蒸馏方法**：具体操作方法参见《900-方法论体系/100-标准规范与方法/020-行业知识深度蒸馏方法论.md》。  
**产出**：行业知识骨架文档、企业业务知识萃取报告、业务实体模型文档

### 阶段二：本体设计与确认
**目标**：设计本体架构，确认Object/Cube/关系  
**产出**：本体设计文档

### 阶段三：本体建模与落地
**目标**：将设计转化为平台可执行的本体模型  
**产出**：本体模型、发布验证报告

#### 执行前检查清单
> **要求**：执行阶段三前必须完成以下检查，确保符合架构师规范

| 检查项 | 检查内容 | 依据文档 |
|--------|---------|---------|
| 架构师文档阅读 | 是否已通读《本体脚本编写指南》《本体命名规范》 | 架构师文档 |
| API规范确认 | 是否理解平台SDK的使用限制（如沙箱环境不支持`open`函数） | 架构师文档 |
| 命令参数验证 | 是否确认所有CLI命令参数格式正确 | 架构师文档 |
| 显示名规范 | 是否确保所有对象的display_name为中文且有意义 | 《本体命名规范》 |
| 维度表完整性 | 是否包含公共维度表（日期表等） | 《本体规划指南》 |
| 业务维度覆盖 | 是否覆盖核心业务维度（业主、部门、地区等） | 阶段二设计文档 |

### 阶段四：数据接入
**目标**：完成数据源映射、接入与对接，数据直接接入平台实际存在的本体建模表  
**产出**：数据源映射分析报告、数据源接入报告、数据对接报告

**数据接入原则**：
1. 数据接入目标为阶段三落地的本体建模表（事实表和维度表）
2. 数据源映射分析需明确标注每个字段的来源和转换规则

### 阶段五：业务应用与分析能力开发
**目标**：基于本体和数据源，开发业务应用与分析能力  
**产出**：业务应用与分析能力

### 阶段六：优化迭代
**目标**：总结成果，持续优化本体模型  
**产出**：本体优化迭代报告

## 五、模板使用说明

> **重要**：本文档为**标准模板**，使用时请遵循以下原则：

### 5.1 详细操作指南
详细操作指南请参考《900-方法论体系/000-核心纲领/020-本体蒸馏萃取与设计方法论.md》

### 5.2 项目-specific 标记
在详细文档中，🔴 标记的内容为**项目-specific**，其他项目复用时必须替换：
- 数据空间ID
- 行业/平台标准
- 可用的企业数据资源列表
- 业务分析材料清单
- 本体物理表到现实数据源的映射

### 5.3 适配步骤
1. 复制本模板及方法论文档
2. 在方法论文档中替换 🔴 标记的项目-specific内容
3. 根据实际项目裁剪或调整阶段（可选）
4. 开始执行
