## 一、物理表设计原则

1. **相对理想型策略**：基于业务需求设计理想表结构，同时考虑现实数据源约束
2. **平台规范对齐**：严格遵循本体平台的Cube自动生成规则
3. **可追溯性**：每个字段必须标记数据来源
4. **扩展性**：设计支持未来业务扩展的表结构

## 二、物理表命名规范（平台标准）

| 表类型 | 命名规则 | 示例 |
|-------|---------|------|
| 业务实体表 | `tb_` + 小写蛇形命名 | tb_project, tb_project_cost |
| 中间表 | `tmp_` + 小写蛇形命名 | tmp_cost_aggregate |
| 维度表 | 平台自动从Cube生成 | 无需手动命名 |
| 事实表 | 平台自动从Cube生成 | 无需手动命名 |

## 五、表间关系设计

| 源表 | 目标表 | 关联字段 | 关系类型 | 业务含义 |
|-----|-------|---------|---------|---------|
| tb_project_output | tb_project | project_id → id | N:1 | 产值属于项目 |
| tb_project_cost | tb_project | project_id → id | N:1 | 成本属于项目 |
| tb_project_indicator | tb_project | project_id → id | N:1 | 指标属于项目 |
| tb_project_risk | tb_project | project_id → id | N:1 | 风险属于项目 |

## 六、Cube设计要求（平台自动生成）

1. **Cube命名规则**：PascalCase + `Cube` 后缀，如 `ProjectCostCube`
2. **自动生成逻辑**：平台将基于物理表自动生成Cube，包含维度和度量
3. **绑定要求**：每个Object Type必须绑定到对应的Cube

## 七、数据源建设策略

| 优先级 | 理想表 | 来源类型 | 建设策略 |
|--------|-------|---------|--------|
| P0 | tb_project_indicator | A（直接使用） | 直接使用现有表 |
| P0 | tb_project_cost | B（合并转换） | 合并tb_project_major_cost、tb_project_cost_pay_detail等表 |
| P0 | tb_project_output | C（Excel导入） | 从Excel表3.1和3.1.1导入数据 |
| P1 | tb_project | B（合并转换） | 从现有表提取+Excel补充项目名称 |
| P1 | tb_project_risk | B（合并转换） | 转换现有风险数据并补充project_id |
| P2 | tb_project_cost_detail | D（新建表） | 新建规范化成本明细表结构 |

## 八、数据质量规则

| 规则ID | 规则描述 | 适用表 | 检查方式 |
|--------|---------|-------|--------|
| DQ001 | project_id不能为空 | 所有表 | NOT NULL约束 |
| DQ002 | report_period格式为YYYY-MM | tb_project_output、tb_project_cost | 正则校验 |
| DQ003 | 数值字段不能为负 | 所有数值字段 | 范围检查 |
| DQ004 | 关联字段必须存在 | 所有关联表 | 外键约束 |
| DQ005 | total_cost = cost_confirmed_acc + cost_unconfirmed_acc | tb_project_cost | 计算校验 |
