## 一、物理表设计原则

1. **相对理想型策略**：基于业务需求设计理想表结构，同时考虑现实数据源约束
2. **平台规范对齐**：严格遵循本体平台的Cube自动生成规则
3. **可追溯性**：每个字段必须标记数据来源
4. **扩展性**：设计支持未来业务扩展的表结构

## 二、物理表命名规范（平台标准）

| 表类型 | 命名规则 | 示例 |
|-------|---------|------|
| 事实表 | `fact_` + 小写蛇形命名 | fact_project_cost, fact_project_output |
| 维度表 | `dim_` + 小写蛇形命名 | dim_project, dim_date |
| 中间表 | `tmp_` + 小写蛇形命名 | tmp_cost_aggregate |

## 六、Cube设计要求（平台自动生成）

1. **Cube命名规则**：PascalCase + `Cube` 后缀，如 `ProjectCostCube`
2. **自动生成逻辑**：平台将基于事实表和维度表自动生成Cube，包含维度和度量
3. **绑定要求**：每个Object Type必须绑定到对应的Cube

## 八、数据质量规则

### 8.1 通用数据质量规则

| 规则ID | 规则描述 | 适用表 | 检查方式 |
|--------|---------|-------|--------|
| DQ001 | project_id不能为空 | 所有表 | NOT NULL约束 |
| DQ002 | report_period格式为YYYY-MM | fact_project_output、fact_project_cost | 正则校验 |
| DQ003 | 数值字段不能为负 | 所有数值字段 | 范围检查 |
| DQ004 | 关联字段必须存在 | 所有关联表 | 外键约束 |
| DQ005 | total_cost = cost_confirmed_acc + cost_unconfirmed_acc | fact_project_cost | 计算校验 |

### 8.2 业务特定质量规则

| 规则ID | 规则描述 | 适用表 | 检查方式 |
|--------|---------|-------|--------|
| DQ006 | bond_amount = returned_amount + unreturned_amount | fact_bond | 计算校验 |
| DQ007 | penalty_amount > 0 | fact_penalty | 范围检查 |
| DQ008 | due_date > payment_date | fact_bond | 时间逻辑校验 |
| DQ009 | expiry_date > purchase_date | fact_insurance | 时间逻辑校验 |
| DQ010 | settlement_amount > 0 | fact_settlement | 范围检查 |
| DQ011 | three_estimate优先级：概算 >= 预算 >= 决算 | fact_settlement | 业务规则校验 |
| DQ012 | cost_level IN ('L1', 'L2', 'L3') | fact_project_cost | 枚举值校验 |
| DQ013 | warning_level IN ('绿', '黄', '红') | fact_project_risk | 枚举值校验 |
| DQ014 | overdue_days >= 0 | fact_receipt | 范围检查 |
| DQ015 | receipt_ratio = receipt_amount / contract_amount <= 1 | fact_receipt | 计算校验 |
