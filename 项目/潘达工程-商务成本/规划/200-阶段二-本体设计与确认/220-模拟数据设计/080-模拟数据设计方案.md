### 1.4 业务主体说明（核心设定）

> **核心原则**：本方案模拟的所有业务数据，均属于**潘达建工集团**这唯一主体。

**主体架构**：
- **集团总部**：潘达建工集团有限公司（唯一法人主体）
- **分子公司**：5个专业子公司（房建公司一、市政公司、安装公司、基础设施公司、投资公司）
- **项目部**：100个项目部（每个项目对应一个）

**关键说明**：
1. **唯一性**：所有项目、合同、成本数据的归属主体只有一个——**潘达建工集团**
2. **业主≠主体**：业主是项目发包方（如华润置地、地方城投等），不是业务主体
3. **客户≠主体**：客户是服务对象，不是业务主体
4. **组织层级**：分子公司、项目部均为潘达建工集团内部组织，不是独立主体

**业务关系示意**：
```
                    ┌─────────────────────┐
                    │   潘达建工集团       │ ← 唯一业务主体
                    │   （法人）           │
                    └──────────┬──────────┘
                               │ 下辖
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
    ┌──────────┐        ┌──────────┐        ┌──────────┐
    │ 房建公司 │        │ 市政公司 │        │ 安装公司 │
    └────┬─────┘        └────┬─────┘        └────┬─────┘
         │                   │                   │
    ┌────┴────┐        ┌────┴────┐        ┌────┴────┐
    │项目部A  │        │项目部B  │        │项目部C  │
    │项目部D  │        │项目部E  │        │项目部F  │
    └─────────┘        └─────────┘        └─────────┘
```

**业主与项目关系（非主体关系）**：
- 业主：项目发包方（如央企、地方平台、民企等）
- 潘达建工：项目承包方（唯一主体）
- 示例：华润置地（业主）将"华润中心二期"项目发包给潘达建工（承包方）

> **重要区分**：
> - **业务主体**：潘达建工集团（唯一）
> - **业主/客户**：外部合作方（多样）
> - **组织**：潘达建工内部机构（分子公司、项目部）

### 1.6 种子数据仿真度要求

> **核心原则**：种子数据虽然只是基础框架数据，但它是模拟数据的地基，必须满足仿真度要求。

#### 1.6.1 数值精度要求

| 数据类型 | 单位 | 精度要求 | 示例 |
|---------|------|---------|------|
| 合同金额 | 元 | 保留两位小数 | 125,683,500.00元 |
| 项目金额 | 元 | 保留两位小数 | 850,000,000.00元 |
| 预算成本 | 元 | 保留两位小数 | 765,000,000.00元 |
| 比率 | - | 保留4位小数 | 0.1578 |
| 面积 | ㎡ | 保留两位小数 | 35,678.90㎡ |

**严禁**：
- 出现 100,000,000 这样的整数（应该是 100,000,000.00）
- 出现 5亿 这样的简化值（应该是 500,000,000.00元）

#### 1.6.2 多样性要求

| 实体 | 维度 | 分布要求 |
|------|------|---------|
| **项目** | 类型 | 房建30/市政30/安装25/基础设施15 |
| | 合作模式 | 直营50/合作20/挂靠15/联合体10/PPP5 |
| | 状态 | 在建60/竣工30/其他10 |
| **业主** | 企业性质 | 国企40/民企50/混改10 |
| | 信用等级 | AAA15/AA+20/AA25/A-BBB25/BB及以下15 |
| **合同** | 类型 | 施工总承包50/专业分包27/劳务分包17/其他6 |
| | 状态 | 已签订20/执行中45/变更中10/结算中15/已关闭8/终止2 |
| **成本科目** | 层级 | L1:4/L2:12/L3:34 |
| **地区** | 区域 | 华东30/华南25/华中15/西南15/其他15 |

#### 1.6.3 经营品质差异化要求

> **核心原则**：种子数据中必须预留经营品质差异化字段。

**项目品质等级分布**：

| 等级 | 数量 | 占比 | 特征 |
|------|------|------|------|
| A级 | 10个 | 10% | 工期0偏差、成本0超支、无风险 |
| B级 | 25个 | 25% | 工期<5%偏差、成本<3%超支 |
| C级 | 35个 | 35% | 工期5-10%偏差、成本3-8%超支 |
| D级 | 20个 | 20% | 工期10-20%偏差、成本8-15%超支 |
| E级 | 10个 | 10% | 工期>20%偏差、成本>15%超支 |

**业主-项目品质关联**：

| 业主类型 | 关联项目品质 | 付款行为 |
|---------|-------------|---------|
| 央企/优质国企 | A/B级为主 | 付款及时 |
| 地方平台 | B/C级为主 | 付款周期较长 |
| 优质民企 | C级为主 | 付款正常 |
| 问题民企 | D/E级为主 | 拖欠严重 |

#### 1.5.4 字段设计建议（为阶段四预留）

#### 1.5.5 阶段衔接说明

## 二、总体规模与时间周期设计

### 2.0 数值精度与波动规范

#### 2.0.1 数值精度与单位规范

> **核心原则**：所有金额统一使用**元**作为单位，保留两位小数，避免"整数感"，模拟真实交易场景。

**数值精度规则**：

| 数据类型 | 单位 | 精度要求 | 示例 |
|---------|------|---------|------|
| 合同金额 | 元 | 保留两位小数 | 125,683,500.00元 |
| 成本金额 | 元 | 保留两位小数 | 8,523,168.36元 |
| 产值金额 | 元 | 保留两位小数 | 15,236,892.50元 |
| 付款金额 | 元 | 保留两位小数 | 5,623,158.75元 |
| 单价 | 元 | 保留4位小数 | 5,680.3500元/吨 |
| 比率 | - | 保留4位小数 | 0.1578 |
| 面积 | ㎡ | 保留两位小数 | 35,678.90㎡ |

#### 2.0.2 时间维度无规律波动保障

> **核心原则**：意外事件不能形成规律，必须是随机发生的。从时间线看，极端事件是偶然发生的，不是每月都必然发生。

**波动事件概率分布**：

| 波动类型 | 定义 | 年发生概率 | 月度发生概率 | 示例 |
|---------|------|-----------|-------------|------|
| **正常波动** | 符合正态分布的常规波动 | 100% | 100% | 合同金额在均值±20%内 |
| **正向极端** | 显著高于常规的异常值 | 20-30% | 2-3% | 突然签10亿大单 |
| **负向极端** | 显著低于常规的异常值 | 10-15% | 1-2% | 突然签500万小单 |
| **结构性变化** | 业务模式变化导致 | 5-10% | 0.5-1% | 安全事故、政策变化 |

**业务波动场景示例**：

| 场景 | 正常情况 | 波动表现 | 数据表现（元） | 年发生概率 |
|------|---------|---------|--------------|-----------|
| 春节后开工 | 月产值1亿 | 2月产值突然下降 | 32,000,000.00 | 100%（季节性） |
| 行业旺季 | 月产值1亿 | 9月产值突然爆发 | 280,000,000.00 | 80% |
| 大客户签约 | 月签3个合同 | 突然签了个央企大单 | 800,000,000.00 | 20% |
| 材料涨价 | 钢材成本5000/吨 | 突然涨价到6500 | 成本飙升30% | 30% |
| 安全事故 | 零事故 | 突然发生重大事故 | 停产3个月 | 5% |

| 品质等级 | 数量 | 项目类型分布 | 核心特征 |

## 九、数据验证方案

### 9.1 技术验证

```sql
-- 验证1：勾稽关系 - 产值确认
SELECT COUNT(*) as violation_count
FROM fact_project_output
WHERE total_amount != confirmed_amount + unconfirmed_amount;

-- 验证2：勾稽关系 - 成本汇总
SELECT COUNT(*) as violation_count
FROM fact_project_cost
WHERE total_cost != (
    SELECT SUM(cost_amount) 
    FROM fact_project_cost_detail 
    WHERE fact_project_cost_detail.project_key = fact_project_cost.project_key
);

-- 验证3：约束条件 - 金额非负
SELECT COUNT(*) as violation_count
FROM fact_project_cost
WHERE cost_amount < 0;
```

### 9.2 经营品质验证

```sql
-- 验证：经营品质分布
SELECT 
    CASE 
        WHEN profit_rate >= 0.15 THEN 'A级'
        WHEN profit_rate >= 0.10 THEN 'B级'
        WHEN profit_rate >= 0.05 THEN 'C级'
        WHEN profit_rate >= 0 THEN 'D级'
        ELSE 'E级'
    END as quality_level,
    COUNT(*) as project_count,
    ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER(), 2) as percentage
FROM fact_project_indicator
GROUP BY quality_level;
-- 期望结果：A级10%, B级25%, C级35%, D级20%, E级10%

-- 验证：问题覆盖率
SELECT 
    COUNT(DISTINCT project_key) as affected_projects
FROM fact_receivable
WHERE overdue_days > 90;
-- 期望：D/E级80%以上项目有逾期
```
