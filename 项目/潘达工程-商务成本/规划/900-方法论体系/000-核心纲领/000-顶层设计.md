## 一、核心策略选择（通用）

### 1.1 策略定位：相对理想型

本项目采用**"相对理想型"策略**，这是整个项目的基础指导思想：

| 维度 | 说明 |
|------|------|
| **理想化** | 本体架构和数据架构应该支撑业务分析的**完整、科学、面向未来**的状态 |
| **相对** | 理想架构必须基于**现实可获得的数据源**，不能脱离实际 |
| **平衡** | 在理想设计与现实约束之间寻求平衡 |

**策略实施要点**：
1. 基于行业通识和大模型知识构建完整的业务本体模型
2. 避免现有资源的约束，先构建理想版本
3. 后续根据实际资源逐步优化和迭代

### 1.2 规划与实施分离原则

> **核心原则**：本规划文档仅负责**设计层面**的规划，**具体技术实施**（本体建模、数据流程开发等）必须使用架构师提供的标准方法执行。架构师的方法具有最高优先级和权威性。

```
规划阶段（本文档）           实施阶段（架构师方法）
──────────────────           ────────────────────
│ 业务域定义       │  →  │  SDK脚本生成        │
│ 物理表设计       │     │  脚本发布验证        │
│ Cube层设计       │     │  数据接入           │
│ 本体对象定义     │     │  Flow流程开发       │
│ 链接关系定义     │     │  应用开发           │
└──────────────────┘     └───────────────────┘
```

### 1.4 规划与架构师文档关系

> **核心原则**：本项目采用"规划与实施分离"策略，规划文档负责业务设计，架构师文档负责技术规约，两者通过明确的引用机制协同工作。

#### 1.4.1 职责边界

| 维度 | 规划文档（核心纲领） | 架构师文档 |
|------|---------------------|-----------|
| **业务定义** | ✅ 业务域、业务对象、业务关系 | ❌ |
| **架构设计** | ✅ 三层架构设计、数据流向 | ❌ |
| **技术规约** | ❌ | ✅ 命名规范、格式要求、API使用 |
| **实施细节** | ❌ | ✅ SDK脚本编写、发布流程、测试方法 |
| **方法论** | ✅ 业务方法论（五层次蒸馏） | ✅ 技术方法论（脚本编写指南） |

#### 1.4.2 引用机制

规划文档必须通过引用而非重复架构师文档内容：

| 架构师文档 | 引用场景 |
|-----------|---------|
| 《本体规划指南》 | 本体规划流程、文档结构规范 |
| 《本体命名规范》 | Object Type、Cube、物理表命名 |
| 《本体分类规划与SDK扩展方案》 | 平台分类、SDK使用规范 |
| 《本体脚本编写指南》 | 实施阶段脚本编写 |

**引用格式**：`参考《本体规划指南》§3 Cube层设计要求`

#### 1.4.3 优先级规则

1. **架构师文档具有最高优先级**
2. 规划文档与架构师文档冲突时，以架构师文档为准
3. 规划文档可以补充业务特定的规则，但不得违反架构师规范
4. 架构师文档未覆盖的业务场景，由规划文档补充

#### 1.4.4 协作流程

```
规划阶段                          实施阶段
─────────                          ─────────
│ 业务域定义       │  →  │  引用架构师文档        │
│ 架构设计         │     │  SDK脚本生成          │
│ 交付物定义       │     │  脚本发布验证          │
└─────────┘        │     └─────────────────┘

                   ↓

```

### 1.5 阶段规划文档与快速启动文档的协作关系

> **核心原则**：阶段规划文档与快速启动文档是**互补关系**，必须同时使用，缺一不可。忽略快速启动文档将严重影响建模实施。

#### 1.5.1 两类文档的定位

| 文档类型 | 定位 | 用途 | 来源 | 适用范围 |
|---------|------|------|------|---------|
| **阶段规划文档** | 项目执行计划 | 定义"做什么"：具体步骤、产出物、时间安排 | 项目特定规划 | 仅适用于当前项目 |
| **快速启动文档** | 通用操作指南 | 定义"怎么做"：SDK API规范、CLI命令模板、任务模式 | 系统自动生成（架构师定义标准） | 所有本体实现 |

#### 1.5.2 两类文档的职责分工

| 维度 | 阶段规划文档负责 | 快速启动文档负责 |
|------|-----------------|-----------------|
| **步骤定义** | ✅ 具体步骤分解、依赖关系、产出物 | ❌ |
| **产出物清单** | ✅ 具体文件名、数量、内容要求 | ❌ |
| **依据文档** | ✅ 项目特定的设计文档清单 | ❌ |
| **检查清单** | ✅ 项目特定的门禁检查项 | ❌ |
| **风险分析** | ✅ 项目特定的风险项及应对措施 | ❌ |
| **SDK API规范** | ❌ | ✅ 禁止操作、推荐操作、API使用规范 |
| **CLI命令模板** | ❌ | ✅ 发布、执行、验证的命令格式 |
| **任务模式定义** | ❌ | ✅ 独立规划、对照实施、快速演示三种模式 |
| **错误处理指引** | ❌ | ✅ 脚本运行常见错误及处理方法 |

#### 1.5.3 实施时必须同时参考

> **强制要求**：实施阶段必须同时参考两类文档，缺一不可。

**阅读顺序**：
1. **先读阶段规划文档**：了解具体步骤和产出物
2. **再读快速启动文档**：了解SDK API规范和CLI命令格式
3. **最后按阶段规划文档的步骤执行**：遵循快速启动文档的规范

**示例**：
```
实施本体初始化脚本时：
1. 从阶段规划文档了解：需要编写 panda_cost_ontology_init.py，包含23张物理表、9个Cube等
2. 从快速启动文档了解：必须使用 register_cube API，禁止使用 s.cubes.upsert
3. 编写脚本时：按照阶段规划文档的内容要求，遵循快速启动文档的API规范
```

#### 1.5.4 不冲突的原因

- **阶段规划文档定义"做什么"**：具体步骤、产出物、时间安排
- **快速启动文档定义"怎么做"**：操作规范、CLI命令、SDK API使用规范
- **两者侧重点完全不同，不存在冲突**

#### 1.5.5 忽略快速启动文档的后果

> **严重警告**：忽略快速启动文档将导致以下问题：

| 问题类型 | 具体后果 | 影响程度 |
|---------|---------|---------|
| **API规范违反** | 使用禁止的API（如`sync_metrics`、`s.cubes.upsert`）导致脚本失败 | 高 |
| **函数输出格式错误** | 使用`output.print_json()`而非`return _ontology_fn_body(p)`导致函数无法运行 | 高 |
| **SQL聚合错误** | 对`query_one()`调用`.get()`导致数据获取失败 | 中 |
| **CLI命令格式错误** | 参数格式不正确导致发布失败 | 中 |
| **任务模式混淆** | 不理解独立规划、对照实施、快速演示的区别，导致工作流程混乱 | 高 |

#### 1.5.6 文档协作流程

```
阶段规划阶段                          实施阶段
─────────────                          ─────────
                    ↓
              阅读阶段规划文档
              （了解"做什么"）
                    ↓
              阅读快速启动文档
              （了解"怎么做"）
                    ↓
              编写脚本
              （内容来自阶段规划，规范来自快速启动）

              （命令格式来自快速启动）

              （验证标准来自阶段规划）
```

### 1.6 方法优化：预研机制

> **优化说明**：为平衡理想设计与现实落地，增加**预研机制**作为相对理想型策略的补充

**预研机制核心内容**：
1. **预研时机**：在阶段二本体设计期间，可同步启动数据源预研
2. **预研内容**：数据源盘点、差距分析、可行性评估
3. **预研成果**：作为参考资料，不影响理想设计的科学性
4. **预研价值**：提前识别风险，缩短后续落地周期

**预研约束**：
- 预研成果不得作为本体设计的直接依据
- 理想设计必须保持独立性和科学性
- 预研仅作为落地风险的预判和准备

| 层级 | 定义 | 示例 |
|------|------|------|

### 1.5 数据来源标记体系规范

> **核心原则**：所有业务知识和规则必须标记来源，确保可追溯性和可信度。采用**轻量级标记体系**，避免过度复杂。

#### 标记维度

| 标记项 | 说明 | 示例 |
|-------|------|------|
| **来源类型** | 标记知识的可信度层级 | `企业业务素材` / `行业通用知识` |
| **来源说明** | 具体来源细节，根据来源类型区分详细程度 | |

#### 不同场景的标记要求

| 场景 | 来源类型 | 来源说明要求 | 示例 |
|------|---------|-----------|------|
| **Excel台账萃取** | 企业业务素材 | 需说明具体文件、报表、单元格/公式 | `Excel台账：潘达建工（房建）商务成本报表模板.xlsx → 表3.1公式分析` |
| **数据库表萃取** | 企业业务素材 | 需说明具体数据库、表名、字段 | `数据库：建工成本01（DuckDB）→ tb_project_indicator表结构分析` |
| **行业通识蒸馏** | 行业通用知识 | 只需标记来源类型，无需详细说明 | `大模型蒸馏：建筑行业通用知识` |
| **业务语义与本体映射** | 混合 | 需标记每个映射关系的来源 | `来源：Excel台账表3.1字段映射` |
| **本体设计建模** | 混合 | 需标记每个属性/关系的来源 | `来源：企业特有规则ER001` |

#### 冲突解决优先级

| 优先级 | 来源类型 | 说明 |
|-------|---------|------|
| 1 | 企业业务素材 | 最高优先级，以企业实际业务为准 |
| 2 | 行业通用知识 | 仅作为参考，需验证后使用 |

### 1.6 编码标准规范

> **核心原则**：本项目采用**"结构编码通识优先，属性值专识优先"**的编码策略，确保行业通用性和企业业务准确性的平衡。

#### 1.6.1 编码分类与优先级

| 编码类型 | 优先级 | 说明 | 示例 |
|---------|-------|------|------|
| **结构编码** | 通识优先 | Object Type、Link Type、维度表结构等框架性编码 | `Project`、`ProjectCost`、`dim_project` |
| **业务编码** | 通识优先，参考专识 | 科目编码、指标编码等业务分类编码 | 成本科目：`01-人工成本`、`02-材料成本` |
| **属性值** | 专识优先 | 具体的科目名称、指标名称等业务术语 | `钢筋`、`商品混凝土` |

#### 1.6.2 编码规则

**结构编码规则**：
- Object Type：PascalCase，无前缀（如 `Project`）
- Link Type：PascalCase，无前缀（如 `belongsTo`）
- 物理表：`fact_*`（事实表）、`dim_*`（维度表）
- Cube：PascalCase + `Cube` 后缀（如 `ProjectCostCube`）

**业务编码规则**：
- 层级编码：采用数字层级结构（如 `L1-L2-L3`）
- 编码格式：`[层级编码]-[业务含义]`
- 编码长度：统一为2位或4位数字

**示例**：
```
成本科目编码（通识优先）：
L1: 01-人工成本, 02-材料成本, 03-机械成本, 04-管理费用
L2: 0101-基本工资, 0102-绩效工资, 0201-主材, 0202-辅材
L3: 020101-钢筋, 020102-混凝土, 020103-水泥

企业现有科目名称映射（专识优先）：
020101 → "钢筋"（企业术语）
020102 → "商品混凝土"（企业术语，非通用术语"商砼"）
```

#### 1.6.3 编码冲突解决

当企业现有编码与行业标准编码冲突时：
1. **结构编码**：采用行业标准编码
2. **业务编码**：采用行业标准编码，企业编码仅作参考
3. **属性值**：采用企业实际业务术语

## 二、项目阶段划分（共6阶段+阶段零）

> **说明**：以下为标准阶段划分，实际项目可根据需要裁剪或调整。

### 阶段零：空间挂接与平台理解
**目标**：完成项目启动的技术准备工作  
**包含**：
- 数据空间挂接
- 平台架构理解

### 阶段一：业务理解与知识萃取
**目标**：**先通过行业通识蒸馏形成完整的业务知识体系，再基于此萃取企业特有知识**  
**核心策略**：行业通识蒸馏是整个项目的**基础和骨架**，具有提纲挈领的作用，其重要性超过企业专识萃取。采用**融合式组织策略**，企业专识内容融合到核心知识体系和量化规则库中。

> **行业领域定位**：本项目的行业领域为**建筑工程行业**（综合型建工企业），而非局限于房建。建筑工程行业涵盖房屋建筑、市政工程、道路桥梁、园林绿化、水利水电、装饰装修等细分领域。本体设计应具备足够的**行业代表性**，确保业务模型的通用性和扩展性。
>
