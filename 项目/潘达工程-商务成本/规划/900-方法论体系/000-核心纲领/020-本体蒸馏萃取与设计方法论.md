## 二、行业领域定位

> **引用**：《020-行业知识深度蒸馏方法论.md》§1.3「行业领域定义」

**建筑工程行业定位**：本项目以**建筑工程行业**（综合型建工企业）为核心定位，业务模型应具备足够的**行业代表性**和**通用性**。

**行业覆盖范围**：

| 细分领域 | 核心业务 | 本体设计考量 |
|---------|---------|-------------|
| 房屋建筑 | 土建工程、安装工程 | 基础核心，业务最复杂 |
| 市政工程 | 道路、桥梁、管网、绿化 | 需考虑政府业主特点 |
| 水利水电 | 水利设施、电力设施 | 需考虑长周期和特殊成本结构 |
| 公路工程 | 道路、隧道、桥梁 | 与房建有相似性 |
| 铁路工程 | 线路、隧道、桥梁、站场 | 需考虑线性工程特点和多专业协同 |

**业务通用性原则**：
- 核心业务概念（如项目、产值、成本、合同）应适用于所有细分领域
- 特有概念作为扩展属性处理
- 本体设计优先考虑通用性，扩展性通过继承机制实现

## 三、核心策略：相对理想型

> **引用**：《000-顶层设计.md》§1.1「核心策略」

**这是本项目的核心指导思想**，必须贯穿所有阶段：

| 维度 | 说明 |
|------|------|
| **理想化** | 本体架构和数据架构应该支撑企业商务成本分析的**完整、科学、面向未来**的状态 |
| **相对** | 理想架构必须基于**现实可获得的数据源**，不能脱离实际 |
| **平衡** | 在理想设计与现实约束之间寻求平衡 |

### 3.1 三层架构的来源关系

```
┌──────────────────────────────────────────────────────────────────────┐
│                         本体层（Ontology）                            │
│  Object Type、Link Type、Action Type                                │
│  ← 基于理想业务模型设计                                              │
├──────────────────────────────────────────────────────────────────────┤
│                      Cube层（平台自动生成）                           │
│  维度、度量、聚合                                                    │
│  ← 基于**实际存在的物理表**自动生成                                  │
├──────────────────────────────────────────────────────────────────────┤
│                        物理层（Table）                               │
│  ┌─────────────────┐    ┌─────────────────┐                        │
│  │  【理想表】      │    │  【实际存在的表】│                        │
│  │  业务需求设计   │    │  现有数据库表    │                        │
│  └────────┬────────┘    └────────┬────────┘                        │
│           └──────────┬───────────┘                                  │
│                      ↓                                              │
│              【数据源映射分析】← 核心！必须明确！                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 3.2 关键原则

- **先设计理想，再找现实**：先明确"应该有什么数据"，再分析"有什么数据可用"
- **差距分析是核心**：`理想表` 与 `现实表` 之间的差距，必须有明确的填补方案
- **数据来源必须可追溯**：每个本体属性必须能追溯到具体的数据源

## 五、知识蒸馏框架

> **引用**：《020-行业知识深度蒸馏方法论.md》§二「平台能力与知识映射」

### 5.1 六层次知识蒸馏框架

行业通识蒸馏采用六层次框架（L6为新增）：

| 层次 | 名称 | 说明 | 平台能力映射 |
|------|------|------|------------|
| **L1** | 基础业务知识 | 概念、流程、术语 | Object Type（业务实体） |
| **L2** | 显性业务关系 | 实体关联、数据流向 | Link Type（包含、归属、关联） |
| **L3** | 隐性业务关系 | 经验常识、关联规则 | 派生属性/计算规则 |
| **L4** | 行业独特诀窍 | 特有规则、门道 | 企业特有规则（ER） |
| **L5** | 行业通用流程规则 | 标准化流程、审批规则 | Action Type（业务动作） |
| **L6** | **维度识别**（新增） | 分类体系、编码体系 | 维度表（dim_*） |

### 5.2 深度蒸馏操作流程

> **执行指南**：具体的蒸馏操作步骤、提示词模板参见《020-行业知识深度蒸馏方法论.md》§三「深度蒸馏流程设计」

**五轮迭代蒸馏法**：

| 轮次 | 覆盖层次 | 核心目标 |
|------|---------|---------|
| 第一轮 | L1+L2 | 快速建立完整的业务世界骨架 |
| 第二轮 | L3+L4 | 深入挖掘隐性知识和行业诀窍 |
| 第三轮 | L5 | 完善流程规则和业务动作库 |
| 第四轮 | L6 | 识别需要独立维护的维度表和编码体系 |
| 第五轮 | 冲突解决 | 与企业实际数据对照，解决冲突 |

## 六、数据来源标记体系规范

> **核心原则**：所有业务知识和规则必须标记来源，确保可追溯性和可信度。

### 6.1 标记维度定义

| 标记项 | 说明 | 取值范围 |
|-------|------|---------|
| **来源类型** | 标记知识的可信度层级 | `企业业务素材` / `行业通用知识` |
| **来源说明** | 具体来源细节，根据来源类型区分详细程度 | 文本描述 |

### 6.2 不同场景的标记方法

| 场景 | 来源类型 | 来源说明要求 | 示例 |
|------|---------|-----------|------|
| **行业通识蒸馏** | 行业通用知识 | 只需标记"大模型蒸馏：[行业名称]" | `行业通用知识（大模型蒸馏：建筑行业）` |
| **Excel台账萃取** | 企业业务素材 | 需说明具体文件、报表、单元格/公式 | `Excel台账：潘达建工商务成本报表模板.xlsx → 表3.1公式分析` |
| **数据库表萃取** | 企业业务素材 | 需说明具体数据库、表名、字段 | `数据库：建工成本01（DuckDB）→ tb_project_indicator表结构分析` |
| **本体设计建模** | 混合 | 需标记每个属性/关系的来源 | `来源：企业特有规则ER001` |

### 6.3 编码标准规范

> **引用**：《070-编码标准规范.md》§二「编码分类与优先级」

| 编码类型 | 优先级 | 说明 |
|---------|-------|------|
| **结构编码** | 通识优先 | Object Type、Link Type、维度表结构等框架性编码 |
| **业务编码** | 通识优先，参考专识 | 科目编码、指标编码等业务分类编码 |
| **属性值** | 专识优先 | 具体的科目名称、指标名称等业务术语 |

### 6.4 冲突解决原则

1. **结构编码通识优先**：框架性编码采用行业标准编码
2. **属性值专识优先**：具体业务术语采用企业实际用语
3. **业务编码通识优先，参考专识**：科目编码采用行业标准编码，企业编码仅作参考
4. **最新数据优先**：当不同来源的企业业务素材冲突时，以最新数据为准
5. **业务确认优先**：当数据存在歧义时，需与业务人员确认

## 七、核心任务与交付物

### 7.1 阶段一：业务理解与知识萃取

> **核心策略优化**：阶段一采用**"融合式组织"**策略，将行业通识与企业专识融合到统一的文档体系中，而非分开管理。

#### 7.1.1 文件结构设计原则

**原则1：成果性质优先于工作流程**
- 文件按**成果性质**划分（核心知识体系、量化规则库），而非按工作流程阶段划分
- 原因：成果性质划分便于后续阶段的引用和检索，避免重复查找

**原则2：企业专识融合而非独立**
- 企业专识内容融合到核心知识体系和量化规则库中
- 原因：企业特有规则（如资金月度批复、产值双轨制）与行业规则存在关联，独立管理会导致关联断裂

**原则3：建立双向关联机制**
- 流程、诀窍、规则库之间建立双向关联（关联流程、关联规则字段）
- 原因：便于后续本体设计时快速定位相关规则和流程

|---------|---------|------|

| **企业特有规则** | 融合到量化规则库 | ER001-ER004（资金月度批复、产值双轨制、成本三级分类、风险三色预警） |
| **企业特有指标** | 融合到业务指标清单 | MI026（收款率）、MI027（资金计划准确率） |
| **企业特有预警** | 融合到预警规则清单 | FR-E004（资金计划偏差预警） |

**融合原则**：
- 企业特有内容必须标注来源类型为"企业业务素材"
- 企业特有规则必须与行业规则建立关联（如ER001关联BR-E004）
- 企业特有内容必须说明适用范围（如"仅适用于潘达建工"）

## 八、架构师规范强制检查

> **强制要求**：所有设计产出必须符合架构师规范，以下为必检项：

| 规范项 | 检查内容 | 依据文档 |
|--------|---------|---------|
| **物理表命名规范** | 所有实体必须映射到 `fact_*`（事实表）或 `dim_*`（维度表） | 本体命名规范 §2 |
| **dim_date 强制关联** | 所有事实表必须包含 `date_key` 字段并关联 `dim_date` | 本体命名规范 §2.3 |
| **Cube 层独立设计** | 指标体系必须明确映射到独立的 Cube | 本体规划指南 |
| **Cube 命名规范** | Cube 名必须为 PascalCase + `Cube` 后缀 | 本体命名规范 §4 |
| **Object Type 命名** | Object Type 的 `code` 必须为无前缀 PascalCase | 本体命名规范 §5 |
| **链接语义** | Link Type 的两端必须是 Object Type `code` | 本体规划指南 |
