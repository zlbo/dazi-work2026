## 问题描述

### 现象
在开发「产值同步」流程的 `database-source` 节点时，发现文档说明与平台实际行为不一致：

**文档说明**（`资源/datasources/建工成本01/建工成本01.md` 第21-32行）：
```json
{
"type": "database-source",
"data": {
    "connectionId": "duckdb__建工成本01",
    "output_variable_name": "查询结果",
    "code": { "sql": "SELECT * FROM your_table LIMIT 100" }
}
}
```

**平台实际要求**（经过多次测试验证）：
```json
{
"type": "database-source",
"data": {
    "connectionId": "duckdb__建工成本01",
    "output_variable_name": "output_raw_data",
    "queryConfig": {
    "tableName": "tb_project_income_outcome_summary",
    "selectedFields": [...]
    },
    "tableName": "tb_project_income_outcome_summary"
}
}
```

### 关键差异（已修正）

| 配置项 | 文档说明 | 平台实际行为 | 最终确认 |
| --- | --- | --- | --- |
| 数据源标识 | `connectionId`（字符串） | `dataSourceId`（数字ID） | **`connectionId`（字符串）** |
| SQL定义方式 | 通过 `code.sql` | 通过 `queryConfig` 配置 | **通过 `queryConfig` 配置** |

## 问题影响

1. **开发效率**：开发者按文档配置会导致「数据连接不存在」错误，需要额外调试
2. **错误排查**：错误信息不明确，难以快速定位问题根源
3. **一致性风险**：文档与实现不同步，增加维护成本

## 最终解决方案

经过多次测试验证，**`connectionId`（字符串形式）是正确的配置方式**：

```json
{
"type": "database-source",
"data": {
    "label": "读取产值数据",
    "type": "database-source",
    "flowId": 63,
    "output_variable_name": "output_raw_data",
    "connectionId": "duckdb__建工成本01",
    "queryConfig": {
    "tableName": "tb_project_income_outcome_summary",
    "selectedFields": ["id", "project_id", ...],
    "useAliases": true,
    "distinct": false,
    "rowLimit": 200000,
    "filters": []
    },
    "tableName": "tb_project_income_outcome_summary"
}
}
```

**关键要点**：
- `connectionId` 值必须与数据源文档中定义的一致（如 `"duckdb__建工成本01"`）
- 使用 `queryConfig` 配置查询，而不是 `code.sql`
- 确保 `selectedFields` 中的字段名与源表完全一致

## 附加问题：数据类型转换

在数据写入 `dataspace-sink` 节点时，发现需要将 numpy 类型转换为 Python 原生类型，否则会导致 ClickHouse 写入失败：

**错误信息**：`object of type 'numpy.float64' has no len()`

**解决方案**：在 Python 节点中确保输出数据都是 Python 原生类型：
```python
# 使用 tolist() 转换
target_df['unconfirmed_output'] = result_df['company_amount'].fillna(0.0).tolist()
```
