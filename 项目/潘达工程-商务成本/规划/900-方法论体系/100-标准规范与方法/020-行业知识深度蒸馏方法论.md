### 1.3 行业领域定义

> **建筑工程行业定位**：本项目以**建筑工程行业**（综合型建工企业）为核心定位，业务模型应具备足够的**行业代表性**和**通用性**。

**行业覆盖范围**：

| 细分领域 | 核心业务 | 本体设计考量 |
|---------|---------|-------------|
| 房屋建筑 | 土建工程、安装工程、装饰装修 | 基础核心，业务最复杂 |
| 市政工程 | 道路、桥梁、管网、绿化 | 需考虑政府业主特点 |
| 水利水电 | 水利设施、电力设施 | 需考虑长周期和特殊成本结构 |
| 公路工程 | 道路、隧道、桥梁 | 与房建有相似性 |
| 铁路工程 | 线路、隧道、桥梁、站场 | 需考虑线性工程特点和多专业协同 |

**业务通用性原则**：
- 核心业务概念（如项目、产值、成本、合同）应适用于所有细分领域
- 特有概念（如水利度汛、公路养护）作为扩展属性处理
- 本体设计优先考虑通用性，扩展性通过继承机制实现

## 二、平台能力与知识映射

> **核心原则**：深度蒸馏的产出必须能够映射到平台能力，确保设计可落地。

### 2.1 平台能力概览

| 平台能力 | 能力说明 | 知识蒸馏的对应产出 |
|---------|---------|------------------|
| **Object Type** | 业务实体类型 | L1基础业务实体 |
| **Link Type** | 实体间关系 | L2显性关系、L3隐性关系 |
| **Action Type** | 业务动作 | L5流程规则中的业务动作 |
| **Cube** | 事实表+维度表 | 业务指标和聚合需求 |
| **派生属性** | 派生计算属性 | L3隐性关系中的量化规则 |

### 2.2 六层次到平台能力的映射

#### 2.2.1 L6 维度识别的核心问题

L6 维度识别是连接"业务理解"和"本体设计"的关键桥梁，需要系统回答以下问题：

| 问题类型 | 核心问题 | 判断标准 |
|---------|---------|---------|
| **字段识别** | 哪些字段是"分类/编码+名称"的组合？ | 字段包含 `*_code` + `*_name` 或类似模式 |
| **层级判断** | 这个分类体系是否需要层级结构？ | 存在明确的父子关系（如 L1/L2/L3） |
| **复用性** | 这个分类是否会重复出现在多个事实表中？ | 同一编码在多个表中出现 |
| **维护性** | 这个分类是否需要独立维护？ | 新增分类项是否需要更新多个事实表 |
| **编码标准** | 编码是否需要符合行业标准？ | 是否存在行业通用的编码规范 |

#### 2.2.2 L6 维度识别的产出

| 产出类型 | 说明 | 示例 |
|---------|------|------|
| **维度表清单** | 需要独立维护的维度表列表 | `dim_subject`、`dim_cost_subject`、`dim_indicator` |
| **维度编码体系** | 维度的编码规则和层级结构 | 成本科目：`01-人工成本`、`0101-基本工资` |
| **维度属性定义** | 维度表的字段定义 | `subject_id`、`subject_code`、`subject_name`、`parent_id` |
| **维度映射关系** | 维度表与事实表的映射关系 | `fact_project_balance.subject_key → dim_subject.subject_key` |

### 2.3 蒸馏产出的平台适配检查

> **检查要求**：每个蒸馏产出必须通过以下检查，确保可映射到平台能力。

| 检查项 | 检查内容 | 不通过时的处理 |
|--------|---------|--------------|
| Object Type 可定义 | 实体是否有明确的属性集合 | 拆分为更小的实体或合并到已有实体 |
| Link Type 可定义 | 关系是否有明确的语义 | 定义为关联属性而非独立关系 |
| Action Type 可定义 | 动作是否有明确的触发条件和执行主体 | 降级为派生属性或查询函数 |
| Cube 可生成 | 是否有事实表+维度表结构 | 调整表结构设计 |
| 派生属性可计算 | 是否有明确的计算规则和数据来源 | 标记为待实现，后续迭代 |

### 2.4 架构师规范强制检查

> **强制要求**：所有蒸馏产出必须符合架构师规范，以下为必检项：

| 规范项 | 检查内容 | 依据文档 |
|--------|---------|---------|
| **物理表命名规范** | 所有实体必须映射到 `fact_*`（事实表）或 `dim_*`（维度表） | 本体命名规范 §2 |
| **dim_date 强制关联** | 所有事实表必须包含 `date_key` 字段并关联 `dim_date` | 本体命名规范 §2.3 |
| **Cube 层独立设计** | 指标体系必须明确映射到独立的 Cube，包含维度、度量、派生度量 | 本体规划指南 |
| **Cube 命名规范** | Cube 名必须为 PascalCase + `Cube` 后缀（如 `ProjectCostCube`） | 本体命名规范 §4 |
| **Cube 类别标注** | 每个 Cube 必须标注 Process / Subject / Comparison / Snapshot 类别 | 本体命名规范 §4 |
| **bind_source 绑定** | 每个 Cube 必须明确绑定数据来源（物理表） | 本体规划指南 |
| **Object Type 命名** | Object Type 的 `code` 必须为无前缀 PascalCase（如 `Project`） | 本体命名规范 §5 |
| **分类标注** | 所有资源必须标注平台分类（Master / Transaction / Analytical 等） | 本体分类规划与SDK扩展方案 |
| **链接语义** | Link Type 的两端必须是 Object Type `code`，而非表名或 Cube 名 | 本体规划指南 |

### 2.5 平台能力深度绑定规则

#### 2.5.1 Object Type → 物理表映射规则

```
Object Type → 物理表映射逻辑：
┌─────────────────────────────────────────────────────────────────┐
│  Object Type 分类       │ 映射表类型        │ 命名示例          │
├─────────────────────────────────────────────────────────────────┤
│  Master（主数据）       │ dim_*（维度表）   │ dim_project      │
│  Transaction（交易数据）│ fact_*（事实表）  │ fact_project_cost│
│  Analytical（分析实体） │ dim_* / fact_*   │ 视情况而定       │
└─────────────────────────────────────────────────────────────────┘
```

#### 2.5.2 Cube 设计规则

| 规则 | 说明 |
|------|------|
| **一个主题一个 Cube** | 每个业务分析主题对应一个 Cube |
| **必须包含时间维度** | 每个 Cube 必须包含 `dim_date` 作为时间维度 |
| **度量必须可聚合** | 所有度量字段必须支持 SUM/AVG/COUNT 等聚合操作 |
| **派生度量必须有计算规则** | 派生度量必须明确计算逻辑和数据来源 |

#### 2.5.3 实施顺序约束

> **强制顺序**：蒸馏产出的实现必须遵循以下顺序：
> 1. 物理表设计 → 2. Cube 注册 → 3. Object Type 定义 → 4. Link Type 定义 → 5. Function/Action 开发

```
## 四、实际执行经验与文件结构优化

> **核心发现**：基于阶段一的实际执行，发现文件结构优化是提升知识体系可用性的关键。

### 4.1 文件结构优化原则

#### 4.1.1 成果性质优先于工作流程阶段

**问题发现**：
- 初期按五轮迭代蒸馏法的工作流程阶段划分文件（广度扫描、深度挖掘、精细打磨、维度识别、交叉验证）
- 实际使用中发现：后续阶段引用知识时需要跨多个文件夹查找，效率低下

**优化方案**：
- 改为按**成果性质**划分：核心知识体系、量化规则库
- 核心知识体系包含L1-L6的所有产出，便于后续本体设计时快速定位

**对比示例**：

| 划分方式 | 文件结构 | 查找效率 |
|---------|---------|---------|
| **工作流程阶段** | 广度扫描/深度挖掘/精细打磨/维度识别/交叉验证 | 需跨5个文件夹查找 |
| **成果性质** | 核心知识体系/量化规则库 | 在1-2个文件夹内完成查找 |

#### 4.1.2 企业专识融合而非独立管理

**问题发现**：
- 初期将企业专识萃取作为独立步骤，产出独立的企业业务知识萃取报告
- 实际使用中发现：企业特有规则（如资金月度批复、产值双轨制）与行业规则存在关联，独立管理导致关联断裂

**优化方案**：
- 企业专识内容融合到核心知识体系和量化规则库中
- 企业特有规则（ER001-ER004）融合到量化规则库的"企业特有规则"章节
- 企业特有动作（A020-A024）融合到流程与动作清单

**融合策略**：

| 融合场景 | 融合方式 | 示例 |
|---------|---------|------|
| **企业特有动作** | 融合到流程与动作清单 | A020-A024（ConfirmOutput、ApplyChange、FileClaim等） |
| **企业特有规则** | 融合到量化规则库 | ER001-ER004（资金月度批复、产值双轨制、成本三级分类、风险三色预警） |
| **企业特有指标** | 融合到业务指标清单 | MI026（收款率）、MI027（资金计划准确率） |

**融合原则**：
- 企业特有内容必须标注来源类型为"企业业务素材"
- 企业特有规则必须与行业规则建立关联（如ER001关联BR-E004）
- 企业特有内容必须说明适用范围（如"仅适用于潘达建工"）

#### 4.1.3 建立双向关联机制

**问题发现**：
- 初期流程清单、诀窍清单、规则库之间缺乏关联
- 后续本体设计时需要手动查找相关规则，效率低下

**优化方案**：
- 在流程清单中新增"关联规则"字段，建立流程与规则的关联
- 在诀窍清单中新增"关联流程"和"关联规则"字段，建立诀窍与流程、规则的关联
- 在规则库中新增"关联流程"字段，建立规则与流程的关联

**关联示例**：

| 文件 | 关联字段 | 关联对象 |
|------|---------|---------|
| **流程与动作清单** | 关联规则 | P001关联BR-E004（资金月度批复） |
| **行业诀窍清单** | 关联流程、关联规则 | CV001关联P009（变更签证流程）、BR-C015（变更签证时效规则） |
| **量化规则库** | 关联流程 | BR-E004关联P001（项目立项流程） |
```

## 六、业务人员输入采集机制

> **核心原则**：业务人员的零散知识和经验是深度蒸馏的重要输入，应在蒸馏过程中系统性地收集和整合。

### 6.1 采集机制设计

| 环节 | 目的 | 方式 | 产出 |
|------|------|------|------|
| **业务人员访谈** | 收集业务人员的"零散"知识和经验 | 开放式访谈、问卷调查 | 业务人员知识补充清单 |
| **术语征集** | 收集行业特有术语和企业专用术语 | 术语征集表、即时通讯收集 | 术语对照表 |
| **规则验证** | 验证蒸馏结果的正确性 | 规则确认表、业务评审会 | 规则验证报告 |
| **问题收集** | 收集业务人员的疑问和建议 | 问题收集表、匿名反馈 | 待调研问题清单 |

### 6.2 业务人员访谈指南

**访谈对象**：
- 商务成本经理（核心）
- 项目商务人员（一线经验）
- 财务人员（成本核算）
- 合同管理员（合同管理）
- 现场工程师（施工经验）

**访谈提纲示例**：

```
## 业务人员访谈提纲

### 一、业务场景理解
1. 请描述您日常工作中最核心的业务场景是什么？
2. 在这些场景中，您最关注哪些业务指标？
3. 您认为哪些业务环节最容易出现问题？

### 二、术语与概念
1. 您在工作中经常使用哪些行业术语？
2. 这些术语的具体含义是什么？
3. 有没有企业内部特有的术语或简称？

### 三、业务规则
1. 在成本管理中，哪些规则是必须遵守的？
2. 有没有一些"不成文"的规则或经验？
3. 哪些业务规则对项目成败至关重要？

### 四、痛点与建议
1. 当前业务流程中存在哪些痛点？
2. 您认为哪些环节可以优化？
3. 对智能决策系统有哪些期望？
```

### 6.3 术语征集机制

**术语征集表结构**：

| 术语 | 业务定义 | 行业通用定义 | 企业特有含义 | 适用场景 |
|------|---------|-------------|-------------|---------|
| 术语1 | 企业内部定义 | 行业标准定义 | 企业特有含义（如有） | 何时使用 |
| 术语2 | 企业内部定义 | 行业标准定义 | 企业特有含义（如有） | 何时使用 |

**采集方式**：
1. 通过即时通讯工具（如企业微信）收集
2. 在业务评审会上现场征集
3. 定期发放术语征集表

### 6.4 规则验证机制

**规则验证流程**：

```
蒸馏产出规则 → 业务人员评审 → 验证通过/修改/补充 → 更新规则库
```

**验证标准**：
| 验证项 | 标准 |
|--------|------|
| 正确性 | 规则是否符合实际业务 |
| 完整性 | 是否遗漏重要规则 |
| 可操作性 | 规则是否可在系统中实现 |
| 优先级 | 规则的重要程度 |

### 6.5 与四轮蒸馏法的集成

业务人员输入应在蒸馏过程中持续收集，并在各轮次中进行验证：

| 蒸馏轮次 | 业务人员输入方式 | 输入用途 |
|---------|----------------|---------|

| 第三轮 | 流程评审会 | 验证流程定义 |
| 第四轮 | 冲突解决会 | 确认企业特有规则 |

## 八、多企业素材冲突处理方法论

> **适用场景**：当存在多个企业的业务素材（如本企业真实数据、其他企业参考模板、行业通用知识）时，如何处理潜在冲突并融合各方价值。

### 8.1 素材类型分析

| 素材类型 | 特点 | 价值 | 风险 | 优先级 |
|---------|------|------|------|--------|
| **本企业真实素材** | 最符合实际业务，有真实数据支撑 | 核心依据，确保落地性 | 可能不够全面，存在企业局限性 | **P1** |
| **行业通识蒸馏** | 通用知识框架，覆盖行业共性 | 扩展视野，补充知识空白 | 可能与企业实际有差异 | **P2** |
| **其他企业参考素材** | 提供不同管理视角和实践经验 | 启发思路，引入最佳实践 | 存在术语和流程冲突 | **P3** |

### 8.2 三层过滤处理法

```
原始素材 → 第一层过滤 → 第二层过滤 → 第三层过滤 → 最终采纳
              ↓              ↓              ↓
           术语对齐      规则验证      价值评估
```

#### 第一层：术语对齐

**目标**：消除术语差异，建立统一的业务语言

**操作步骤**：

| 步骤 | 操作 | 产出 |
|------|------|------|
| 1 | 收集所有素材中的术语 | 术语收集表 |
| 2 | 识别同义术语和多义术语 | 术语映射表 |
| 3 | 标记企业特有术语 | 特有术语清单 |
| 4 | 建立统一术语命名规范 | 术语对照矩阵 |

**术语映射表示例**：

| 术语来源 | 术语名称 | 统一术语 | 说明 |
|---------|---------|---------|------|
| 本企业 | 合同结算 | 结算 | 合同最终价款结算 |
| 其他企业A | 工程决算 | 结算 | 同义术语 |
| 行业通识 | 竣工结算 | 结算 | 同义术语 |
| 本企业 | 分包班组 | 劳务分包商 | 特有称谓 |

#### 第二层：规则验证

**目标**：验证业务规则的合理性和适用性

**验证维度**：

| 维度 | 验证标准 | 通过条件 | 处理方式 |
|------|---------|---------|---------|
| **业务合理性** | 是否符合行业逻辑 | 符合行业通用规则 | 保留 |
| **数据可获取性** | 是否有数据源支持 | 能从现有系统获取 | 标记为可落地 |
| **企业适用性** | 是否符合企业实际 | 与企业流程一致 | 采纳 |
| **合规性** | 是否符合法规要求 | 符合相关法规 | 通过 |

**验证决策矩阵**：

| 业务合理性 | 数据可获取性 | 企业适用性 | 决策 |
|-----------|-------------|-----------|------|
| ✅ | ✅ | ✅ | 直接采纳 |
| ✅ | ✅ | ❌ | 修改后采纳 |
| ✅ | ❌ | ✅ | 标记为待实现 |
| ❌ | - | - | 舍弃 |

#### 第三层：价值评估

**目标**：评估规则的业务价值，确定优先级

**评估指标**：

| 指标 | 评分标准（1-5分） | 权重 |
|------|------------------|------|
| **覆盖度** | 是否填补现有知识空白 | 30% |
| **独特性** | 是否提供独特视角或创新 | 25% |
| **可落地性** | 是否可在企业中实施 | 25% |
| **风险度** | 是否引入重大冲突 | 20% |

**价值分类**：

| 分类 | 总分范围 | 处理方式 |
|------|---------|---------|
| **高价值** | 16-20分 | 优先采纳 |
| **中价值** | 11-15分 | 作为备选 |
| **低价值** | 6-10分 | 仅作参考 |
| **无价值** | ≤5分 | 舍弃 |

### 8.3 冲突处理原则

> **核心原则**：以本企业真实素材为基础，行业通识为补充，其他企业素材为启发

**冲突处理优先级**：

| 优先级 | 来源类型 | 处理方式 | 示例 |
|--------|---------|---------|------|
| 1 | 本企业真实数据 | 优先采用，作为基准 | 企业特有的成本科目结构 |
| 2 | 本企业业务规则 | 严格遵守 | 企业内部审批流程 |
| 3 | 行业通用知识 | 参考补充，填补空白 | 通用的指标定义和计算规则 |
| 4 | 其他企业素材 | 启发参考，选择性借鉴 | 其他企业的流程优化思路 |

**冲突类型及处理策略**：

| 冲突类型 | 表现 | 处理策略 |
|---------|------|---------|
| **术语冲突** | 同一概念不同名称 | 建立术语映射表 |
| **规则冲突** | 业务规则不一致 | 以本企业规则为准，标记差异 |
| **流程冲突** | 流程步骤不同 | 分析优劣，选择性融合 |
| **指标冲突** | 指标定义不同 | 统一计算口径 |

### 8.4 实施步骤

