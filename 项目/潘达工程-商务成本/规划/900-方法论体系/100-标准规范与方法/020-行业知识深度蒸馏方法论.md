### 1.3 行业领域定义

> **建筑工程行业定位**：本项目以**建筑工程行业**（综合型建工企业）为核心定位，业务模型应具备足够的**行业代表性**和**通用性**。

**行业覆盖范围**：

| 细分领域 | 核心业务 | 本体设计考量 |
|---------|---------|-------------|
| 房屋建筑 | 土建工程、安装工程、装饰装修 | 基础核心，业务最复杂 |
| 市政工程 | 道路、桥梁、管网、绿化 | 需考虑政府业主特点 |
| 水利水电 | 水利设施、电力设施 | 需考虑长周期和特殊成本结构 |
| 公路工程 | 道路、隧道、桥梁 | 与房建有相似性 |
| 园林绿化 | 景观、养护 | 需考虑养护期成本 |
| 装饰装修 | 室内外装饰、改造 | 需考虑短周期和多变更 |

**业务通用性原则**：
- 核心业务概念（如项目、产值、成本、合同）应适用于所有细分领域
- 特有概念（如水利度汛、公路养护）作为扩展属性处理
- 本体设计优先考虑通用性，扩展性通过继承机制实现

## 二、平台能力与知识映射

> **核心原则**：深度蒸馏的产出必须能够映射到平台能力，确保设计可落地。

### 2.1 平台能力概览

| 平台能力 | 能力说明 | 知识蒸馏的对应产出 |
|---------|---------|------------------|
| **Object Type** | 业务实体类型 | L1基础业务实体 |
| **Link Type** | 实体间关系 | L2显性关系、L3隐性关系 |
| **Action Type** | 业务动作 | L5流程规则中的业务动作 |
| **Cube** | 事实表+维度表 | 业务指标和聚合需求 |
| **派生属性** | 派生计算属性 | L3隐性关系中的量化规则 |

### 2.2 五层次到平台能力的映射

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         知识蒸馏 → 平台能力映射                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  L1 基础业务知识                                                         │
│  ├── 核心概念    → Object Type（业务实体）                              │
│  ├── 术语表      → Object Type 属性定义                                 │
│  └── 业务流程    → Action Type（业务动作）                               │
│                                                                         │
│  L2 显性业务关系                                                         │
│  ├── 实体关联    → Link Type（包含、归属、关联）                         │
│  └── 数据流向    → Flow/ETL 映射                                        │
│                                                                         │
│  L3 隐性业务关系                                                         │
│  ├── 经验规则    → 派生属性/计算规则                                     │
│  ├── 量化指标    → Cube 度量定义                                        │
│  └── 预警规则    → 派生属性 + Action Type（预警触发）                    │
│                                                                         │
│  L4 行业独特诀窍                                                         │
│  ├── 特有规则    → 企业特有规则（ER）                                    │
│  └── 操作模式    → Action Type（业务动作）                               │
│                                                                         │
│  L5 行业通用流程规则                                                     │
│  ├── 流程定义    → Action Type（业务动作链）                             │
│  ├── 审批规则    → Action Type（审批节点）                               │
│  └── 角色定义    → Object Type（参与者）                                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.3 蒸馏产出的平台适配检查

> **检查要求**：每个蒸馏产出必须通过以下检查，确保可映射到平台能力。

| 检查项 | 检查内容 | 不通过时的处理 |
|--------|---------|--------------|
| Object Type 可定义 | 实体是否有明确的属性集合 | 拆分为更小的实体或合并到已有实体 |
| Link Type 可定义 | 关系是否有明确的语义 | 定义为关联属性而非独立关系 |
| Action Type 可定义 | 动作是否有明确的触发条件和执行主体 | 降级为派生属性或查询函数 |
| Cube 可生成 | 是否有事实表+维度表结构 | 调整表结构设计 |
| 派生属性可计算 | 是否有明确的计算规则和数据来源 | 标记为待实现，后续迭代 |

## 三、深度蒸馏流程设计

### 3.1 流程概述：四轮迭代蒸馏法

> **方法论依据**：《000-顶层设计.md》中"相对理想型策略"的具体执行方式。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         四轮迭代蒸馏流程                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  第一轮：广度扫描（覆盖 L1+L2）                                         │
│  ├── 目标：快速建立完整的业务世界骨架                                   │
│  ├── 时长：1-2小时                                                      │
│  └── 产出：业务实体清单、显性关系图                                     │
│                          ↓                                              │
│  第二轮：深度挖掘（覆盖 L3+L4）                                         │
│  ├── 目标：深入挖掘隐性知识和行业诀窍                                    │
│  ├── 时长：2-3小时                                                      │
│  └── 产出：隐性关系清单、行业诀窍清单、量化规则                          │
│                          ↓                                              │
│  第三轮：精细打磨（覆盖 L5 流程规则）                                    │
│  ├── 目标：完善流程规则和业务动作库                                     │
│  ├── 时长：1-2小时                                                      │
│  └── 产出：流程规则库、业务动作库、指标体系                              │
│                          ↓                                              │
│  第四轮：交叉验证（覆盖冲突解决）                                         │
│  ├── 目标：与企业实际数据对照，解决冲突                                  │
│  ├── 时长：1-2小时                                                      │
│  └── 产出：冲突清单、验证报告、最终知识骨架                              │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.2 各轮次详细设计

#### 第一轮：广度扫描（L1 基础知识 + L2 显性关系）

**核心目标**：快速建立完整的业务世界骨架

**前置准备**：
1. 明确建筑工程行业的业务边界
2. 确定业务分析的视角（商务成本管理）
3. 准备建筑工程行业的标准术语参考

**蒸馏步骤**：

| 步骤 | 任务 | 提示词要点 | 产出 |
|------|------|-----------|------|
| 1.1 | 领域全景扫描 | "建筑工程企业商务成本管理的完整业务领域包含哪些？" | 领域全景图 |
| 1.2 | 核心实体识别 | "列举建筑工程商务成本管理中的所有核心业务实体" | 实体清单（≥20个） |
| 1.3 | 实体属性定义 | "每个核心实体包含哪些关键属性？" | 实体-属性矩阵 |
| 1.4 | 显性关系梳理 | "这些实体之间存在哪些明确的业务关系？" | 显性关系图（≥30条） |
| 1.5 | 基数关系定义 | "每个关系的基数（1:1、1:N、N:M）是什么？" | 基数关系矩阵 |

**提示词模板**：

```
## 提示词1：领域全景扫描

你是一名拥有20年经验的建筑工程行业商务成本管理专家。

请详细描述建筑工程企业（涵盖房建、市政、水利、公路、园林等细分领域）商务成本管理的完整业务领域：

1. 从项目立项到竣工决算的完整业务链条包含哪些环节？
2. 每个环节涉及哪些核心业务活动？
3. 商务成本管理的核心目标是什么？
4. 哪些业务活动直接影响项目利润和成本？

请提供全面、具体的行业知识，确保覆盖建筑工程行业的各个细分领域。
```

```
## 提示词2：核心实体识别

基于上述业务领域，请列举建筑工程商务成本管理中的所有核心业务实体。

要求：
- 每个实体必须包含：实体名称、业务定义、核心属性
- 实体类型应涵盖：管理对象（项目、合同）、业务活动（产值、成本）、参与方（业主、供应商）、管理要素（指标、风险）
- 数量要求：不少于20个核心实体
- 行业覆盖：确保覆盖房建、市政、水利、公路等不同细分领域的实体

请按以下格式输出：
【实体编号】实体名称
- 业务定义：（一句话定义）
- 核心属性：（3-5个关键属性）
- 适用领域：（房建/市政/通用）
```

```
## 提示词3：显性关系梳理

请描述上述核心实体之间的显性业务关系。

关系类型包括：
- 包含关系：整体与部分（如项目包含合同）
- 归属关系：从属关系（如合同归属项目）
- 关联关系：业务关联（如供应商提供资源）
- 计算关系：派生计算（如利润=产值-成本）
- 影响关系：因果关系（如变更影响成本）

要求：
- 每个关系必须说明具体的业务含义
- 识别建筑工程行业的特有关系（如变更签证对成本的影响）
- 数量要求：不少于30条关系
- 确保与平台能力可映射（关系必须有明确的语义）

请按以下格式输出：
【关系编号】关系名称
- 关系类型：（包含/归属/关联/计算/影响）
- 源实体 → 目标实体
- 基数：（1:1 / 1:N / N:M）
- 业务含义：（具体说明）
```

**平台能力映射检查点**：

| 产出 | 平台能力映射 | 检查结果 |
|------|-------------|---------|
| 实体清单 | Object Type 定义 | 每个实体是否可定义为 Object Type |
| 实体-属性矩阵 | Object Type 属性 | 属性是否有明确的数据类型 |
| 显性关系图 | Link Type 定义 | 关系是否有明确语义 |
| 基数关系矩阵 | 数据模型基数 | 基数是否影响 Cube 设计 |

#### 第二轮：深度挖掘（L3 隐性关系 + L4 行业诀窍）

**核心目标**：深入挖掘隐性知识和行业诀窍

**前置条件**：第一轮产出已通过平台能力映射检查

**蒸馏步骤**：

| 步骤 | 任务 | 提示词要点 | 产出 |
|------|------|-----------|------|
| 2.1 | 隐性关系挖掘 | "哪些业务关系是数据中不直接体现但决策重要的？" | 隐性关系清单（≥20条） |
| 2.2 | 量化规则提取 | "这些隐性关系能否量化？阈值是多少？" | 量化规则库 |
| 2.3 | 行业诀窍识别 | "建筑工程行业有哪些特有的规则和门道？" | 行业诀窍清单（≥15条） |
| 2.4 | 异常场景分析 | "哪些业务场景看起来正常但可能隐藏风险？" | 异常场景库 |
| 2.5 | 决策模式提炼 | "经验丰富的成本经理如何做决策？" | 决策模式库 |

**提示词模板**：

```
## 提示词4：隐性关系挖掘

你是一名资深的建筑工程行业商务成本管理专家。请深入挖掘行业中不直接体现在数据中，但对业务决策有重要影响的隐性关系。

重点关注以下方面：

1. **指标间的隐性关联**
   - 哪些指标之间存在关联？一个指标变化如何影响其他指标？
   - 例如：成本偏差率与进度偏差率是否存在关联？

2. **业务规则的前提假设**
   - 哪些业务规则的成立有隐含前提？
   - 例如：产值确认规则的前提是什么？何时可能不成立？

3. **异常征兆识别**
   - 哪些早期征兆预示着后续问题？
   - 例如：哪些情况出现后，成本超支风险会显著增加？

4. **时间维度的延迟效应**
   - 哪些决策的影响有延迟？
   - 例如：收款周期延长如何影响利润率？

要求：
- 每条隐性关系必须说明具体的业务含义
- 必须说明该关系在决策中的价值
- 数量要求：不少于20条隐性关系
- 优先挖掘建筑工程行业的深层逻辑

请按以下格式输出：
【关系编号】关系名称
- 关系描述：（具体说明）
- 关联对象：（涉及的实体）
- 业务含义：（为什么重要）
- 决策价值：（如何利用该关系做决策）
```

```
## 提示词5：量化规则提取

基于上述隐性关系，请尝试量化这些关系。

对于每条隐性关系，请尝试回答：
1. 是否可以量化？
2. 量化公式是什么？
3. 阈值是多少？（临界点）
4. 如何用于预警？

示例格式：
【关系编号】关系名称
- 量化可能性：（可量化/部分可量化/难以量化）
- 量化公式：（如果可量化，给出公式）
- 阈值定义：（临界点或预警值）
- 预警逻辑：（如何触发预警）
```

```
## 提示词6：行业诀窍识别

请列举建筑工程行业商务成本管理中的特有规则、操作技巧和行业门道。

这些规则必须满足以下条件之一：
1. 只在建筑工程行业存在，其他行业不适用
2. 建筑工程行业有独特的表现形式
3. 经验丰富的从业者才知道的"门道"

重点关注：
1. **合同管理诀窍**
   - 合同条款中的哪些细节影响成本？
   - 如何通过合同条款控制风险？

2. **变更签证诀窍**
   - 变更签证的哪些处理方式影响项目利润？
   - 如何最大化变更签证的收益？

3. **成本控制诀窍**
   - 哪些成本控制技巧是建筑工程行业特有的？
   - 如何平衡质量和成本？

4. **资金管理诀窍**
   - 如何优化资金流，减少资金占用成本？
   - 质保金、履约保函等如何处理？

5. **分包管理诀窍**
   - 如何选择和管理分包商？
   - 分包合同的哪些条款需要特别关注？

要求：
- 每条诀窍必须说明具体的应用场景
- 必须说明该诀窍的业务影响
- 数量要求：不少于15条行业诀窍
- 确保与平台能力可映射

请按以下格式输出：
【诀窍编号】诀窍名称
- 适用场景：（何时使用）
- 规则描述：（具体做法）
- 业务影响：（对成本/利润/风险的影响）
- 实现建议：（如何在本体中体现）
```

**平台能力映射检查点**：

| 产出 | 平台能力映射 | 检查结果 |
|------|-------------|---------|
| 隐性关系清单 | 派生属性/Link Type | 关系是否有计算规则 |
| 量化规则库 | Cube 度量/派生属性 | 公式是否可实现 |
| 行业诀窍清单 | 企业特有规则 | 是否需要 Action Type |
| 异常场景库 | 预警规则 | 是否可触发 Action |
| 决策模式库 | Action Type | 是否有明确的触发条件 |

#### 第三轮：精细打磨（L5 流程规则）

**核心目标**：完善流程规则和业务动作库

**前置条件**：第二轮产出已通过平台能力映射检查

**蒸馏步骤**：

| 步骤 | 任务 | 提示词要点 | 产出 |
|------|------|-----------|------|
| 3.1 | 核心流程识别 | "建筑工程商务成本的核心业务流程有哪些？" | 流程清单（≥10个） |
| 3.2 | 流程节点细化 | "每个流程包含哪些步骤？涉及哪些角色？" | 详细流程定义 |
| 3.3 | 业务动作提取 | "每个流程中的关键业务动作是什么？" | 业务动作库（≥20个） |
| 3.4 | 指标体系构建 | "商务成本管理的完整指标体系是什么？" | 指标体系文档 |
| 3.5 | 规则优先级排序 | "哪些规则对项目成功最关键？" | 规则优先级矩阵 |

**提示词模板**：

```
## 提示词7：核心流程识别与细化

请详细描述建筑工程商务成本管理的核心业务流程。

对于每个流程，请说明：
1. 流程名称和目标
```

### 5.2 平台能力映射检查

| 检查项 | 检查内容 | 不通过时的处理 |
|--------|---------|--------------|

| Cube 可生成性 | 是否有事实表+维度表结构 | 调整表结构 |
| 派生属性可计算性 | 是否有计算规则和数据来源 | 标记为后续迭代 |
