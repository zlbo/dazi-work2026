## 一、融合原则

以项目方法论为主线，在具体技术落地环节调用平台标准能力：

| 领域 | 项目方法论 | 平台标准能力 |
|------|-----------|-------------|
| **流程开发** | 阶段四数据对接 | Flow流程开发规范、节点代码规范 |
| **应用开发** | 阶段五/六能力开发 | DRAP应用框架、组件库 |

## 二、流程开发融合规范

### 2.1 目录结构规范

```
流程/
└── flows/
    └── flow-<流程名>/
        ├── flow.json              # 流程配置（必填）
        ├── flow.info.json         # 流程元数据（必填）
        ├── 快速启动_<流程名>.md    # 启动说明（必填）
        └── 节点/
            └── <节点名>/
                ├── node.info.json  # 节点元数据（必填）
                └── code.<ext>      # 节点代码（sql/python/js，必填）
```

### 2.2 流程配置规范

| 配置项 | 要求 | 示例 |
|--------|------|------|
| flowId | 整数，唯一标识 | `130` |
| name | 中文名称 | `产值数据同步` |
| description | 业务描述 | `同步项目产值数据至本体` |
| nodes | 节点数组 | 包含至少1个节点 |
| edges | 连线数组 | 定义节点间数据流向 |

### 2.3 节点类型与规范

| 节点类型 | 用途 | 代码文件 | 配置要求 |
|----------|------|---------|---------|
| `sql-query` | 内存SQL查询 | `code.sql` | 无额外配置 |
| `database-source` | 数据库查询 | `code.sql` | 需配置`dataSourceId` |
| `python-script` | Python脚本 | `code.py` | 需配置`output_variable_name` |
| `cube-sink` | Cube写入 | 无代码文件 | 需配置`cubeName` |

### 2.4 变量处理规范

```powershell
# 获取上游变量
upstream_data = get_variable("output_data")

# 设置输出变量
set_variable("result", processed_data)
```

### 2.5 发布与测试命令

```powershell
# 1. 提交节点代码
dazi flow node push --node <node_uuid> --dir "<流程目录>"

# 2. 单节点测试
dazi flow run node-exec --node <node_uuid> --dir "<流程目录>"

# 3. 核对输出变量
dazi flow variable pull --name <var_name> --dir "<流程目录>"

# 4. 流程测试
dazi flow run --flow <flow_id> --dir "<流程目录>"
```

## 三、应用开发融合规范

### 3.1 目录结构规范

```
应用/
└── apps/
    └── <应用名>/
        ├── package.json           # 依赖配置（必填）
        ├── tsconfig.json          # TypeScript配置（必填）
        ├── vite.config.ts         # Vite配置（必填）
        ├── src/
        │   ├── main.ts            # 入口文件（必填）
        │   ├── App.vue            # 根组件（必填）
        │   ├── components/        # 业务组件
        │   ├── views/             # 页面视图
        │   ├── api/               # API接口
        │   ├── stores/            # 状态管理
        │   └── utils/             # 工具函数
        └── public/
            └── index.html         # HTML模板
```

### 3.2 DRAP框架规范

| 组件类型 | 命名规范 | 说明 |
|----------|---------|------|
| 页面组件 | `Page<业务名>.vue` | 如`PageProjectList.vue` |
| 业务组件 | `<业务名>Component.vue` | 如`ProjectCardComponent.vue` |
| 通用组件 | `Base<组件名>.vue` | 如`BaseTable.vue` |

### 3.3 状态管理规范

```typescript
// stores/project.ts
import { defineStore } from 'pinia';

export const useProjectStore = defineStore('project', {
  state: () => ({
    projects: [],
    selectedProject: null,
  }),
  actions: {
    async fetchProjects() {
      const response = await api.get('/projects');
      this.projects = response.data;
    },
    selectProject(project) {
      this.selectedProject = project;
    },
  },
});
```

### 3.4 API调用规范

```typescript
// api/project.ts
import { request } from '@/utils/request';

export const projectApi = {
  getList: (params) => request.get('/api/v1/projects', { params }),
  getDetail: (id) => request.get(`/api/v1/projects/${id}`),
  create: (data) => request.post('/api/v1/projects', data),
  update: (id, data) => request.put(`/api/v1/projects/${id}`, data),
  delete: (id) => request.delete(`/api/v1/projects/${id}`),
};
```

### 3.5 发布命令

```powershell

npm run build

# 2. 发布到平台
dazi app publish --app <应用名> --space space__panda_cost01

# 3. 预览
dazi app preview --app <应用名> --space space__panda_cost01
```

## 四、流程与应用集成规范

### 4.1 数据流向

```
数据源 → Flow流程 → Cube → 本体层 → API → 应用层
```

### 4.2 调用方式

| 集成场景 | 调用方式 | 说明 |
|----------|---------|------|
| 应用触发流程 | `dazi flow run --flow <flow_id>` | 通过CLI命令触发 |
| 流程回调应用 | Webhook | 流程完成后回调应用接口 |
| 应用读取Cube | `dazi cube query` | 查询Cube数据 |
| 应用调用本体函数 | SDK调用 | 通过`dazi.onto.function.call` |

### 4.3 变量传递规范

```typescript
// 应用中触发流程并传递变量
const result = await dazi.flow.run({
  flowId: 130,
  variables: {
    projectId: 'PROJ_0001',
    reportPeriod: '202401',
  },
});

// 流程中获取变量
const projectId = get_variable("projectId");
```

## 五、代码质量规范

### 5.1 代码审查清单

| 检查项 | 要求 |
|--------|------|
| 命名规范 | 组件名使用PascalCase，函数名使用camelCase |
| 类型定义 | TypeScript必须定义完整类型 |
| 错误处理 | 所有异步操作必须有try-catch |
| 日志记录 | 关键节点必须有日志输出 |
| 性能优化 | 避免不必要的重渲染和重复请求 |

### 5.2 测试规范

| 测试类型 | 覆盖率要求 |
|----------|-----------|
| 单元测试 | ≥80% |
| 集成测试 | 覆盖核心业务流程 |
| E2E测试 | 覆盖关键用户路径 |

## 六、版本控制规范

| 文件类型 | 版本策略 |
|----------|---------|
| 流程配置 | 随项目版本一起管理 |
| 节点代码 | 独立版本，支持回滚 |
| 应用代码 | 遵循Semantic Versioning |
| SDK脚本 | 与本体版本同步 |
