## 二、模拟数据的核心原则

### 2.1 模拟数据的两个核心维度

> **重要概念**：模拟数据需要同时满足两个核心维度，这是本方法论的基础认知框架。

| 维度 | 定义 | 衡量标准 | 示例 |
|------|------|---------|------|
| **数据质量** | 数据本身的技术正确性 | 业务规则、勾稽关系、取值范围、数据标准 | `total_amount = confirmed + unconfirmed`、`amount >= 0` |
| **经营品质** | 数据所反映的业务运营状态 | 经营健康度、风险水平、问题严重程度 | 应收账款逾期率、项目延期率、风险暴露度 |

**核心区分**：

```
数据质量高 ≠ 经营品质好

示例：
- 一份数据可能完全满足所有业务规则（数据质量高）
- 但反映的是大量应收账款逾期、工期严重延误（经营品质差）
- 这恰恰是我们需要模拟的"真实业务场景"
```

### 2.2 模拟数据的时机

**原则**：模拟数据在本体设计完成后即可开始设计，本体建模完成后进行验证。

**原因**：
1. 本体设计完成后，业务规则、勾稽关系、约束条件已经明确
2. 基于明确的业务规则生成模拟数据，可以确保数据质量
3. 避免模拟数据与本体规则不一致

**流程**：
```
业务理解 → 本体设计 → 表结构设计 → 业务规则定义 
                                              ↓
                                    模拟数据设计（现在开始）
                                              ↓
本体建模 → 脚本发布 → 模拟数据生成与验证 → ETL与数据融合
```

### 2.3 模拟数据的来源

**来源优先级**：

| 来源 | 优先级 | 说明 |
|------|-------|------|
| **本体规则** | 最高 | 基于本体定义的业务规则生成模拟数据 |
| **目标企业素材** | 高 | 选取真实企业业务数据作为参考 |
| **现有数据参考** | 中 | 参考现有数据库表的量级和分布 |
| **企业专识** | 低 | 企业专识中的具体值作为参考 |

## 三、模拟数据的目标

### 3.1 核心目标

**核心目标**：生成具有**高仿真度**、**可信度**的模拟数据，且模拟动作可行，**经营品质差异化分布**支撑决策分析场景。

### 3.2 具体目标

#### 3.2.1 数据质量目标

| 目标 | 说明 | 验证标准 |
|------|------|---------|
| 业务规则满足 | 满足本体定义的212条业务规则 | 规则验证100%通过 |
| 勾稽关系满足 | 满足指标间、层级汇总、时间序列勾稽关系 | 勾稽关系验证100%通过 |
| 约束条件满足 | 满足状态流转、业务约束规则 | 约束验证100%通过 |
| 数据标准满足 | 满足取值范围、格式、完整性规则 | 数据标准验证100%通过 |

#### 3.2.2 经营品质目标

| 目标 | 说明 | 分布要求 |
|------|------|---------|
| 差异化分布 | 经营品质呈正态分布，从卓越到危机 | 覆盖所有经营品质等级 |
| 问题常态化 | 各种业务问题（逾期、延期、风险等）合理分布 | 问题覆盖率≥80% |
| 极端案例 | 包含少量极端案例（卓越、危机） | 极端案例占比10-15% |
| 场景丰富度 | 支撑多种决策分析场景 | 场景覆盖率≥90% |

#### 3.2.3 数据体量目标

| 类型 | 最低数量 | 推荐数量 | 说明 |
|------|---------|---------|------|
| 项目数 | 50个 | 100个 | 确保统计分析有代表性 |
| 合同数 | 150个 | 300个 | 平均每个项目3个合同 |
| 成本记录 | 5000条 | 10000条 | 确保时间序列分析有深度 |
| 产值记录 | 3000条 | 6000条 | 确保月度/季度分析有支撑 |
| 付款记录 | 3000条 | 6000条 | 确保付款流程分析有依据 |

## 四、目标企业选择策略

### 4.1 选择原则

> **核心原则**：选取真实企业作为素材来源，而非凭空杜撰。工作量小、仿真度高、可追溯。

**选择标准**：

| 标准 | 要求 | 说明 |
|------|------|------|
| 业务匹配度 | ≥80% | 与本项目业务范围匹配（房建、市政、安装等） |
| 体量匹配度 | 中大型 | 年营收10亿+企业，确保数据丰富度 |
| 信息公开度 | 高 | 有年报、信息披露等公开资料 |
| 问题多样性 | 高 | 经历过行业周期，有各类业务问题 |

### 4.2 目标企业类型

**建议选取3-5家不同类型的企业**：

| 类型 | 数量 | 说明 | 示例 |
|------|------|------|------|
| **行业龙头** | 1家 | 体量大、业务全、管理规范 | 中国建筑、中国中铁等 |
| **区域龙头** | 1-2家 | 区域领先、业务聚焦 | 省属建工集团 |
| **问题企业** | 1家 | 经历过风险事件，有历史问题 | 出现过债务违约的企业 |
| **新兴企业** | 1家 | 近年成立，成长型业务 | 新晋建筑科技公司 |

### 4.3 素材采集方法

**公开资料采集**：

| 资料类型 | 来源 | 用途 |
|---------|------|------|
| 年报 | 上交所/深交所披露 | 营收规模、成本结构、项目信息 |
| 招标公告 | 全国公共资源交易平台 | 项目信息、合同金额 |
| 信用评级报告 | 评级机构 | 经营状况、风险评估 |
| 司法文书 | 裁判文书网 | 法律纠纷、违约情况 |
| 新闻报道 | 行业媒体 | 重大事件、风险事件 |
| 上市问询函 | 交易所 | 监管关注点、经营问题 |

**采集优先级**：

| 优先级 | 内容 | 价值 |
|--------|------|------|
| P0 | 项目清单、合同信息、营收数据 | 直接映射到本体 |
| P1 | 成本结构、利润分布、现金流情况 | 生成指标数据 |
| P2 | 风险事件、法律纠纷、舆情信息 | 生成问题场景 |
| P3 | 管理流程、组织架构、业务特点 | 补充业务细节 |

## 五、经营品质差异化分布策略

### 5.1 经营品质分级模型

> **核心策略**：模拟数据必须呈现差异化的经营品质，支撑从"卓越"到"危机"的完整分析场景。

**五级分类模型**：

| 等级 | 名称 | 占比 | 特征描述 | 分析场景 |
|------|------|------|---------|---------|
| **A级** | 卓越 | 10% | 各项指标优秀，无风险预警 | 标杆学习 |
| **B级** | 优良 | 25% | 指标良好，有少量小问题 | 经验借鉴 |
| **C级** | 一般 | 35% | 指标正常，存在常规问题 | 常规管理 |
| **D级** | 预警 | 20% | 多项指标异常，存在较大风险 | 风险管控 |
| **E级** | 危机 | 10% | 严重问题，面临经营困难 | 危机处理 |

### 5.2 各维度经营品质标准

#### 5.2.1 项目维度

| 等级 | 工期偏差率 | 成本偏差率 | 安全事故 | 质量评级 |
|------|-----------|-----------|---------|---------|
| A级 | ≤0% | ≤0% | 无 | 优良 |
| B级 | 0-5% | 0-3% | 0次 | 合格 |
| C级 | 5-10% | 3-8% | 0-1次 | 合格 |
| D级 | 10-20% | 8-15% | 1-2次 | 不合格 |
| E级 | >20% | >15% | ≥3次 | 不合格 |

#### 5.2.2 财务维度

| 等级 | 毛利率 | 应收账款逾期率 | 现金流状况 | 资产负债率 |
|------|-------|--------------|---------|-----------|
| A级 | ≥15% | <5% | 充足 | <70% |
| B级 | 10-15% | 5-15% | 良好 | 70-75% |
| C级 | 5-10% | 15-30% | 正常 | 75-80% |
| D级 | 0-5% | 30-50% | 紧张 | 80-85% |
| E级 | <0% | >50% | 紧张/断裂 | >85% |

#### 5.2.3 风险维度

| 等级 | 风险数量 | 高风险占比 | 风险敞口 | 预警数量 |
|------|---------|----------|---------|---------|
| A级 | 0-2 | 0% | <100万 | 0 |
| B级 | 2-5 | 0-10% | 100-500万 | 0-1 |
| C级 | 5-10 | 10-20% | 500-2000万 | 1-3 |
| D级 | 10-20 | 20-40% | 2000-5000万 | 3-5 |
| E级 | >20 | >40% | >5000万 | >5 |

### 5.3 经营品质生成策略

#### 5.3.1 项目分配策略

**按等级分配项目数量**：

| 项目类型 | 数量 | 等级分布 | 说明 |
|---------|------|---------|------|
| 大型房建 | 30个 | A:3, B:8, C:10, D:6, E:3 | 体量大、周期长 |
| 中型市政 | 30个 | A:3, B:7, C:11, D:6, E:3 | 周期中等 |
| 小型安装 | 25个 | A:2, B:6, C:9, D:5, E:3 | 周期短 |
| 基础设施 | 15个 | A:2, B:4, C:5, D:3, E:1 | 投资大 |

#### 5.3.2 问题注入策略

> **核心原则**：为D级和E级项目注入各类业务问题，模拟真实经营场景。

**问题类型与注入规则**：

| 问题类型 | 注入条件 | 注入比例 | 数据表现 |
|---------|---------|---------|---------|
| 应收账款逾期 | D/E级项目 | 80% |逾期天数>90天 |
| 工期延误 | D/E级项目 | 85% | 实际工期>计划工期15%+ |
| 成本超支 | D/E级项目 | 75% | 实际成本>预算成本10%+ |
| 变更签证频繁 | D/E级项目 | 60% | 变更金额>合同金额10% |
| 索赔纠纷 | D/E级项目 | 40% | 有未决索赔 |
| 资金紧张 | E级项目 | 90% | 现金流为负 |
| 安全事故 | D/E级项目 | 30% | 有安全事故记录 |
| 法律诉讼 | E级项目 | 50% | 有未结诉讼 |

### 5.4 经营品质验证

**验证方法**：

```sql
-- 验证经营品质分布
SELECT 
    CASE 
        WHEN indicator_score >= 90 THEN 'A级-卓越'
        WHEN indicator_score >= 80 THEN 'B级-优良'
        WHEN indicator_score >= 70 THEN 'C级-一般'
        WHEN indicator_score >= 60 THEN 'D级-预警'
        ELSE 'E级-危机'
    END as quality_level,
    COUNT(*) as project_count,
    COUNT(*) * 100.0 / SUM(COUNT(*)) OVER() as percentage
FROM fact_project_indicator
GROUP BY quality_level
ORDER BY quality_level;

-- 验证问题覆盖率
SELECT 
    problem_type,
    COUNT(DISTINCT project_key) as affected_projects,
    COUNT(DISTINCT project_key) * 100.0 / (SELECT COUNT(*) FROM dim_project) as coverage_rate
FROM (
    -- 应收账款逾期

    FROM fact_receivable
    WHERE overdue_days > 90

    SELECT project_key, '工期延误' as problem_type
    FROM fact_project_output

    SELECT project_key, '成本超支' as problem_type
    FROM fact_project_cost
    WHERE actual_cost > budget_cost * 1.10
```

### 6.2 波动性保障策略

> **核心原则**：数据必须有合理的波动，不是"波澜不惊"，而是"有意外的真实"。**意外不能形成规律**，必须是随机发生的。

**波动性类型**：

| 波动类型 | 定义 | 示例 | 年发生概率 | 月度发生概率 |
|---------|------|------|-----------|-------------|
| **正常波动** | 符合正态分布的常规波动 | 合同金额在均值的±20%内 | 100% | 100% |
| **正向极端** | 显著高于常规的异常值 | 常规1亿合同，突然签了10亿大单 | 20-30% | 2-3% |
| **负向极端** | 显著低于常规的异常值 | 常规5000万合同，突然签了500万小单 | 10-15% | 1-2% |
| **结构性变化** | 业务模式变化导致 | 新业务线、突发事件 | 5-10% | 0.5-1% |

**时间维度无规律保障**：

```python
import numpy as np
import random
from datetime import datetime, timedelta

def generate_volatility_event(base_value, year_start, year_end, volatility_config):
    """
    生成具有时间随机性的波动事件
    
    参数：
        base_value: 基准值
        year_start: 起始年份
        year_end: 结束年份
        volatility_config: 波动配置
        
    返回：
        波动事件列表（包含时间和波动值）
    """
    events = []
    current_date = datetime(year_start, 1, 1)
    end_date = datetime(year_end + 1, 1, 1)
    
    while current_date < end_date:
        # 计算本月发生波动事件的概率
        month_prob = volatility_config['monthly_prob']
        
        if random.random() < month_prob:
            # 决定波动类型
            type_weights = [
                volatility_config['positive_extreme_weight'],
                volatility_config['negative_extreme_weight'],
                volatility_config['structural_change_weight']
            ]
            event_type = random.choices(
                ['positive_extreme', 'negative_extreme', 'structural_change'],
                weights=type_weights
            )[0]
            
            if event_type == 'positive_extreme':
                # 正向极端：2-8倍
                multiplier = random.uniform(2, 8)
                value = round(base_value * multiplier, 2)
            elif event_type == 'negative_extreme':
                # 负向极端：0.1-0.5倍
                multiplier = random.uniform(0.1, 0.5)
                value = round(base_value * multiplier, 2)
            else:
                # 结构性变化：根据业务场景决定
                value = base_value * random.uniform(0.8, 1.2)
            
            events.append({
                'date': current_date.strftime('%Y-%m-%d'),
                'type': event_type,
                'value': value,
                'multiplier': value / base_value
            })
        
        # 随机跳过1-3个月，确保不会每月都发生
        current_date += timedelta(days=random.randint(28, 90))
    
    return events

# 配置示例：每年约2-3次正向极端事件，1-2次负向极端事件
volatility_config = {
    'monthly_prob': 0.03,  # 每月3%概率发生极端事件
    'positive_extreme_weight': 0.6,
    'negative_extreme_weight': 0.3,
    'structural_change_weight': 0.1
}

# 生成2024-2026年的波动事件
extreme_events = generate_volatility_event(
    base_value=100000000,  # 1亿基准
    year_start=2024,
    year_end=2026,
    volatility_config=volatility_config
)
```

### 6.3 噪声注入策略

**方法**：

```python
import numpy as np
import random

def inject_noise(value, noise_level=0.05, distribution='normal'):
    """
    在真实值基础上注入随机噪声
    
    参数：
        value: 原始值
        noise_level: 噪声水平（0.05表示5%的波动）
        distribution: 噪声分布类型 ('normal', 'uniform', 'lognormal')
    
    返回：
        添加噪声后的值
    """
    if distribution == 'normal':
        noise = np.random.normal(0, value * noise_level)
    elif distribution == 'uniform':
        noise = random.uniform(-value * noise_level, value * noise_level)
    elif distribution == 'lognormal':
        log_value = np.log(value + 1)
        noise = np.random.lognormal(log_value, noise_level) - value
    
    return round(value + noise, 2)

# 示例：为成本金额添加5%噪声
cost_amount_with_noise = inject_noise(1000000, noise_level=0.05, distribution='normal')
```

| 工期天数 | 5-10% | 施工波动 |

| 比率类 | 1-2% | 精度限制 |

```python
import random

def multi_source_mix(data_sources, mix_ratio=None):
    """
    从多个数据源混合数据
    
    参数：
        data_sources: 数据源列表 [{'source': 'A', 'data': [...], 'ratio': 0.5}, ...]
        mix_ratio: 混合比例，如果为None则按ratio参数
    
    返回：
        混合后的数据集
    """
    if mix_ratio is None:
        mix_ratio = [ds['ratio'] for ds in data_sources]
    
    total_ratio = sum(mix_ratio)
    mix_ratio = [r / total_ratio for r in mix_ratio]  # 归一化
    
    mixed_data = []
    for ds, ratio in zip(data_sources, mix_ratio):
        sample_size = int(len(ds['data']) * ratio)
        sampled = random.sample(ds['data'], min(sample_size, len(ds['data'])))
        mixed_data.extend(sampled)
    
    random.shuffle(mixed_data)
    return mixed_data

# 示例：混合3个目标企业的项目数据
data_sources = [
    {'source': '中国建筑', 'data': projects_cib, 'ratio': 0.4},
    {'source': '区域龙头A', 'data': projects_regional_a, 'ratio': 0.35},
    {'source': '问题企业B', 'data': projects_problem_b, 'ratio': 0.25}
]

mixed_projects = multi_source_mix(data_sources)
```

### 6.5 极端值处理策略

**极端值类型**：

| 类型 | 定义 | 处理策略 |
|------|------|---------|
| 合理极端 | 业务上可能发生 | 保留，但添加业务说明 |
| 不合理极端 | 业务上不可能发生 | 修正到合理范围 |
| 错误数据 | 明显错误 | 修正或删除 |

**极端值生成规则**：

```python
def generate_extreme_value(base_value, is_extreme=True, extreme_direction='both'):
    """
    生成极端值
    
    参数：
        base_value: 基础值
        is_extreme: 是否生成极端值
        extreme_direction: 极端方向 ('positive', 'negative', 'both')
    
    返回：
        极端值
    """
    if not is_extreme:
        return base_value
    
    # 极端值范围：基础值的2-5倍
    multiplier = random.uniform(2, 5)
    
    if extreme_direction == 'positive':
        return round(base_value * multiplier, 2)
    elif extreme_direction == 'negative':
        return round(base_value * multiplier * -1, 2)
    else:  # both
        direction = random.choice([-1, 1])
        return round(base_value * multiplier * direction, 2)

# 示例：生成极端成本
extreme_cost = generate_extreme_value(1000000, is_extreme=True, extreme_direction='positive')
```
