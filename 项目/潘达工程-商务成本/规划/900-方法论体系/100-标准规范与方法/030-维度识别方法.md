## 二、维度识别的核心问题

> **核心原则**：维度识别是连接"业务理解"和"本体设计"的关键桥梁，必须系统回答以下问题。

### 2.1 五大判断问题

| 问题类型 | 核心问题 | 判断标准 | 示例 |
|---------|---------|---------|------|
| **字段识别** | 哪些字段是"分类/编码+名称"的组合？ | 字段包含 `*_code` + `*_name` 或类似模式 | `subject_code` + `subject_name` |
| **层级判断** | 这个分类体系是否需要层级结构？ | 存在明确的父子关系（如 L1/L2/L3） | 成本科目三级分类 |
| **复用性** | 这个分类是否会重复出现在多个事实表中？ | 同一编码在多个表中出现 | 收支科目在多个事实表中出现 |
| **维护性** | 这个分类是否需要独立维护？ | 新增分类项是否需要更新多个事实表 | 新增成本科目需更新多个成本表 |
| **编码标准** | 编码是否需要符合行业标准？ | 是否存在行业通用的编码规范 | 建筑工程行业成本科目编码标准 |

### 2.2 维度识别的决策树

```
┌─────────────────────────────────────────────────────────────┐
│                     维度识别决策树                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  字段是否为"编码+名称"组合？                                 │
│     │                                                       │
│     ├─ 否 → 不是维度                                        │
│     │                                                       │
│     └─ 是 → 是否需要层级结构？                              │
│              │                                              │
│              ├─ 否 → 扁平维度（如项目状态）                 │
│              │                                              │
│              └─ 是 → 是否在多个事实表中出现？               │
│                       │                                     │
│                       ├─ 否 → 嵌入事实表                    │
│                       │                                     │
│                       └─ 是 → 是否需要独立维护？            │
│                                │                            │
│                                ├─ 否 → 嵌入事实表           │
│                                │                            │
│                                └─ 是 → 是否有行业标准编码？ │
│                                         │                   │
│                                         ├─ 否 → 企业编码     │
│                                         │                   │
│                                         └─ 是 → 行业编码     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 步骤2：复用性分析

**目标**：分析字段在多个表中的出现情况

**分析方法**：
```sql
-- 统计字段出现频率
SELECT
    field_name,
    COUNT(DISTINCT table_name) as table_count,
    GROUP_CONCAT(table_name) as tables
FROM table_fields
WHERE field_name LIKE '%_code'
GROUP BY field_name
ORDER BY table_count DESC;
```

**分析产出**：
```yaml
复用性分析结果:
  - 字段名: cost_code
    出现表数: 3
    出现表名: tb_project_major_cost, tb_project_cost_pay_detail, tb_project_cost_unconfirmed_analyse
    复用性: 高
    建议操作: 抽取为独立维度表

  - 字段名: subject_code
    出现表数: 1
    出现表名: tb_project_income_outcome_summary
    复用性: 低
    建议操作: 评估是否需要独立维度表（考虑未来扩展性）
```

### 3.4 步骤3：层级结构分析

**目标**：分析分类体系的层级关系

**分析方法**：
1. **父子关系识别**：分析编码的前缀关系
2. **层级深度识别**：统计编码的层级数量
3. **层级编码规则**：识别编码的命名规则

**分析产出**：
```yaml
层级结构分析结果:
  成本科目:
    - 层级深度: 3
    - 层级编码规则: L1(2位) + L2(2位) + L3(3位)
    - 示例: 01(人工成本) → 0101(基本工资) → 010101(计时工资)
    - 层级关系:
      - L1: 01-人工成本, 02-材料成本, 03-机械成本, 04-管理费用
      - L2: 0101-基本工资, 0102-绩效工资, 0201-主材, 0202-辅材
      - L3: 020101-钢筋, 020102-混凝土, 020103-水泥

  收支科目:
    - 层级深度: 2
    - 层级编码规则: L1(2位) + L2(2位)
    - 示例: 01(收入) → 0101(工程款收入)
    - 层级关系:
      - L1: 01-收入, 02-支出
      - L2: 0101-工程款收入, 0102-其他收入, 0201-材料费, 0202-人工费
```

### 3.5 步骤4：编码标准分析

**目标**：分析编码是否符合行业标准

**分析方法**：
1. **查询行业标准编码**：通过大模型查询行业通用编码标准
2. **对比企业现有编码**：对比企业编码与行业标准
3. **确定编码策略**：根据《000-顶层设计.md》编码标准规范确定策略

**分析产出**：
```yaml
编码标准分析结果:
  成本科目:
    - 行业标准: 存在建筑工程行业成本科目编码标准
    - 企业编码: 部分符合，部分自定义
    - 编码策略: 采用行业标准编码，企业编码仅作参考
    - 行业标准示例:
      - L1: 01-人工成本, 02-材料成本, 03-机械成本, 04-管理费用
      - L2: 0101-基本工资, 0102-绩效工资, 0201-主材, 0202-辅材
      - L3: 020101-钢筋, 020102-混凝土, 020103-水泥

  收支科目:
    - 行业标准: 无明确的行业标准
    - 企业编码: 企业自定义编码
    - 编码策略: 参考企业编码，按通识标准优化
    - 优化后编码:
      - L1: 01-收入, 02-支出
      - L2: 0101-工程款收入, 0102-其他收入, 0201-材料费, 0202-人工费
```

### 3.6 步骤5：维度表设计

**目标**：设计维度表结构

**设计规范**：
| 规范项 | 说明 | 示例 |
|-------|------|------|
| **表命名** | `dim_*` 前缀 + 维度名称 | `dim_cost_subject` |
| **主键** | `*_key` 字段，唯一标识 | `cost_subject_key` |
| **编码字段** | `*_code` 字段，业务编码 | `cost_code` |
| **名称字段** | `*_name` 字段，业务名称 | `cost_name` |
| **层级字段** | `*_level` 字段，层级标识 | `cost_level` |
| **父级字段** | `parent_*_key` 字段，父级关联 | `parent_cost_subject_key` |
| **时间字段** | `created_at`、`updated_at` | 创建时间、更新时间 |

**维度表设计示例**：
```sql
-- 成本科目维度表
CREATE TABLE dim_cost_subject (
    cost_subject_key VARCHAR(50) PRIMARY KEY COMMENT '主键',
    cost_code VARCHAR(10) NOT NULL COMMENT '成本科目编码',
    cost_name VARCHAR(100) NOT NULL COMMENT '成本科目名称',
    cost_level INT NOT NULL COMMENT '成本科目层级（1/2/3）',
    parent_cost_subject_key VARCHAR(50) COMMENT '父级科目主键',
    cost_type VARCHAR(20) COMMENT '成本类型（刚性/弹性）',
    is_rigid BOOLEAN COMMENT '是否为刚性成本',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_cost_code (cost_code),
    INDEX idx_parent_key (parent_cost_subject_key)
) COMMENT='成本科目维度表';

-- 收支科目维度表
CREATE TABLE dim_subject (
    subject_key VARCHAR(50) PRIMARY KEY COMMENT '主键',
    subject_code VARCHAR(10) NOT NULL COMMENT '收支科目编码',
    subject_name VARCHAR(100) NOT NULL COMMENT '收支科目名称',
    subject_type VARCHAR(20) NOT NULL COMMENT '科目类型（收入/支出）',
    subject_level INT NOT NULL COMMENT '科目层级（1/2）',
    parent_subject_key VARCHAR(50) COMMENT '父级科目主键',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_subject_code (subject_code),
    INDEX idx_parent_key (parent_subject_key)
) COMMENT='收支科目维度表';
```

### 3.7 步骤6：维度映射关系设计

**目标**：设计维度表与事实表的映射

**设计规范**：
| 规范项 | 说明 | 示例 |
|-------|------|------|
| **关联字段** | 事实表中的 `*_key` 字段 | `cost_subject_key` |
| **外键约束** | 关联维度表的主键 | `FOREIGN KEY (cost_subject_key) REFERENCES dim_cost_subject(cost_subject_key)` |
| **索引设计** | 在关联字段上创建索引 | `INDEX idx_cost_subject_key (cost_subject_key)` |

**映射关系设计示例**：
```sql
-- 项目成本事实表（修改后）
CREATE TABLE fact_project_cost (
    project_cost_key VARCHAR(50) PRIMARY KEY,
    project_key VARCHAR(50) NOT NULL,
    date_key VARCHAR(50) NOT NULL,
    cost_subject_key VARCHAR(50) NOT NULL COMMENT '关联成本科目维度表',
    cost_amount DECIMAL(18,2) NOT NULL COMMENT '成本金额',
    cost_level INT NOT NULL COMMENT '成本层级',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_key) REFERENCES dim_project(project_key),
    FOREIGN KEY (date_key) REFERENCES dim_date(date_key),
    FOREIGN KEY (cost_subject_key) REFERENCES dim_cost_subject(cost_subject_key),
    INDEX idx_project_key (project_key),
    INDEX idx_date_key (date_key),
    INDEX idx_cost_subject_key (cost_subject_key)
) COMMENT='项目成本事实表';

-- 项目收支事实表（修改后）
CREATE TABLE fact_project_balance (
    balance_key VARCHAR(50) PRIMARY KEY,
    project_key VARCHAR(50) NOT NULL,
    date_key VARCHAR(50) NOT NULL,
    subject_key VARCHAR(50) NOT NULL COMMENT '关联收支科目维度表',
    amount DECIMAL(18,2) NOT NULL COMMENT '金额',
    balance_type VARCHAR(20) NOT NULL COMMENT '收支类型（收入/支出）',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_key) REFERENCES dim_project(project_key),
    FOREIGN KEY (date_key) REFERENCES dim_date(date_key),
    FOREIGN KEY (subject_key) REFERENCES dim_subject(subject_key),
    INDEX idx_project_key (project_key),
    INDEX idx_date_key (date_key),
    INDEX idx_subject_key (subject_key)
) COMMENT='项目收支事实表';
```

## 五、维度识别的常见问题

### 5.1 问题1：如何判断是否需要独立维度表？

**判断标准**：
1. 字段是否为"编码+名称"组合？
2. 是否在多个事实表中出现？
3. 是否需要独立维护？
4. 是否有层级结构？

**决策树**：见§2.2

### 5.2 问题2：如何处理企业现有编码与行业标准编码的冲突？

**处理策略**：
> **引用**：《000-顶层设计.md》，编码标准规范（核心定义）。

| 编码类型 | 优先级 | 处理方式 |
|---------|-------|---------|
| **结构编码** | 通识优先 | 采用行业标准编码 |
| **业务编码** | 通识优先，参考专识 | 采用行业标准编码，企业编码仅作参考 |
| **属性值** | 专识优先 | 采用企业实际业务术语 |

### 5.3 问题3：如何处理维度表的层级关系？

**处理方式**：
1. **父子关系**：通过 `parent_*_key` 字段关联
2. **层级标识**：通过 `*_level` 字段标识层级
3. **层级编码**：通过编码前缀表示层级关系

**示例**：
```sql
-- 成本科目维度表（包含层级关系）
CREATE TABLE dim_cost_subject (
    cost_subject_key VARCHAR(50) PRIMARY KEY,
    cost_code VARCHAR(10) NOT NULL,
    cost_name VARCHAR(100) NOT NULL,
    cost_level INT NOT NULL COMMENT '1=L1, 2=L2, 3=L3',
    parent_cost_subject_key VARCHAR(50) COMMENT '父级科目主键',
    ...
    FOREIGN KEY (parent_cost_subject_key) REFERENCES dim_cost_subject(cost_subject_key)
);
```

### 5.4 问题4：如何处理维度表的数据填充？

**填充策略**：
1. **通识优先**：基于行业标准编码填充维度数据
2. **专识补充**：用企业实际业务术语补充属性值
3. **模拟数据**：用模拟数据填充缺失的数据

## 六、总结

### 6.1 核心原则

| 原则 | 说明 |
|------|------|
| **结构编码通识优先** | 维度编码采用行业标准编码 |
| **属性值专识优先** | 维度属性值采用企业实际业务术语 |
| **维度表独立维护** | 需要独立维护的分类体系必须设计为维度表 |
| **层级结构清晰** | 维度表的层级关系必须清晰定义 |
| **映射关系明确** | 维度表与事实表的映射关系必须明确 |

### 6.2 关键产出

| 产出 | 说明 |
|------|------|
| **维度表清单** | 需要独立维护的维度表列表 |
| **维度编码体系** | 维度的编码规则和层级结构 |
| **维度属性定义** | 维度表的字段定义 |
| **维度映射关系** | 维度表与事实表的映射关系 |
