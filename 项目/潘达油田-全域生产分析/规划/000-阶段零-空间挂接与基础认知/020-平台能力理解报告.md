## 一、平台架构总览

本体驱动智能决策平台将数据组织为**五层结构**，并在其上挂载函数、动作、规则，支撑业务决策。

### 1.1 五层架构模型

```
┌─────────────────────────────────────────────────────────────────────┐
│  应用层（面向用户）                                                    │
│  AI助手问答 │ 自然语言查询 │ 数据预览 │ Web报表 │ 可视化看板           │
├─────────────────────────────────────────────────────────────────────┤
│  函数 / 动作 / 规则层                                               │
│  Function（查询、分析）│ Action（可执行动作）│ Rule（规则）          │
├─────────────────────────────────────────────────────────────────────┤
│  本体层（Ontology）                                                  │
│  Object Type（业务对象）│ Link Type（对象关系）│ Action Type        │
│  └── 平台分类（ads_categories，强制对齐）                            │
├─────────────────────────────────────────────────────────────────────┤
│  Cube层（分析语义视图，需手动注册 register_cube）                     │
│  维度 │ 度量 │ 派生度量；bind_source 指向物理表                       │
├─────────────────────────────────────────────────────────────────────┤
│  物理层（Table）                                                      │
│  fact_*（事实表）│ dim_*（维表）│ bridge_*（桥表）│ dim_date（强制）  │
│  └── 表间关系 add_relationship（事实表 → 维表，many_to_one）           │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 三层区分的核心思想

| 层级 | 是什么 | 面向谁 | 命名风格 | 核心原则 |
|---|---|---|---|---|
| **物理表** | ClickHouse 中的事实表/维表 | DBA、ETL工程师 | snake_case，`{前缀}_{实体}` | 从表名即可判断存储角色（事实/维/桥） |
| **Cube** | 面向分析语义的数据集视图 | 分析人员、报表 | PascalCase + `Cube` 后缀 | 表达**分析主题与粒度**，可共享同一 fact 表 |
| **本体对象** | 业务世界中的"对象类型" | 业务人员、产品、AI | PascalCase，**无前缀** | 贴近**业务世界**，用**分类**而非前缀区分角色 |

**核心原则**：三层**刻意区分命名**，避免"一张表→一个对象→一个Cube同名"的一一复制。

```
  业务世界（本体）          分析语义（Cube）           存储落地（物理表）
  ─────────────────        ─────────────────         ─────────────────
  Well              ←──    WellProductionCube  ←──   fact_well_production
  Field             ←──    FieldAnalysisCube   ←──   dim_field
  Reservoir         ←──    ReservoirCube       ←──   dim_reservoir
  (时间分析)         ←──    (Cube 时间维)        ←──   dim_date + date_key
```

## 二、关键平台能力与规范

### 2.1 物理表能力与命名规范

| 规范项 | 要求 | 依据文档 |
|---|---|---|
| **7类表前缀（强制）** | `fact_`（事实表）、`dim_`（维表）、`bridge_`（桥表/多对多）、`map_`（映射/码表）、`cfg_`（配置）、`agg_`（预聚合）、`tmp_`（临时） | 命名规范 §2 |
| **`fact_*` 必须含 date_key** | 每一个事实表必须包含 `date_key` 字段并关联 `dim_date` | 命名规范 §2.3 |
| **`dim_date` 强制存在** | 每一个本体实现单元必须包含一张日期维表 `dim_date`，字段 `date_key` | 命名规范 §3 |
| **全小写 snake_case** | 物理表名、字段名统一小写 + 下划线 | 命名规范 §2.1 |
| **表间关系（强制分层）** | init 脚本中必须调用 `s.tables.add_relationship(...)`，通常为事实表 `many_to_one` 维表 | 规划指南 §物理表间关系 |

### 2.2 Cube 层能力与规范

| 规范项 | 要求 | 依据文档 |
|---|---|---|
| **Cube 层须独立成章** | 规划文档中必须有独立章节专门描述 Cube 层（维度、度量、派生度量） | 规划指南 |
| **PascalCase + Cube 后缀** | 命名：`WellProductionCube`、`FieldAnalysisCube` | 命名规范 §4 |
| **类别标注** | 标注 Process / Subject / Comparison / Snapshot 类别 | 命名规范 §4 |
| **bind_source** | Cube 须明确绑定数据来源（物理表） | 规划指南 |
| **手动注册** | init 脚本中需调用 `s.cubes.register_cube(...)` | 规划指南 |

### 2.3 本体层能力与规范

| 规范项 | 要求 | 依据文档 |
|---|---|---|
| **Object Type `code` 无前缀** | `Well`、`Field`、`Reservoir`、`ProductionRecord` —— 直接业务名词 PascalCase | 命名规范 §5 |
| **分类标注（而非前缀）** | 标注 Master / Transaction / Analytical 等**分类**，而非用前缀 | 命名规范 §5 |
| **链接两端必须是对象类型 `code`** | `Well` → `Reservoir`（业务关系），不是 Cube 名，不是表名 | 规划指南 §「业务世界」方法 |
| **平台分类对齐（强制）** | 规划语义分类（表/Cube/对象/关系/链接/函数）须与平台 `ads_categories` 使用同一套标准中文名 | 规划指南 §平台分类 |
| **分类挂载（独立脚本）** | 须有独立脚本 `*_category_mount.py`，在 init + seed + 全部函数 publish **之后** 执行 `apply_registry` | 规划指南 §平台分类 |

### 2.4 函数、动作、规则能力

| 能力 | 说明 | 技术实现 |
|---|---|---|
| **Function（函数）** | 封装业务查询、分析逻辑，可被 AI 调用 | Python 脚本，`s.functions.publish(...)` |
| **Action（动作）** | 可执行的业务动作，如预警、通知 | Python 脚本，`s.actions.publish(...)` |
| **Rule（规则）** | 业务规则定义 | 规则脚本 |
| **种子数据** | `*_seed_data.py`，幂等灌数，独立于 init | 本体脚本编写指南 |

### 2.5 实施顺序（强制）

来自《本体规划指南》的实施顺序，各项目阶段三严格按此顺序执行：

```
1. 建表 → TABLE_REGISTRY 定义表结构
2. tables.register_with_meta → 注册表元数据（显示名、说明）
3. tables.add_relationship → 添加表间关系（事实表 many_to_one 维表）
4. register_cube → 注册 Cube（分析语义视图）
5. define_object_type / define_link_type → 定义本体对象与链接
6. *_seed_data.py → 灌数（幂等）
7. functions/*.py → 发布函数并 --register-function-id
8. *_category_mount.py → 分类挂载（最后一步）
```

## 三、阶段执行要点

| 阶段 | 平台能力/规范要点 |
|---|---|
| **阶段零** | 理解上述规范，确认 space_id |
| **阶段一** | 基于"业务世界"方法蒸馏业务知识，**先对象和链接，再反推表/Cube** |
| **阶段二** | 规划文档中**独立成章**描述：物理表、表间关系、Cube层、本体对象、链接 |
| **阶段三** | 严格按上述实施顺序写 init 脚本：建表→注册→加关系→Cube→本体→灌数→函数→分类 |
| **阶段四** | 基于本体和 Cube 开发 Function/Action，数据接入须符合表结构 |
| **阶段五** | 业务应用与分析能力开发，依托已注册的 Function |
| **阶段六** | 本体优化迭代，调用 `dazi.onto` SDK 更新 |

## 四、须特别注意的"反模式"

以下情形在架构师规范中被列为**评审不合格**，须避免：

| 反模式 | 说明 | 如何避免 |
|---|---|---|
| **规划示例表名/Cube/对象逐字相同** | 与示例文件相同但无业务章节论证 | 所有命名须为业务论证后的产物 |
| **缺 Cube 独立章节即进入 register_cube** | 规划文档中没有 Cube 层章节描述 | 在阶段二规划文档中必须有独立 Cube 章节 |
| **本体对象设计与业务世界脱节** | Object Type 命名不符合业务语义 | 遵循"业务世界"方法，对象名贴近业务 |
