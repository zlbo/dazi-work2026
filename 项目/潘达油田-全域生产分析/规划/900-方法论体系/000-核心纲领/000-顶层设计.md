## 一、核心策略选择（潘达油田定制）

### 1.1 策略定位：相对理想型

本项目采用**"相对理想型"策略**，这是整个项目的基础指导思想：

| 维度 | 说明 |
|------|------|
| **理想化** | 本体架构和数据架构应该支撑油田全域生产分析的**完整、科学、面向未来**的状态 |
| **相对** | 理想架构必须基于**现实可获得的数据源**，不能脱离实际 |
| **平衡** | 在理想设计与现实约束之间寻求平衡 |

**策略实施要点**：
1. 基于石油天然气行业通识和大模型知识构建完整的业务本体模型
2. 避免现有资源的约束，先构建理想版本
3. 后续根据实际资源逐步优化和迭代

### 1.2 规划与实施分离原则

> **核心原则**：本规划文档仅负责**设计层面**的规划，**具体技术实施**（本体建模、数据流程开发等）必须使用架构师提供的标准方法执行。架构师的方法具有最高优先级和权威性。

### 1.4 油田行业特征适配原则

> **核心原则**：潘达油田作为石油天然气生产企业，其业务复杂性远超传统建筑工程行业，需针对油气行业特征进行特殊适配。

#### 1.4.1 行业特征分析

| 特征维度 | 石油天然气行业特征 | 对本体设计的影响 |
|---------|------------------|-----------------|
| **业务链长度** | 勘探→钻井→完井→采油→集输→处理，链条长且复杂 | 需要设计跨域关联关系，核心实体"井"贯穿多域 |
| **数据类型多样性** | 实时传感器数据、地质数据、设备数据、生产数据 | 需要支持多类型数据源的融合分析 |
| **空间分布特征** | 井场分布广、地理跨度大 | 需要设计空间维度和区域管理机制 |
| **专业领域深度** | 地质、油藏、钻井、采油、安全等多专业融合 | 需要多专业领域的本体对象设计 |
| **生产连续性要求** | 24/7不间断生产，设备故障影响大 | 需要设备健康管理和预警机制 |

#### 1.4.2 核心实体识别

**"井"(Well)是贯穿全业务链的核心实体**：
- 钻井工程（D2）：井身结构、钻井记录
- 完井改造（D3）：射孔、压裂、防砂
- 采油生产（D4）：产量记录、举升方式
- 增产措施（D5）：措施方案、效果评价
- 油藏工程（D7）：注采井组关联
- 设备资产（D9）：抽油机、电潜泵
- QHSE（D10）：完整性状态、安全监控

#### 1.4.3 业务域划分原则

采用**10域业务模型**（参考WorkBuddy框架）：

| 业务域 | 核心对象 | 业务定位 |
|--------|---------|---------|
| D1 地质勘探 | 地质构造、储层、圈闭、勘探区块 | 资源发现与评价 |
| D2 钻井工程 | 井、井身结构、钻井平台、作业记录 | 井筒建设 |
| D3 完井与改造 | 完井方案、射孔、压裂、防砂 | 储层改造与投产准备 |
| D4 采油采气生产 | 生产井、油气田、产量记录、生产状态 | 油气产出 |
| D5 增产措施 | 措施方案、作业记录、效果评价、修井 | 产量提升 |
| D6 生产动态管理 | 产量指标、递减分析、含水率分析、异常事件 | 生产监控与分析 |
| D7 油藏工程 | 油藏、注采井组、注水井、压力监测 | 油藏管理与能量补充 |
| D8 地面集输 | 集输管网、处理站、计量站、注水系统 | 油气收集与输送 |
| D9 设备资产 | 抽油机、电潜泵、压缩机、设备维护记录 | 设备管理 |
| D10 QHSE与井完整性 | 井完整性状态、安全事件、环保记录 | 安全与合规 |

### 1.5 规划与架构师文档关系

> **核心原则**：本项目采用"规划与实施分离"策略，规划文档负责业务设计，架构师文档负责技术规约，两者通过明确的引用机制协同工作。

#### 1.5.1 职责边界

| 维度 | 规划文档（核心纲领） | 架构师文档 |
|------|---------------------|-----------|
| **业务定义** | ✅ 业务域、业务对象、业务关系 | ❌ |
| **架构设计** | ✅ 三层架构设计、数据流向 | ❌ |
| **技术规约** | ❌ | ✅ 命名规范、格式要求、API使用 |
| **实施细节** | ❌ | ✅ SDK脚本编写、发布流程、测试方法 |
| **方法论** | ✅ 业务方法论（六层次蒸馏） | ✅ 技术方法论（脚本编写指南） |

#### 1.5.2 引用机制

规划文档必须通过引用而非重复架构师文档内容：

| 架构师文档 | 引用场景 |
|-----------|---------|
| 《本体规划指南》 | 本体规划流程、文档结构规范 |
| 《本体命名规范》 | Object Type、Cube、物理表命名 |

**引用格式**：`参考《本体规划指南》§3 Cube层设计要求`

#### 1.5.3 优先级规则

1. **架构师文档具有最高优先级**
2. 规划文档与架构师文档冲突时，以架构师文档为准
3. 规划文档可以补充业务特定的规则，但不得违反架构师规范
4. 架构师文档未覆盖的业务场景，由规划文档补充

### 1.8 数据来源标记体系规范

> **核心原则**：所有业务知识和规则必须标记来源，确保可追溯性和可信度。采用**轻量级标记体系**，避免过度复杂。

#### 标记维度

| 标记项 | 说明 | 示例 |
|-------|------|------|
| **来源类型** | 标记知识的可信度层级 | `企业业务素材` / `行业通用知识` |
| **来源说明** | 具体来源细节 | |

#### 不同场景的标记要求

| 场景 | 来源类型 | 来源说明要求 | 示例 |
|------|---------|-----------|------|
| **行业通识蒸馏** | 行业通用知识 | 标记行业领域 | `大模型蒸馏：石油天然气行业` |
| **数据库表萃取** | 企业业务素材 | 说明具体数据库、表名、字段 | `数据库：潘达油田 → tb_well_production` |
| **本体设计建模** | 混合 | 标记每个属性/关系的来源 | `来源：行业通用知识` |

#### 冲突解决优先级

| 优先级 | 来源类型 | 说明 |
|-------|---------|------|
| 1 | 企业业务素材 | 最高优先级，以企业实际业务为准 |
| 2 | 行业通用知识 | 仅作为参考，需验证后使用 |

### 1.9 编码标准规范

> **核心原则**：本项目采用**"结构编码通识优先，属性值专识优先"**的编码策略，确保行业通用性和企业业务准确性的平衡。

#### 编码分类与优先级

| 编码类型 | 优先级 | 说明 | 示例 |
|---------|-------|------|------|
| **结构编码** | 通识优先 | Object Type、Link Type、维度表结构 | `Well`、`belongsToField`、`dim_field` |
| **业务编码** | 通识优先，参考专识 | 井号、区块编码等业务编码 | 井号：`W-XX-001` |
| **属性值** | 专识优先 | 具体业务术语 | `抽油机`、`电潜泵ESP` |

## 二、项目阶段划分（共6阶段+阶段零）

> **说明**：以下为潘达油田项目定制的阶段划分，基于行业特征进行优化。

### 阶段零：空间挂接与基础认知
**目标**：完成项目启动的技术准备和业务认知工作  
**核心任务**：

> **【标准动作】方法论体系适配**：在任何具体工作开展之前，必须首先完成方法论体系对当前项目、当前行业的适配优化。

**标准动作详解**：
| 动作 | 说明 | 产出 |
|------|------|------|
| **方法论文档评估** | 评估现有方法论文档对当前行业的适用性 | 评估报告（保留/优化/删除清单） |
| **行业特征分析** | 分析目标行业的业务特征、核心实体、关键流程 | 行业特征分析报告 |
| **方法论适配优化** | 根据行业特征优化方法论文档（Object Type、Link Type、编码标准等） | 适配后的方法论文档 |
| **成果验收标准制定** | 根据新方法论制定阶段成果的验收标准 | 各阶段验收标准文档 |

**Step 0a**：数据空间挂接（确认space__panda_oilfield）  
**Step 0b**：平台架构理解（学习本体平台能力）  
**Step 0c**：对标企业业务体系梳理（梳理新疆油田行业特征、业务链、核心实体）

**产出**：
- 项目与数据空间绑定确认
- 平台能力理解报告
- 对标企业业务体系梳理报告
- 业务领域划分确认文档
- **方法论体系适配成果**（评估报告 + 优化后的方法论文档 + 验收标准）

### 阶段一：业务知识蒸馏萃取
**目标**：**先通过行业通识蒸馏形成完整的业务知识体系，再基于此萃取企业特有知识**  
**核心策略**：行业通识蒸馏是整个项目的**基础和骨架**，采用**融合式组织策略**，企业专识内容融合到核心知识体系和量化规则库中。

> **行业领域定位**：本项目的行业领域为**石油天然气行业**，涵盖油气勘探、开发、生产、集输全业务链。本体设计应具备足够的**行业代表性**和**专业性**。

> **深度蒸馏方法**：采用**六层次蒸馏框架**（L1-L6），具体操作方法参见《900-方法论体系/100-标准规范与方法/020-行业知识深度蒸馏方法论.md》。

**产出**：核心知识体系（8个文件）、量化规则库（7个文件）、知识骨架总览（2个文件）、业务语义映射（若干文件）

### 阶段二：本体设计与确认
**目标**：完成完整的本体规划文档，包括物理表、Cube、对象、链接、规则设计  
**核心任务**：
- 物理表设计（fact_*/dim_*表结构、表间关系）
- Cube层设计（维度、度量、派生度量、平台分类）
- Object Type设计（基于10域框架的核心实体）
- Link Type设计（跨域关系）
- Action Type设计（业务动作）
- 业务规则定义
- bind_source映射（Object Type到Cube绑定）

**产出**：
- 物理表设计文档
- Cube层设计文档
- Object Type设计文档
- Link Type设计文档
- Action Type设计文档
- 业务规则定义文档
- 本体设计确认文档

### 阶段三：本体建模与落地
**目标**：根据阶段二规划文档编写SDK脚本并发布验证，将设计转化为平台可执行的本体模型  
**核心任务**：
- SDK脚本编写（本体初始化脚本）
- 物理表注册
- 表间关系注册
- Cube注册
- Object Type注册
- Link Type注册
- Action Type注册
- bind_source绑定
- 脚本发布验证

**产出**：
- 本体初始化脚本
- 灌数脚本
- 分类挂载脚本
- 本体发布验证报告

### 阶段四：数据接入
**目标**：完成数据源映射、接入与对接，数据直接接入平台实际存在的本体建模表  
**产出**：数据源映射分析报告、数据源接入报告、数据对接报告、模拟数据计划

### 阶段五：业务应用与分析能力开发
**目标**：基于本体和数据源，开发业务应用与分析能力  
**产出**：基础查询函数、深度分析函数、预警Action、智能问答能力

### 阶段六：优化迭代
**目标**：总结成果，持续优化本体模型
