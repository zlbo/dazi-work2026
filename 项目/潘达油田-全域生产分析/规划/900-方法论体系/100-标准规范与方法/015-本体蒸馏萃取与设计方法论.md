## 二、行业领域定位（石油天然气行业）

> **引用**：《000-顶层设计.md》，§1.4「油田行业特征适配原则」

**石油天然气行业定位**：本项目以**石油天然气行业**为核心定位，聚焦油气勘探开发、生产运营全业务链。业务模型应具备足够的**行业代表性**和**专业性**。

### 2.1 行业特征分析

| 特征维度 | 石油天然气行业特征 | 对本体设计的影响 |
|---------|------------------|-----------------|
| **业务链长度** | 勘探→钻井→完井→采油→集输→处理，链条长且复杂 | 需要设计跨域关联关系 |
| **数据类型多样性** | 实时传感器、地质数据、设备数据、生产数据 | 需要支持多类型数据源融合 |
| **空间分布特征** | 井场分布广、地理跨度大 | 需要空间维度和区域管理 |
| **专业领域深度** | 地质、油藏、钻井、采油、安全等多专业融合 | 需要多专业领域本体对象 |
| **生产连续性** | 24/7不间断生产，设备故障影响大 | 需要设备健康管理和预警 |

### 2.2 10域业务模型定位

| 业务域 | 核心对象 | 业务定位 |
|--------|---------|---------|
| D1 地质勘探 | 地质构造、储层、圈闭、勘探区块 | 资源发现与评价 |
| D2 钻井工程 | 井、井身结构、钻井平台、作业记录 | 井筒建设 |
| D3 完井与改造 | 完井方案、射孔、压裂、防砂 | 储层改造与投产准备 |
| D4 采油采气生产 | 生产井、油气田、产量记录、生产状态 | 油气产出（核心） |
| D5 增产措施 | 措施方案、作业记录、效果评价、修井 | 产量提升 |
| D6 生产动态管理 | 产量指标、递减分析、含水率分析、异常事件 | 生产监控与分析（核心） |
| D7 油藏工程 | 油藏、注采井组、注水井、压力监测 | 油藏管理与能量补充 |
| D8 地面集输 | 集输管网、处理站、计量站、注水系统 | 油气收集与输送 |
| D9 设备资产 | 抽油机、电潜泵、压缩机、设备维护记录 | 设备管理 |
| D10 QHSE与井完整性 | 井完整性状态、安全事件、环保记录 | 安全与合规 |

## 三、核心策略：相对理想型

> **引用**：《000-顶层设计.md》§1.1「核心策略」

**这是本项目的核心指导思想**，必须贯穿所有阶段：

| 维度 | 说明 |
|------|------|
| **理想化** | 本体架构和数据架构应该支撑油田全域生产分析的**完整、科学、面向未来**的状态 |
| **相对** | 理想架构必须基于**现实可获得的数据源**，不能脱离实际 |
| **平衡** | 在理想设计与现实约束之间寻求平衡 |

### 3.1 三层架构的来源关系

```
┌──────────────────────────────────────────────────────────────────────┐
│                         本体层（Ontology）                            │
│  Object Type、Link Type、Action Type                                │
│  ← 基于油气业务模型设计（10域框架）                                   │
├──────────────────────────────────────────────────────────────────────┤
│                      Cube层（平台自动生成）                           │
│  维度、度量、聚合                                                    │
│  ← 基于**实际存在的物理表**自动生成                                  │
├──────────────────────────────────────────────────────────────────────┤
│                        物理层（Table）                               │
│  ┌─────────────────┐    ┌─────────────────┐                        │
│  │  【理想表】      │    │  【实际存在的表】│                        │
│  │  业务需求设计   │    │  现有数据库表    │                        │
│  └────────┬────────┘    └────────┬────────┘                        │
│           └──────────┬───────────┘                                  │
│                      ↓                                              │
│              【数据源映射分析】← 核心！必须明确！                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 3.2 关键原则

- **先设计理想，再找现实**：先明确"应该有什么数据"，再分析"有什么数据可用"
- **差距分析是核心**：`理想表` 与 `现实表` 之间的差距，必须有明确的填补方案
- **数据来源必须可追溯**：每个本体属性必须能追溯到具体的数据源
- **核心实体贯穿**："井"作为核心实体必须贯穿多领域

## 五、知识蒸馏框架（油田定制版）

> **引用**：《000-顶层设计.md》，§1.4「油田行业特征适配原则」

### 5.1 六层次知识蒸馏框架（针对油田复杂性优化）

针对油田业务的复杂性，六层次框架进行如下优化：

| 层次 | 名称 | 说明 | 油田业务重点内容 | 平台能力映射 |
|------|------|------|----------------|------------|
| **L1** | 基础业务知识 | 概念、流程、术语 | 油井、油田、储层、产量等核心概念 | Object Type（业务实体） |
| **L2** | 显性业务关系 | 实体关联、数据流向 | 井→油藏→油田的归属关系、生产→集输的流向 | Link Type（包含、归属、关联） |
| **L3** | 隐性业务关系 | 经验常识、关联规则 | 产量递减规律、含水率与产量的关系、设备故障预警规则 | 派生属性/计算规则 |
| **L4** | 行业独特诀窍 | 特有规则、门道 | "黄金靶体"钻井技术、体积压裂工艺、页岩油开发经验 | 企业特有规则（ER） |
| **L5** | 行业通用流程规则 | 标准化流程、审批规则 | 开井/关井流程、措施作业流程、安全检查流程 | Action Type（业务动作） |
| **L6** | 维度识别 | 分类体系、编码体系 | 井型分类、区块编码、设备类型编码、时间维度 | 维度表（dim_*） |

### 5.2 油田业务的层次扩展

针对油田业务链长、专业领域多的特点，增加**跨域蒸馏**环节：

| 层次 | 跨域蒸馏重点 | 说明 |
|------|-------------|------|
| **跨域L1** | 核心实体识别 | 识别"井"作为贯穿7个领域的核心实体 |
| **跨域L2** | 跨域关系梳理 | 梳理井与各领域对象的关联关系 |
| **跨域L3** | 跨域规则融合 | 建立跨领域的业务规则（如产量-设备-安全关联） |

### 5.3 深度蒸馏操作流程

**五轮迭代蒸馏法（油田定制）**：

| 轮次 | 覆盖层次 | 核心目标 | 油田业务重点 |
|------|---------|---------|-------------|
| 第一轮 | L1+L2+跨域L1 | 快速建立完整的业务世界骨架 | 定义10域核心对象，识别"井"核心实体 |
| 第二轮 | L3+L4+跨域L2 | 深入挖掘隐性知识和行业诀窍 | 梳理产量递减、压裂工艺等专业知识 |
| 第三轮 | L5+跨域L3 | 完善流程规则和业务动作库 | 定义开井、关井、措施等业务流程 |
| 第四轮 | L6 | 识别需要独立维护的维度表和编码体系 | 井型、区块、设备类型等维度 |
| 第五轮 | 冲突解决 | 与企业实际数据对照，解决冲突 | 验证规则合理性，优化设计 |

## 六、数据来源标记体系规范

> **核心原则**：所有业务知识和规则必须标记来源，确保可追溯性和可信度。

### 6.1 标记维度定义

| 标记项 | 说明 | 取值范围 |
|-------|------|---------|
| **来源类型** | 标记知识的可信度层级 | `企业业务素材` / `行业通用知识` |
| **来源说明** | 具体来源细节 | 文本描述 |

### 6.2 不同场景的标记方法

| 场景 | 来源类型 | 来源说明要求 | 示例 |
|------|---------|-----------|------|
| **行业通识蒸馏** | 行业通用知识 | 标记行业领域 | `行业通用知识（大模型蒸馏：石油天然气行业）` |
| **数据库表萃取** | 企业业务素材 | 说明具体数据库、表名、字段 | `数据库：潘达油田 → tb_well_production` |
| **本体设计建模** | 混合 | 标记每个属性/关系的来源 | `来源：行业通用知识` |

### 6.3 编码标准规范

| 编码类型 | 优先级 | 说明 | 示例 |
|---------|-------|------|------|
| **结构编码** | 通识优先 | Object Type、Link Type、维度表结构 | `Well`、`belongsToField`、`dim_field` |
| **业务编码** | 通识优先，参考专识 | 井号、区块编码等 | 井号：`W-XX-001` |
| **属性值** | 专识优先 | 具体业务术语 | `抽油机`、`电潜泵ESP` |

#### 7.1.1 文件结构设计原则

**原则1：成果性质优先于工作流程**
- 文件按**成果性质**划分（核心知识体系、量化规则库）
- 便于后续阶段的引用和检索

- 企业专识内容融合到核心知识体系和量化规则库中

**原则3：建立双向关联机制**
- 流程、诀窍、规则库之间建立双向关联
- 便于本体设计时快速定位相关规则和流程

**原则4：跨域关系显性化**
- 明确标注跨域关系，体现"井"的核心贯穿作用

## 八、架构师规范强制检查

> **强制要求**：所有设计产出必须符合架构师规范，以下为必检项：

| 规范项 | 检查内容 | 依据文档 |
|--------|---------|---------|
| **物理表命名规范** | 所有实体必须映射到 `fact_*`（事实表）或 `dim_*`（维度表） | 本体命名规范 §2 |
| **dim_date 强制关联** | 所有事实表必须包含 `date_key` 字段并关联 `dim_date` | 本体命名规范 §2.3 |
| **Cube 层独立设计** | 指标体系必须明确映射到独立的 Cube | 本体规划指南 |
| **Cube 命名规范** | Cube 名必须为 PascalCase + `Cube` 后缀 | 本体命名规范 §4 |
| **Object Type 命名** | Object Type 的 `code` 必须为无前缀 PascalCase | 本体命名规范 §5 |
| **链接语义** | Link Type 的两端必须是 Object Type `code` | 本体规划指南 |
