# 项目文档管理规范

## 一、文件命名规范

### 1.1 命名格式
```
[三位数字前缀]-[文档名称].md
```

### 1.2 数字前缀号段划分

| 号段范围 | 用途 | 说明 |
|---------|------|------|
| 000-049 | 项目规划与目标 | 项目整体规划、阶段目标、需求分析 |
| 050-099 | 业务背景与知识 | 行业研究、业务术语、指标体系 |
| 100-199 | 数据与报表分析 | 结构分析、字段映射、数据血缘 |
| 200-299 | 本体与模型设计 | 本体模型、数据模型、指标定义 |
| 300-399 | 技术实现方案 | ETL流程、API设计、系统架构 |
| 400-499 | 测试与验证 | 测试用例、验证报告、数据质量 |
| 500-599 | 部署与运维 | 部署方案、运维手册、监控配置 |
| 600-699 | 用户文档 | 使用手册、操作指南、培训材料 |
| 700-799 | 复盘与改进 | 项目复盘、问题总结、优化建议 |
| 900-999 | 方法论与经验 | 方法论、经验总结、最佳实践 |

### 1.3 命名原则
- **唯一性**：同一文件夹内文件名不得重复
- **可读性**：文档名称应清晰反映内容，避免缩写
- **可排序性**：利用数字前缀实现自然排序
- **稳定性**：文件名确定后尽量不修改，如需修改需同步更新引用

## 二、文件夹组织规范

### 2.1 文件夹命名格式
```
[三位数字前缀]-[文件夹名称]
```

### 2.2 创建子文件夹的时机

| 场景 | 是否创建子文件夹 | 说明 |
|------|----------------|------|
| **阶段划分** | 是 | 如"阶段一-数据与报表映射分析" |
| **模块划分** | 是 | 如不同系统模块、功能模块 |
| **版本管理** | 是 | 如V1.0、V2.0版本 |
| **成果类型** | 是 | 如图表、代码、文档等 |
| **临时文件** | 是 | 独立存放，便于清理 |

### 2.3 文件夹结构示例
```
规划/
├── 000-阶段零-空间挂接与基础认知/
│   ├── 010-数据空间挂接确认.md
│   └── 020-平台能力理解报告.md
├── 100-阶段一-业务知识蒸馏萃取/
│   ├── 010-业务背景研究.md
│   ├── 020-L1业务实体蒸馏成果.md
│   └── 030-L2业务关系蒸馏成果.md
├── 200-阶段二-本体设计与建模/
│   ├── 010-本体设计文档.md
│   └── 020-维度识别成果.md
└── 900-方法论体系/
    ├── 000-核心纲领/
    │   ├── 000-顶层设计.md
    │   └── 010-项目总体规划.md
    └── 100-标准规范与方法/
        └── 010-项目文档管理规范.md
```

### 2.4 文件夹排序优势
- **数字前缀**：确保文件夹按逻辑顺序排列
- **统一格式**：便于快速定位和查找
- **扩展性**：预留号段便于后续新增

## 三、文档内容规范

### 3.1 单一数据源原则
- **核心定义唯一**：所有核心概念、框架、标准必须在顶层设计文档中定义
- **引用机制**：其他文档通过引用指向顶层设计，避免重复定义
- **变更流程**：核心定义的变更必须首先更新顶层设计，再同步更新引用文档

### 3.2 引用格式规范
```markdown
> **引用**：[文档名称]，[章节名称]。
```

**示例**：
```markdown
> **引用**：《000-顶层设计.md》，五层次知识蒸馏框架（核心定义）。
```

### 3.3 避免嵌套写法
- 禁止在文档中嵌入容易过时的具体内容（如"含三个层次的知识"）
- 如需说明内容范围，应使用引用或链接，而非直接嵌入
- 保持文档内容的灵活性和可维护性

### 3.4 文档版本控制
- 文档版本号格式：Vx.y（主版本.次版本）
- 主版本变更：核心定义、结构发生重大变化
- 次版本变更：内容补充、细节优化
- 变更日志：记录版本变更内容、日期、责任人

## 四、业务背景研究方法论

### 4.1 研究时机

| 阶段 | 是否进行研究 | 研究重点 |
|------|------------|---------|
| **项目启动前** | 是 | 行业概况、核心业务流程 |
| **需求分析阶段** | 是 | 业务术语、指标体系 |
| **数据映射阶段** | 持续 | 指标计算规则、业务逻辑 |
| **模型构建阶段** | 持续 | 业务实体关系、领域知识 |
| **项目复盘阶段** | 是 | 知识沉淀、经验总结 |

### 4.2 研究方法

#### 方法一：资料收集法
- 搜索行业报告、白皮书
- 收集行业标准、规范文件
- 研究竞争对手解决方案

#### 方法二：指标体系分析法
- 整理核心业务指标
- 明确指标定义和计算公式
- 建立指标间的逻辑关系

#### 方法三：专家访谈法
- 与业务人员沟通
- 了解实际业务场景
- 验证分析结论

### 4.3 研究成果输出
- 业务背景知识文档
- 核心指标字典
- 业务术语表
- 业务流程图

## 五、数据与本体映射分析方法

### 5.1 分析流程

```
1. 业务理解 → 2. 本体设计 → 3. 数据映射 → 4. 验证确认
     ↓              ↓              ↓              ↓
   业务背景研究   Object Type定义   物理表映射    数据质量验证
```

### 5.2 具体步骤

#### 步骤1：业务理解
- 研究行业背景和业务知识
- 识别核心业务实体
- 理解业务指标和计算规则

#### 步骤2：本体设计
- 定义Object Type（井、区块、油藏等）
- 定义Link Type（井-区块关系、井-油藏关系等）
- 定义Action Type（开井、关井、注水等）

#### 步骤3：数据映射
- **实体映射**：业务实体 → Object Type
- **关系映射**：业务关系 → Link Type
- **动作映射**：业务动作 → Action Type
- **指标映射**：业务指标 → 派生度量

#### 步骤4：验证确认
- 验证本体设计的完整性
- 验证业务覆盖度
- 确认数据模拟方案的合理性

### 5.3 映射矩阵模板

| 业务概念 | Object Type | 属性映射 | Link Type | 备注 |
|---------|------------|---------|----------|------|
| 井 | Well | well_code, well_name | locatedIn | 核心实体 |
| 区块 | Block | block_code, block_name | - | 维度实体 |

### 5.4 油田行业特点

| 特点 | 说明 |
|------|------|
| **核心实体** | 井(Well)是最核心的跨域实体 |
| **多域关联** | 一口井涉及地质、工程、生产等多个域 |
| **时间序列** | 生产数据具有明显的时间序列特征 |
| **模拟数据** | 本项目采用模拟数据，无需对接真实数据源 |

## 六、迭代式工作方法

### 6.1 迭代原则
- **小步快跑**：将大任务拆分为小目标
- **持续验证**：每完成一部分就进行验证
- **反馈驱动**：根据反馈及时调整方向
- **文档同步**：保持文档与实际工作同步更新

### 6.2 迭代流程
```
计划 → 执行 → 验证 → 复盘 → 调整 → 再计划
```

### 6.3 适用场景
- 需求不明确时
- 业务复杂难以一次性理解时
- 需要快速产出阶段性成果时
- 需要根据反馈调整方向时

## 七、跨项目复用建议

### 7.1 通用方法论
- **文档管理规范**：适用于所有项目
- **本体设计方法**：适用于本体驱动平台
- **迭代工作流程**：适用于复杂项目管理

### 7.2 行业适配点
- **业务背景**：需针对具体行业重新研究（油田 vs 建筑工程）
- **核心实体**：Object Type定义需完全替换（井 vs 项目）
- **业务术语**：需重新定义行业术语（井型 vs 项目类型）
- **指标体系**：需适配行业特定指标（产量 vs 产值）

### 7.3 从建筑工程到油田的适配示例

| 原建筑工程概念 | 油田行业对应概念 |
|--------------|----------------|
| 项目 (Project) | 井 (Well) / 区块 (Block) |
| 合同 (Contract) | 井生产合同 / 作业合同 |
| 产值 (Output Value) | 产量 (Production) |
| 成本 (Cost) | 作业成本 / 开发成本 |
| 供应商 (Supplier) | 作业队伍 / 设备供应商 |

### 7.4 复用步骤
1. 保留通用方法论（文档管理、版本控制、迭代流程）
2. 基于新行业重新定义Object Type和Link Type
3. 重新研究行业背景和业务术语
4. 调整指标体系和计算规则
5. 持续迭代优化
