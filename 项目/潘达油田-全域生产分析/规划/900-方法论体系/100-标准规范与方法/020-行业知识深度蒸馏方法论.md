> **版本**：V2.0（潘达油田-全域生产智能决策体系定制）  
> **创建日期**：2026年6月13日  
> **适用范围**：潘达油田-全域生产智能决策体系建设  
> **依据规范**：《000-顶层设计.md》（六层次知识蒸馏框架）、《本体规划指南》《本体命名规范》

### 1.3 行业领域定义

> **石油天然气行业定位**：本项目以**石油天然气行业**（油气生产企业）为核心定位，业务模型应具备足够的**行业代表性**和**专业性**。

**行业覆盖范围**：

| 细分领域 | 核心业务 | 本体设计考量 |
|---------|---------|-------------|
| 地质勘探 | 地震勘探、储层评价、储量计算 | 提供资源基础数据 |
| 钻井工程 | 井身结构、钻井作业、固井测井 | 井筒建设核心 |
| 完井改造 | 射孔、压裂、防砂 | 储层改造与投产准备 |
| **油气生产** | 采油采气、产量监控、生产状态 | **核心域，直接产出** |
| 增产措施 | 酸化、压裂、修井 | 产量提升手段 |
| **生产动态管理** | 产量分析、递减分析、预警管理 | **核心域，决策支持** |
| 油藏工程 | 注采井组、压力监测、开发方案 | 油藏管理核心 |
| 地面集输 | 管网输送、油气处理、计量 | 基础设施 |
| 设备资产 | 抽油机、电潜泵、压缩机 | 生产装备支撑 |
| QHSE | 井完整性、安全环保、合规检测 | 安全合规底线 |

**业务通用性原则**：
- 核心业务概念（如井、产量、设备）应适用于所有细分领域
- 特有概念（如压裂工艺、注采井组）作为扩展属性处理
- 本体设计优先考虑通用性，通过继承机制实现扩展性
- **核心原则**："井"作为贯穿多领域的核心实体，必须在各领域中保持一致性

## 二、平台能力与知识映射

> **核心原则**：深度蒸馏的产出必须能够映射到平台能力，确保设计可落地。

| 平台能力 | 能力说明 | 知识蒸馏的对应产出 |
|---------|---------|------------------|
| **Object Type** | 业务实体类型 | L1基础业务实体 |
| **Link Type** | 实体间关系 | L2显性关系、L3隐性关系、跨域关系 |
| **Action Type** | 业务动作 | L5流程规则中的业务动作 |

### 2.3 蒸馏产出的平台适配检查

> **检查要求**：每个蒸馏产出必须通过以下检查，确保可映射到平台能力。

| 检查项 | 检查内容 | 不通过时的处理 |
|--------|---------|--------------|
| Object Type 可定义 | 实体是否有明确的属性集合 | 拆分为更小的实体或合并到已有实体 |
| Link Type 可定义 | 关系是否有明确的语义 | 定义为关联属性而非独立关系 |
| Action Type 可定义 | 动作是否有明确的触发条件和执行主体 | 降级为派生属性或查询函数 |
| Cube 可生成 | 是否有事实表+维度表结构 | 调整表结构设计 |
| 派生属性可计算 | 是否有明确的计算规则和数据来源 | 标记为待实现，后续迭代 |

### 2.4 架构师规范强制检查

> **强制要求**：所有蒸馏产出必须符合架构师规范，以下为必检项：

| 规范项 | 检查内容 | 依据文档 |
|--------|---------|---------|
| **物理表命名规范** | 所有实体必须映射到 `fact_*`（事实表）或 `dim_*`（维度表） | 本体命名规范 §2 |
| **dim_date 强制关联** | 所有事实表必须包含 `date_key` 字段并关联 `dim_date` | 本体命名规范 §2.3 |
| **Cube 层独立设计** | 指标体系必须明确映射到独立的 Cube，包含维度、度量、派生度量 | 本体规划指南 |
| **Cube 命名规范** | Cube 名必须为 PascalCase + `Cube` 后缀（如 `WellProductionCube`） | 本体命名规范 §4 |
| **Object Type 命名** | Object Type 的 `code` 必须为无前缀 PascalCase（如 `Well`） | 本体命名规范 §5 |
| **分类标注** | 所有资源必须标注平台分类（Master / Transaction / Analytical 等） | 本体分类规划与SDK扩展方案 |
| **链接语义** | Link Type 的两端必须是 Object Type `code`，而非表名或 Cube 名 | 本体规划指南 |

### 3.2 各轮次详细设计

#### 第一轮：广度扫描（L1 基础知识 + L2 显性关系）

**核心目标**：快速建立完整的业务世界骨架，识别核心实体"井"

**前置准备**：
1. 明确石油天然气行业的业务边界（10域业务模型）
2. 确定业务分析的视角（全域生产分析）
3. 准备石油天然气行业的标准术语参考

**蒸馏步骤**：

| 步骤 | 任务 | 提示词要点 | 产出 |
|------|------|-----------|------|
| 1.1 | 领域全景扫描 | "石油天然气生产企业的完整业务领域包含哪些？" | 领域全景图（10域） |
| 1.2 | 核心实体识别 | "列举油田生产管理中的所有核心业务实体" | 实体清单（≥50个） |
| 1.3 | 实体属性定义 | "每个核心实体包含哪些关键属性？" | 实体-属性矩阵 |
| 1.4 | 显性关系梳理 | "这些实体之间存在哪些明确的业务关系？" | 显性关系图（≥50条） |
| 1.5 | 核心实体确认 | "哪个实体贯穿多个领域？为什么？" | 核心实体确认报告 |

## 四、文件结构优化原则（油田定制）

### 4.1 优化原则

| 原则 | 说明 | 适用场景 |
|------|------|---------|
| **成果性质优先** | 文件按成果性质划分，而非工作流程阶段 | 便于后续引用和检索 |
| **企业专识融合** | 企业专识融合到核心知识体系，而非独立管理 | 保持规则间的关联性 |
| **双向关联机制** | 流程、诀窍、规则库之间建立双向关联 | 便于快速定位相关规则 |
| **跨域关系显性化** | 明确标注跨域关系，体现核心实体贯穿作用 | 业务链长、多领域融合的项目 |
| **核心实体识别** | 识别贯穿多领域的核心实体作为本体设计枢纽 | 复杂业务体系的本体建模 |
| **数据来源标记** | 所有内容必须标注来源类型和来源说明 | 确保知识可追溯性 |
