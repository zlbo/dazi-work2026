## 二、维度识别的核心问题

> **核心原则**：维度识别是连接"业务理解"和"本体设计"的关键桥梁，必须系统回答以下问题。

### 2.1 五大判断问题

| 问题类型 | 核心问题 | 判断标准 | 油田行业示例 |
|---------|---------|---------|-------------|
| **字段识别** | 哪些字段是"分类/编码+名称"的组合？ | 字段包含 `*_code` + `*_name` 或类似模式 | `well_type_code` + `well_type_name` |
| **层级判断** | 这个分类体系是否需要层级结构？ | 存在明确的父子关系（如 L1/L2/L3） | 区块三级分类（区域→区块→井组） |
| **复用性** | 这个分类是否会重复出现在多个事实表中？ | 同一编码在多个表中出现 | 井型在产量表、注入表、设备表中出现 |
| **维护性** | 这个分类是否需要独立维护？ | 新增分类项是否需要更新多个事实表 | 新增油藏类型需更新多个生产表 |
| **编码标准** | 编码是否需要符合行业标准？ | 是否存在行业通用的编码规范 | 石油行业井型分类标准 |

### 2.2 油田行业维度识别决策树

```
┌─────────────────────────────────────────────────────────────┐
│                   油田行业维度识别决策树                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  字段是否为"编码+名称"组合？                                 │
│     │                                                       │
│     ├─ 否 → 不是维度                                        │
│     │                                                       │
│     └─ 是 → 是否需要层级结构？                              │
│              │                                              │
│              ├─ 否 → 扁平维度（如井状态）                   │
│              │                                              │
│              └─ 是 → 是否在多个事实表中出现？               │
│                       │                                     │
│                       ├─ 否 → 嵌入事实表                    │
│                       │                                     │
│                       └─ 是 → 是否需要独立维护？            │
│                                │                            │
│                                ├─ 否 → 嵌入事实表           │
│                                │                            │
│                                └─ 是 → 是否有行业标准编码？ │
│                                         │                   │
│                                         ├─ 否 → 企业编码     │
│                                         │                   │
│                                         └─ 是 → 行业编码     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 三、油田行业核心维度识别

| 维度类别 | 维度名称 | 说明 | 层级 | 行业标准 |
|---------|---------|------|------|---------|
| **空间维度** | 井 | 井基础信息 | L1-L2 | SY/T 5473 |

| | 作业区 | 采油厂/作业区 | L1-L2 | 企业内部标准 |

### 4.2 步骤1：字段扫描

**目标**：从业务素材中提取所有"编码+名称"字段

**扫描对象**：
| 扫描对象 | 扫描方法 | 油田行业示例 |
|---------|---------|-------------|
| **生产数据表** | 分析表结构、字段命名 | `well_type_code`、`well_type_name` |
| **设备台账** | 分析表头、公式、下拉列表 | `equipment_type_code`、`equipment_type_name` |
| **行业知识骨架** | 分析术语表、分类体系 | 井型分类、举升方式分类 |

**扫描产出**：
```yaml
油田行业字段扫描清单:
  - 字段名: well_type_code
    字段类型: string
    关联字段: well_type_name
    出现位置: fact_well_production, fact_well_injection, fact_equipment_operation
    来源类型: 企业业务素材
    来源说明: 产量表、注入表、设备表中均包含井型字段

  - 字段名: block_code
    字段类型: string
    关联字段: block_name
    出现位置: fact_well_production, dim_well
    来源类型: 企业业务素材
    来源说明: 区块维度在产量表和井维度表中出现
```

### 4.3 步骤2：复用性分析

**目标**：分析字段在多个表中的出现情况

**分析方法**：
```sql
-- 统计字段出现频率
SELECT
    field_name,
    COUNT(DISTINCT table_name) as table_count,
    GROUP_CONCAT(table_name) as tables
FROM table_fields
WHERE field_name LIKE '%_code'
GROUP BY field_name
ORDER BY table_count DESC;
```

**油田行业分析产出**：
```yaml
复用性分析结果:
  - 字段名: well_key
    出现表数: 5
    出现表名: fact_well_production, fact_well_injection, fact_stimulation, fact_equipment_operation, dim_well
    复用性: 极高
    建议操作: 必须抽取为独立维度表（dim_well）

  - 字段名: block_key
    出现表数: 3
    出现表名: fact_well_production, dim_well, dim_block
    复用性: 高
    建议操作: 抽取为独立维度表（dim_block）

  - 字段名: well_type_code
    出现表数: 3
    出现表名: fact_well_production, fact_well_injection, dim_well
    复用性: 高
    建议操作: 抽取为独立维度表（dim_well_type）

  - 字段名: lifting_type_code
    出现表数: 2
    出现表名: fact_well_production, dim_well
    复用性: 中
    建议操作: 抽取为独立维度表（dim_lifting_type）或有井维度表内嵌
```

### 4.4 步骤3：层级结构分析

**目标**：分析分类体系的层级关系

**分析方法**：
1. **父子关系识别**：分析编码的前缀关系
2. **层级深度识别**：统计编码的层级数量
3. **层级编码规则**：识别编码的命名规则

**油田行业分析产出**：
```yaml
层级结构分析结果:
  # 区块层级
  区块:
    - 层级深度: 3
    - 层级编码规则: [区域(2位)]-[区块(2位)]-[井组(2位)]
    - 示例: 01(北) → 0101(北A区) → 010101(北A区1号井组)
    - 层级关系:
      - L1: 01-北, 02-南, 03-东, 04-西
      - L2: 0101-北A区, 0102-北B区, 0201-南A区, 0202-南B区
      - L3: 010101-北A区1号井组, 010102-北A区2号井组

  # 井型层级
  井型:
    - 层级深度: 2
    - 层级编码规则: [类型代码(2位)]-[序号(2位)]
    - 示例: PR(采油) → PR-01(自喷井), PR-02(抽油井)
    - 层级关系:
      - L1: PR-采油井, IN-注水井, MO-观察井, EX-评价井
      - L2: PR-01-自喷采油井, PR-02-抽油井, PR-03-电泵井, IN-01-注水井

  # 油藏类型层级
  油藏类型:
    - 层级深度: 2
    - 层级编码规则: [类型前缀(3位)]-[序号(2位)]
    - 示例: OVL-稠油, OSL-稀油, OGS-气顶
    - 层级关系:
      - L1: OVL-稠油, OSL-稀油, OGS-气顶
      - L2: OVL-01-常规稠油, OVL-02-超稠油, OSL-01-稀油油藏
```

### 4.5 步骤4：编码标准分析

**目标**：分析编码是否符合行业标准

**分析方法**：
1. **查询行业标准编码**：通过大模型查询行业通用编码标准
2. **对比企业现有编码**：对比企业编码与行业标准
3. **确定编码策略**：根据《000-顶层设计.md》编码标准规范确定策略

**油田行业分析产出**：
```yaml
编码标准分析结果:
  井型:
    - 行业标准: SY/T 5473-2020《石油天然气工业 油气井分类》
    - 企业编码: 部分符合，部分自定义
    - 编码策略: 采用石油行业标准编码，企业编码仅作参考
    - 行业标准示例:
      - PR: 采油井
      - IN: 注水井
      - MO: 观察井
      - EX: 评价井

  区块:
    - 行业标准: 无明确的行业标准
    - 企业编码: 企业自定义编码
    - 编码策略: 参考企业编码，按通识标准优化
    - 优化后编码:
      - L1: 01-北, 02-南, 03-东, 04-西
      - L2: 0101-北A区, 0102-北B区
```

### 4.6 步骤5：维度表设计

**目标**：设计维度表结构

**设计规范**：
| 规范项 | 说明 | 油田行业示例 |
|-------|------|-------------|
| **表命名** | `dim_*` 前缀 + 维度名称 | `dim_well`、`dim_block` |
| **主键** | `*_key` 字段，唯一标识 | `well_key`、`block_key` |
| **编码字段** | `*_code` 字段，业务编码 | `well_code`、`block_code` |
| **名称字段** | `*_name` 字段，业务名称 | `well_name`、`block_name` |
| **层级字段** | `*_level` 字段，层级标识 | `block_level` |
| **父级字段** | `parent_*_key` 字段，父级关联 | `parent_block_key` |
| **时间字段** | `created_at`、`updated_at` | 创建时间、更新时间 |

**油田行业维度表设计示例**：
```sql
-- 井维度表
CREATE TABLE dim_well (
    well_key VARCHAR(50) PRIMARY KEY COMMENT '主键',
    well_code VARCHAR(20) NOT NULL COMMENT '井号',
    well_name VARCHAR(100) NOT NULL COMMENT '井名称',
    well_type_code VARCHAR(10) NOT NULL COMMENT '井型编码',
    well_type_name VARCHAR(50) NOT NULL COMMENT '井型名称',
    block_key VARCHAR(50) NOT NULL COMMENT '所属区块主键',
    block_code VARCHAR(10) NOT NULL COMMENT '区块编码',
    lifting_type_code VARCHAR(10) COMMENT '举升方式编码',
    lifting_type_name VARCHAR(50) COMMENT '举升方式名称',
    reservoir_code VARCHAR(10) COMMENT '油藏类型编码',
    reservoir_name VARCHAR(50) COMMENT '油藏类型名称',
    spud_date DATE COMMENT '开钻日期',
    completed_date DATE COMMENT '完井日期',
    status VARCHAR(20) DEFAULT '生产' COMMENT '井状态',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_well_code (well_code),
    INDEX idx_block_key (block_key),
    INDEX idx_well_type (well_type_code),
    INDEX idx_lifting_type (lifting_type_code)
) COMMENT='井维度表';

-- 区块维度表
CREATE TABLE dim_block (
    block_key VARCHAR(50) PRIMARY KEY COMMENT '主键',
    block_code VARCHAR(10) NOT NULL COMMENT '区块编码',
    block_name VARCHAR(100) NOT NULL COMMENT '区块名称',
    block_level INT NOT NULL COMMENT '区块层级（1/2/3）',
    parent_block_key VARCHAR(50) COMMENT '父级区块主键',
    block_type VARCHAR(20) COMMENT '区块类型（开发区/滚动区）',
    reservoir_code VARCHAR(10) COMMENT '主要油藏类型编码',
    area_km2 DECIMAL(10,2) COMMENT '面积（平方公里）',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_block_code (block_code),
    INDEX idx_parent_key (parent_block_key),
    INDEX idx_block_level (block_level)
) COMMENT='区块维度表';

-- 井型维度表
CREATE TABLE dim_well_type (
    well_type_key VARCHAR(50) PRIMARY KEY COMMENT '主键',
    well_type_code VARCHAR(10) NOT NULL COMMENT '井型编码',
    well_type_name VARCHAR(50) NOT NULL COMMENT '井型名称',
    well_type_level INT NOT NULL COMMENT '井型层级（1/2）',
    parent_well_type_key VARCHAR(50) COMMENT '父级井型主键',
    description VARCHAR(200) COMMENT '井型说明',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_well_type_code (well_type_code),
    INDEX idx_parent_key (parent_well_type_key)
) COMMENT='井型维度表';

-- 举升方式维度表
CREATE TABLE dim_lifting_type (
    lifting_type_key VARCHAR(50) PRIMARY KEY COMMENT '主键',
    lifting_type_code VARCHAR(10) NOT NULL COMMENT '举升方式编码',
    lifting_type_name VARCHAR(50) NOT NULL COMMENT '举升方式名称',
    category VARCHAR(20) COMMENT '举升类别（自喷/机采）',
    description VARCHAR(200) COMMENT '说明',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_lifting_type_code (lifting_type_code)
) COMMENT='举升方式维度表';
```

### 4.7 步骤6：维度映射关系设计

**目标**：设计维度表与事实表的映射

**设计规范**：
| 规范项 | 说明 | 油田行业示例 |
|-------|------|-------------|
| **关联字段** | 事实表中的 `*_key` 字段 | `well_key`、`block_key` |
| **外键约束** | 关联维度表的主键 | `FOREIGN KEY (well_key) REFERENCES dim_well(well_key)` |
| **索引设计** | 在关联字段上创建索引 | `INDEX idx_well_key (well_key)` |

**油田行业映射关系设计示例**：
```sql
-- 井产量事实表
CREATE TABLE fact_well_production (
    production_key VARCHAR(50) PRIMARY KEY COMMENT '主键',
    well_key VARCHAR(50) NOT NULL COMMENT '关联井维度表',
    date_key VARCHAR(50) NOT NULL COMMENT '关联日期维度表',
    daily_oil_production DECIMAL(10,2) COMMENT '日产油量（吨/天）',
    daily_liquid_production DECIMAL(10,2) COMMENT '日产液量（方/天）',
    daily_gas_production DECIMAL(10,2) COMMENT '日产气量（方/天）',
    water_cut DECIMAL(5,2) COMMENT '含水率（%）',
    pump_efficiency DECIMAL(5,2) COMMENT '泵效（%）',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (well_key) REFERENCES dim_well(well_key),
    FOREIGN KEY (date_key) REFERENCES dim_date(date_key),
    INDEX idx_well_key (well_key),
    INDEX idx_date_key (date_key),
    INDEX idx_production_date (well_key, date_key)
) COMMENT='井产量事实表';

-- 井注入事实表
CREATE TABLE fact_well_injection (
    injection_key VARCHAR(50) PRIMARY KEY COMMENT '主键',
    well_key VARCHAR(50) NOT NULL COMMENT '关联井维度表',
    date_key VARCHAR(50) NOT NULL COMMENT '关联日期维度表',
    daily_water_injection DECIMAL(10,2) COMMENT '日注水量（方/天）',
    daily_gas_injection DECIMAL(10,2) COMMENT '日注气量（方/天）',
    injection_pressure DECIMAL(10,2) COMMENT '注入压力（MPa）',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (well_key) REFERENCES dim_well(well_key),
    FOREIGN KEY (date_key) REFERENCES dim_date(date_key),
    INDEX idx_well_key (well_key),
    INDEX idx_date_key (date_key)
) COMMENT='井注入事实表';

-- 设备运行事实表
CREATE TABLE fact_equipment_operation (
    equipment_key VARCHAR(50) PRIMARY KEY COMMENT '主键',
    well_key VARCHAR(50) NOT NULL COMMENT '关联井维度表',
    date_key VARCHAR(50) NOT NULL COMMENT '关联日期维度表',
    equipment_type_code VARCHAR(10) NOT NULL COMMENT '设备类型编码',
    equipment_type_name VARCHAR(50) NOT NULL COMMENT '设备类型名称',
    running_hours DECIMAL(10,2) COMMENT '运行时率（小时）',
    downtime_hours DECIMAL(10,2) COMMENT '停机时间（小时）',
    failure_count INT DEFAULT 0 COMMENT '故障次数',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (well_key) REFERENCES dim_well(well_key),
    FOREIGN KEY (date_key) REFERENCES dim_date(date_key),
    INDEX idx_well_key (well_key),
    INDEX idx_date_key (date_key),
    INDEX idx_equipment_type (equipment_type_code)
) COMMENT='设备运行事实表';
```

## 六、总结

### 6.1 核心原则

| 原则 | 说明 |
|------|------|
| **维度复用性优先** | 优先识别跨表复用字段，设计独立维度表 |
| **行业标准优先** | 存在行业标准的维度，优先采用行业标准编码 |
| **层级结构清晰** | 有层级结构的维度，必须设计层级关系 |
| **维度完整性** | 确保维度表覆盖所有业务分析需求 |
