## 一、方法论概述

### 1.1 定义与目的

**演示场景**是项目中用于展示本体能力的具体业务场景，是项目成果的集中展示窗口。本方法论定义演示场景的设计、实施、验证全流程，确保演示场景具有**高仿真度、业务价值、可推广性**。

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **业务代表性** | 场景必须代表企业核心业务流程 |
| **决策价值** | 提供有价值的决策支持能力 |
| **技术完整性** | 展示平台核心能力（本体、AI、可视化） |
| **可复用性** | 设计模式可推广到其他场景 |

### 1.3 适用范围

| 场景类型 | 适用说明 |
|---------|---------|
| 单实体深度分析 | 如单井、单设备、单项目 |
| 业务流程演示 | 如生产监控、设备维护 |
| 智能决策支持 | 如AI问答、预警分析 |
| 沙盘推演 | 如方案对比、预测分析 |

| 优先级 | 判定条件 | 说明 |
|-------|---------|------|

## 四、Phase 2：场景设计

### 4.1 功能设计方法

#### 4.1.1 功能分层模型

```
┌─────────────────────────────────────────────────────────────┐
│                      功能分层                              │
├─────────────────────┬─────────────────────┬───────────────┤
│   展示层            │   分析层            │   数据层      │
│   (Presentation)   │   (Analysis)        │   (Data)      │
├─────────────────────┼─────────────────────┼───────────────┤
│ • 可视化看板       │ • 趋势分析          │ • 数据采集    │
│ • AI对话交互       │ • 根因分析          │ • 数据清洗    │
│ • 报表生成         │ • 预警检测          │ • 数据存储    │
│ • 导出功能         │ • 预测模拟          │ • 数据接口    │
└─────────────────────┴─────────────────────┴───────────────┘
```

#### 4.1.2 功能设计原则

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个功能模块只负责一个核心任务 |
| **松耦合** | 模块间通过接口交互，降低依赖 |
| **可扩展** | 预留扩展接口，支持未来迭代 |
| **用户友好** | 符合用户操作习惯 |

### 4.2 数据需求设计方法

#### 4.2.1 数据需求分析步骤

1. **识别核心指标**：列出场景需要的所有指标
2. **定义数据来源**：确定每个指标的数据来源
3. **分析数据质量**：评估数据完整性、准确性要求
4. **设计验证规则**：定义数据验证方法

#### 4.2.2 数据质量要求

| 指标 | 要求 |
|------|------|
| 完整性 | 数据覆盖率 ≥95% |
| 准确性 | 数值误差 ≤1% |
| 时效性 | 数据延迟 ≤24小时 |
| 一致性 | 跨表数据一致 |

| 分类 | 说明 | 示例 |
|------|------|------|
| **查询类** | 获取基础信息 | 产量查询、设备状态 |

## 五、Phase 3：场景实施

### 5.1 本体建模要求

#### 5.1.1 核心Object优先原则

| 优先级 | Object Type | 说明 |
|-------|------------|------|
| P0 | 核心实体（如Well） | 场景核心，必须优先 |
| P1 | 关联实体（如Equipment） | 支撑核心功能 |
| P2 | 辅助实体（如Price） | 完善场景数据 |

#### 5.1.2 本体建模检查清单

| 检查项 | 要求 |
|--------|------|
| Object Type | 核心属性完整 |
| Link Type | 关系定义正确 |
| Action Type | 业务动作定义 |
| bind_source | 正确绑定数据源 |

### 5.2 数据接入方法

#### 5.2.1 数据接入流程

1. **数据源评估**：评估数据源类型和质量
2. **数据映射**：建立物理表到本体的映射
3. **ETL开发**：开发数据抽取转换加载流程
4. **数据验证**：验证数据质量和完整性

#### 5.2.2 模拟数据生成原则

| 原则 | 说明 |
|------|------|
| **业务真实性** | 数据分布符合实际业务 |
| **勾稽完整性** | 数据间满足业务规则 |
| **趋势合理性** | 变化趋势符合行业规律 |
| **异常可解释** | 异常数据有业务原因 |

### 5.3 应用开发方法

#### 5.3.1 开发技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 | React/Vue | UI框架 |
| 图表 | ECharts/D3 | 可视化 |
| AI能力 | 平台API | 对话接口 |
| 数据 | 平台SDK | 数据查询 |

#### 5.3.2 开发规范

| 规范 | 要求 |
|------|------|
| 代码风格 | 统一编码规范 |
| 组件复用 | 通用组件抽离 |
| 性能优化 | 按需加载、缓存 |
| 错误处理 | 统一错误处理机制 |

## 六、Phase 4：场景验证

### 6.1 验证维度

| 维度 | 验证内容 | 方法 |
|------|---------|------|
| **功能验证** | 功能完整性、正确性 | 测试用例执行 |
| **数据验证** | 数据准确性、完整性 | 数据校验脚本 |
| **性能验证** | 响应时间、并发能力 | 性能测试 |
| **业务验证** | 业务逻辑正确性 | 业务专家评审 |

### 6.2 验证指标

| 指标 | 目标值 |
|------|-------|
| 功能测试通过率 | ≥95% |
| 数据准确率 | ≥99% |
| 页面响应时间 | <3秒 |
| 业务验收通过率 | 100% |

### 6.3 验收标准

| 验收项 | 标准 |
|--------|------|
| 功能完整性 | 所有设计功能实现 |
| 数据质量 | 满足数据质量要求 |
| 演示效果 | 符合预期展示效果 |
| 文档完整性 | 交付文档齐全 |
